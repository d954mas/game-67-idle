# Runway Awakening Poki MVP Plan

## Outcome

Ship a bright, juicy, responsive Poki artifact for a 500-player Player Fit test.
The MVP contains three Essences, six visually distinct awakenings, thirty
wearables and an eight-round 8-12 minute path. It does not attempt to be the
full future game.

## Hard gates

1. Moon+Bloom and Moon+Flame runtime reveals both look strong enough to market
   in text-free thumbnails and clearly contrast with one another.
2. Initial release payload is <=6,500,000 bytes and below Poki's hard requirement.
3. Web audio and Poki Game Events work in release.
4. Six results remain distinguishable by 128 px black silhouette.
5. Cold players understand clothing causality and voluntarily start round two.

## Wave 1: prove value before volume

### Visual proof

- Lock or replace the master doll before producing the first capsule.
- Produce accepted Moon, Bloom and Flame main/accent focus pieces in T0404's six-item mini-capsule.
- Implement two contrasting proofs: Moon+Bloom Dreamgarden and Moon+Flame Eclipse.
- Capture both in portrait and landscape plus text-free thumbnail candidates.
- Test with 10 cold target players: >=8 connect clothing to magic and >=6
  voluntarily start a second outfit to seek the contrasting recipe.

### Size proof

- Remove full-body `_full` thumbnail duplicates from the runtime pack.
- Trim transparent bounds and reconstruct previews from body plus garment layer.
- Enable and visually verify supported atlas compression.
- Remove template-only assets from the release pack.
- Include HTML, JavaScript, WASM, platform adapter, pack and every other
  initially requested network file in a generated byte inventory.
- Reserve <=5,000,000 bytes for body, garments, reveals, backgrounds, UI,
  personalization, NPC/crowd and audio; reserve <=1,300,000 bytes for WASM,
  JavaScript, HTML and the Poki adapter; retain 200,000 bytes as contingency.
- Make the release build fail when initial bytes exceed exactly 6,500,000 bytes.

Do not batch-generate thirty garments until both contrasting reveals, cold
comprehension/round-two behavior, and the build-enforced size proof pass.

T0403 owns the thinnest production-quality spike needed for that test: a minimal
focus-pair resolver, two focus selectors, Dress->Restyle transition, shared
timeline seed and audio calls. T0405, T0406 and T0407 then harden and generalize
those proven seams for all recipes; they do not replace the spike with parallel
throwaway implementations.

## Wave 2: playable vertical system

- Replace themes, score and random rank with main-focus plus accent-focus recipes.
- Add state sequence: DRESS, RUNWAY_INTRO, CHARGE, FLASH, REVEAL, VICTORY, RECIPE_CARD.
- Persist discoveries, unlocks, best cards and last outfit.
- Rebuild the Dress Room around a large avatar, cropped item previews and one AWAKEN CTA.
- Create six signature awakening compositions using static layers and shared motion graphics.
- Integrate real web audio through `features/audio-core`; no second game-local backend.

## Wave 3: content and measurement

- Generate and accept thirty garments through the locked worn-peel pipeline.
- Prove every garment in source, peel, composite, mixed-outfit and compressed runtime views.
- Add six recipe cards and two remix slots per recipe.
- Award cosmetics after discoveries 1, 3 and 6; never gate core with ads or currency.
- Add approved low-cardinality Poki measure events for FTUE, rounds, recipes, Lookbook and active-time milestones.

## Ten-minute content contract

- Six first-discovery rounds at 55-90 seconds each.
- Two remix rounds using different support pieces for an existing pair.
- No timer, loading, ad or repeated cutscene is counted as content.
- Five cold internal testers must reach exhaustion in a median 8-12 minutes.
- Discovered focus items retain visible Essence marks. The pair matrix lets a
  player tap an unknown silhouette to highlight an owned main/accent route while
  preserving the exact result as a surprise. Repeated known pairs record remixes
  and route back toward a reachable unknown cell.

## Verification

### Automated

- Native build and all CTest targets.
- Unit tests for all six pairs, order independence, blocked focus state, always-win and save round-trip.
- DevAPI scenario for eight rounds, all recipes, reload and event tail.
- Responsive capture at 640x360, 640x480, 844x390, 390x844 and 360x640.
- One hundred automated cold starts with zero fatal/uncaught errors.
- Platform adapter tests for lifecycle, ads and `measure` once-guards.
- Artifact inspection proves Poki-only adapter and absence of DevAPI/debug payload.
- Release byte-inventory check sums all initial network files and fails >6,500,000 bytes.
- Representative mobile reveal meets median 60 FPS, p95 <=33 ms, no hitch
  >100 ms and memory high-water <=180 MB.

### Visual

- 30-item contact sheet and cross-slot collision matrix.
- Six black silhouettes at 128 px, distinguishable by 5/5 reviewers.
- Twelve final screenshots: six recipes in portrait and landscape.
- Runtime recording of full Dress -> Awaken -> Card -> Restyle loop.
- ETC1S/compressed alpha-edge comparison.

### Manual browser and portal

- Chrome/Edge desktop, Android Chrome and iPhone Safari.
- Touch, mouse, resize/orientation changes, blur/focus and reload during Dress and Reveal.
- Mute, reduced-effects, ad success/skip/fail and storage unavailable/incognito.
- Poki Inspector file-size, loading, scaling, QR/mobile and Event Log checks.
- Minimum five cold recorded playtests before Player Fit submission.

## 500-player decision contract

Track first equip, first AWAKEN visibility/interact, rounds 1/2/3/6/8,
six recipe discoveries, recipe counts 2/3/6, Lookbook interaction/save and active
60/180/300/600-second milestones.

Healthy internal targets:

- first equip >=85%;
- round 1 complete >=70%;
- round 2 start >=55%;
- round 3 complete >=30%;
- active 5 minutes >=20%;
- recipe 6 reached >=12%;
- save look >=15%;
- critical failure <1%.

Official Poki Player Fit gate is average playtime >3 minutes and >=125/500
players over 3 minutes. Internal `go` additionally requires average >=4 minutes,
>=35% over 3 minutes and round-2 start >=55%. Official pass below those numbers
gets one focused iteration. Official failure after that, or round-2 start <40%,
kills/redesigns the loop.

Do not split the 500-player sample into a 250/250 A/B test. Run a small local
thumbnail/fake-shot comparison first; use sequential full portal tests only if a
second external test is justified.

## Kill rules

- Projected or final initial payload cannot be brought to <=6,500,000 bytes without unacceptable art loss.
- The Moon+Bloom reveal is not more desirable than the ordinary dressed doll.
- Fewer than 5/6 awakenings are visually distinguishable.
- Fewer than 40% start a second round after one focused UX revision.
- Players cannot explain that the main outfit and accent caused the result.
- The worn-peel pipeline cannot reliably produce coherent clean capsules.

When a kill rule fires, do not compensate with story, economy or more content.
Fix the failed core gate or abandon the awakening loop.

## Accessibility and quality ownership

- UI: >=44x44 CSS px targets, contrast 4.5:1 normal/3:1 large, non-color
  selected state and responsive evidence (`QCLR_001`, `QCLR_002`).
- Reveal: no flash above 3 Hz, mute and reduced-effects path; runtime beauty and
  silhouette evidence (`QART_001`, `QDES_001`).
- Assets: provenance, integrity, peel/contact sheet and compression evidence
  (`QASSET_001`).
- Runtime/release: performance, lifecycle, browser and artifact proof
  (`QTECH_001`).
