## Context Rules

- Generic workflow policy: `AI_PIPELINE.md` and
  `docs/ai-pipeline/agent-workflow.md`.
- Start substantial work with `node tools/taskboard/cli.mjs context`, one
  relevant task/evidence file, and one matching skill.
- Skip archives, old logs, broad design folders, build outputs, and generated
  files unless task-linked or requested for review/history.
- Non-trivial game/visual/pipeline/tooling sessions need
  `node tools/ai.mjs start <task-id> <iteration>` or an explicit unavailable
  note.

## Game Work Rules

- Native PC is the playable harness. No web/mobile prototypes, servers,
  localhost, or browser/frontend tooling for playable work without explicit
  request or approval.
- Start new games with a fresh `gamedesign/projects/<game-id>/` wiki and fresh
  `tasks/active/` work items.
- Prefer small playable native slices over broad speculative systems.
- Test runs, spikes, prototypes, and real slices use the same quality gates; the
  label changes durability, not quality.
- Game visuals must use real assets through the engine asset path. Shape/debug
  renderers are debug-only.
- First playable screen: one location, one active primary path, one next action,
  visible reward/progress, and clear locked/future states.
- References are required when the user names one or says the build mismatches
  one. Record the smallest honest Reference Digest before implementing.

## Gates

- Gate taxonomy and validation defaults:
  `docs/ai-pipeline/quality-validation.md`.
- Builds/probes/manifests/audits are evidence, not acceptance. A slice is not
  done if it does not look/feel right or a new player cannot operate it.
- Visual/readability work needs screenshot evidence; UI/text needs zoomed
  readability proof and before/after whole-screen judgment.
- Lead rejection of a screenshot/build as ugly, unclear, or unplayable freezes
  feature/content expansion until the product failure is fixed.
- Repeated strict/product failures must change path. Guard:
  `node tools/product_gate/repeated_failure_guard.mjs`.

## Validation

- Docs/tasks: `node tools/taskboard/cli.mjs validate`.
- Skills/process: `node tools/skills_eval.mjs`.
- Reusable pipeline: `node tools/pipeline_validate.mjs`; use `--full` for
  portable export/runtime/deep asset gates.
- Product/readability: `node tools/product_gate/review.mjs` or
  `node tools/ai.mjs gate` with screenshot evidence.
- Native playable changes: smallest proving native build/run, plus
  screenshots/video when visual or playable.

When a task reveals repeated friction, prefer a small tool, validator, skill, or
source-of-truth fix over adding another long rule.
