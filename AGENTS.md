# AGENTS.md

## Project

- This is an AI-first game development experiment: improve the game and the AI workflow together.
- The engine lives in `external/neotolis-engine` as a submodule; do not edit it unless explicitly asked.
- Game design lives in `gamedesign/`; game code lives in `src/`.
- Active game concept: `67 World`, a child-friendly meme merge/evolution game
  inspired by the broad Cow Evolution loop. Every collectible character is a
  variation of the 67 character; strawberry, banana, and similar memes are
  skins, mutations, costumes, or world motifs for 67.
- Reusable design knowledge lives in `gamedesign/knowledge/`.
- Universal runtime infrastructure in this repo is intentionally kept:
  `state/` schemas/fixtures/migrations, `tools/state_codegen/`,
  `src/devapi/`, `tools/devapi/`, `src/game_storage.*`, and
  `external/cjson/`.
- Do not delete or replace that runtime infrastructure during concept cleanup;
  adapt it only when the new game needs different state or automation.
- Reusable project skills live in `.codex/skills/` (canonical; `.claude/skills/` is generated by `node tools/skills_sync.mjs`); keep them generic enough to reuse in other games.
- Work items and live project status live in `tasks/`; rules are in `tasks/README.md` and `.codex/skills/task-manager/`. Capture deferred ideas there; never track work status in design data files, GSD, or `.planning/`.
- Shared human/agent process and portable-base export: `AI_PIPELINE.md`.
- Temporary generation, scripts, rejected images, screenshots, and audit logs go in `tmp/` or another ignored temp/source folder.
- Final durable docs/data/assets go in their project folder.

## Direction

- The current product target is a release-quality `67 World` native PC/mobile
  game with roughly one hour of progression, FTUE/tutorial, child-friendly
  generated visuals, readable UI, validated balance, release packaging, and
  child-test/user acceptance.
- The current runtime surface in `src/main.c` is the active native 67 World
  game, not a placeholder. Treat `tasks/STATUS.md` as the live gate before
  deciding what to build next.
- Prefer small playable release-quality iterations over broad speculative
  systems. Do not change the active concept or restart the project unless the
  user explicitly asks for a new concept.
- For new GDD work, use `.codex/skills/primary-gdd-pipeline/`.
- For polished, child-testable, generated-art, UI, or release-quality visual
  work, use `.codex/skills/game-visual-art-direction/`. Do not treat
  shape-renderer rectangles, debug buttons, or programmer-art placeholders as
  the main visual solution when the user asked for generated art/UI.
- Reference study is a hard implementation gate. If a user names a reference
  game/style, do not implement gameplay, UI, economy, balance, or art from
  memory, vibes, store text, or a generic feature list. First create/update a
  durable deconstruction doc in the active design folder using
  `gamedesign/knowledge/reference_deconstruction.md`: source packet, checked
  date, visible-action ledger, player transcript, systems extraction, and
  translation gate. A central gameplay ref needs at least one gameplay
  video/walkthrough or long screenshot sequence, one official/store/trailer
  visual source, and the current native screenshot/capture for mismatch audit.
  The doc must answer: what is visible first, what the player does first, what
  visibly changes, where reward feedback appears, what UI is primary/secondary,
  what to borrow/avoid/copy-risk, why the current build differs, and which
  native screenshot/scenario will prove the next pass. If this artifact cannot
  answer a user challenge like "did you study the ref?", improve the study
  before coding. If sources are unavailable, ask for user material or record a
  narrow memory-only exception before implementation.
- When the user names a reference or challenges that the current build is not
  like the reference, run Reference Intake before any defense, redesign, code,
  or final art. The intake must state the exact reference question, selected
  study mode, required source packet, durable doc path, current native capture
  path or capture plan, and the first proof screenshot/scenario. Claims must be
  labeled as observed, inferred, user-provided, or unknown; unknown or inferred
  claims cannot drive implementation until supported or explicitly accepted as
  a narrow exception.
- Use the smallest reference-study mode that is honest for the task:
  quick check for a minor visual/UI detail, central deconstruction for a named
  gameplay/economy/art driver, and deep deconstruction only when the ref defines
  balance, first-session pacing, or release-critical UX. State the selected mode
  before research. A quick check still needs source links/paths and concrete
  visible observations; it cannot justify core gameplay or balance changes.
  A central/deep study must end with a native PC mismatch audit and the exact
  screenshot/scenario that proves the next implementation pass.
- Do not say a reference was "studied" unless the artifact proves it. For a
  named gameplay reference, the deconstruction must include links or local
  paths for the sources, at least one gameplay video/walkthrough or long
  screenshot sequence, official/store/trailer visuals when available, any
  guide/review/deconstruction used for balance or player-friction claims, a
  current native capture, and concrete timestamps/frames for observed beats.
  If those are missing, say the reference is not fully studied yet and either
  gather sources, ask the user for material, or record a narrow exception.
- Study refs in this source order before drawing conclusions: user-provided
  material first, official/store/trailer visuals second, raw gameplay
  video/walkthrough or a long screenshot sequence third, then guides, reviews,
  lectures, deconstructions, wikis, and community notes as supporting context.
  For central/deep refs, the durable doc must include a Source Ladder section
  with each source's role, checked date, what was watched/read, and whether it
  is observation evidence or secondary interpretation. Do not let secondary
  summaries replace observed gameplay frames, and do not proceed from search
  snippets or thumbnails.
- Central/deep refs need a Reference Evidence Board before conclusions: at
  least six saved or cited frames/screenshots covering first screen, first
  input, visible response, reward feedback, upgrade/progression UI, and a
  friction or blocked/full-board state; plus one raw gameplay segment,
  walkthrough, or long screenshot sequence for interaction timing. If balance,
  pacing, or player-friction claims are used, add at least one guide, review,
  lecture, wiki, or deconstruction as supporting interpretation. Record links,
  local paths, timestamps/frame ids, checked date, and what each item proves in
  the durable doc. If the agent cannot point to these frames or timestamps, the
  ref has not been studied enough for implementation.
- Start reference work with a Reference Lock before coding or generating final
  assets. The lock must state: study mode, exact question the ref must answer,
  durable deconstruction doc path, required source packet, current native
  capture path or capture plan, no-coding/no-final-art boundary, and the
  expected native screenshot/scenario proof. After research, implementation
  only unlocks when the doc records at least three observed facts from sources,
  one current-build mismatch, and one scoped next pass tied to proof.
- Treat reference study as closed until its Definition of Ready passes. Before
  coding or final art, the agent must be able to cite the durable doc path,
  source matrix, current native capture, at least five visible observation
  beats for central/deep gameplay refs, borrow/avoid/copy-risk, current-build
  mismatch, and the exact native proof for the next pass. If any part is
  missing, state "reference study is not ready for implementation" and continue
  with source gathering/deconstruction or ask for user material. Do not bypass
  the gate by relabeling loop, first-screen, control, economy, balance, art
  direction, or release-readability work as small polish.
- Reference study must be visible to the user before implementation resumes.
  After Reference Intake or after a deconstruction update, provide a short
  Reference Digest: study mode, sources actually checked, 3-5 observed facts,
  current-build mismatch, borrow/avoid/copy-risk, and the exact native proof
  for the next pass. If the digest cannot be written from the durable artifact,
  do not code or generate final art yet; say which evidence is missing.
- Parallel reference work is allowed only as a research lane until the gate is
  ready. Agents may gather sources, capture frames, transcribe observed actions,
  or update the durable deconstruction while another lane handles unrelated
  setup, but gameplay/UI/economy/balance/final-art implementation that depends
  on the named reference stays locked until the Reference Digest and native
  mismatch audit exist. Do not "study while building" and then retrofit the
  conclusions to the implementation.

## Validation

- Product target is mobile + PC. Native desktop/PC is the preferred development and automation harness once implementation starts.
- Run web/mobile builds only when the task explicitly targets web/mobile behavior or visual GDD surfaces.
- Hard gate: for playable game work, do not create, serve, validate, or pivot to
  a web prototype/page/app because it seems faster or prettier. Use the native
  PC build first. Web is allowed only when the user's current request explicitly
  says web/mobile/browser/visual GDD/web build, or when the user approves a
  clearly stated exception after being asked.
- Before any command that starts a web server, opens localhost for a web build,
  creates HTML/CSS/JS prototype files, or installs browser/frontend tooling,
  restate the explicit user permission that allows web work. If that permission
  is absent, stop and continue on the native PC path.
- When validating playable or visual changes, use screenshots and emulated input. Cover desktop and mobile portrait when the surface is web-based.
- Pack building is explicit; do not wire pack generation into every normal game build.
- If a task reveals repeated friction, propose updating `AGENTS.md` or creating/updating a project skill.
