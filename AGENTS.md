- For substantial work, load taskboard context, one task/evidence file, and one matching skill.
- Skip archives, old logs, broad design folders, build outputs, and generated files unless task-linked or requested.
- Non-trivial game/visual/pipeline/tooling work needs an AI profiling scope or unavailable note.

## Game Work Rules

- Native PC is the playable harness. No web/mobile prototypes, servers,
  localhost, or frontend tooling for playable work without explicit approval.
- Start new games with a fresh project wiki and `tasks/active/` work items.
- Prefer small playable native slices; tests, spikes, prototypes, and slices use the same gates.
- Game visuals use real assets through the engine asset path; shape/debug renderers are debug-only.
- First playable screen: one location, one primary path, one next action, visible progress, clear locked/future states.
- If the user names a reference or mismatch, write the smallest honest Reference Digest before implementation.

## Gates

- Gate taxonomy and validation defaults: `docs/ai-pipeline/quality-validation.md`.
- Builds/probes/audits are evidence, not acceptance.
- Visual/readability work needs screenshot evidence, zoomed UI/text proof, and before/after judgment.
- Lead rejection freezes feature/content expansion until fixed.
- Repeated strict/product failures must change path; guard with `node tools/product_gate/repeated_failure_guard.mjs`.

## Validation

- Docs/tasks: `node tools/taskboard/cli.mjs validate`.
- Skills/process: `node tools/skills_eval.mjs`.
- Reusable pipeline: `node tools/pipeline_validate.mjs`; use `--full` for portable export/runtime/deep asset gates.
- Product/readability: `node tools/product_gate/review.mjs` or `node tools/ai.mjs gate` with screenshot evidence.
- Native playable changes: smallest proving native build/run, plus screenshot or video when visual/playable.

When a task reveals repeated friction, prefer a small tool, validator, skill, or
source-of-truth fix over adding another long rule.
