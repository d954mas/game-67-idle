# State system v2: полный план (синтез двух Opus-проектов, 2026-07-06)

Полная переделка стейта и генератора. Мандат лида: «гибко и удобно для
использования мной в играх и фичах; база на годы расширения».
Входные решения зафиксированы: templates/design/reviews_synthesis_2026-07-06.md §8.
Контекст: templates/design/feature_architecture_2026-07-06.md,
templates/design/item_system_design_2026-07-06.md §6.

## 1. Роли

- **L0 шелл (руками, один раз, в шаблоне)**: game_storage (байты на диск:
  native atomic / web localStorage), game_save (конверт + реестр фрагментов
  + оркестрация + dirty/debounce/flush + DevAPI-диспатч),
  game_state_json (общий JSON-тулкит gsj_*: типизированные ридеры с
  дефолтами, int64-строки, транзакционный validate-copy-swap).
- **Фрагмент (у фичи; руками ИЛИ генерируется)**: типизированный
  <Id>State, дескриптор, версия, миграционные шаги, схема.
- **Тулинг**: generate_state.py v2 (--fragment) + схема = источник и для
  редактора, и для DevAPI.

## 2. Формат на диске

Один атомарный документ на слот:

```json
{ "format": 1, "save_version": 1, "saved_at": 1720080000000,
  "app": "mygame", "build": "0.3.1",
  "features": {
    "game":        { "v": 1, "quest_stage": "accepted" },
    "settings":    { "v": 1, "master_volume": 0.75 },
    "items":       { "v": 2, "instances": {...},
                     "containers": { "backpack": [{"id":"a1"}],
                                     "purse": [{"id":"gold","n":"128000"}] },
                     "quarantine": [...] },
    "progression": { "v": 1, "tracks": { "hero": {"level":7,"xp":"1240"} } } } }
```

- Конверт (шелл): format (версия контейнера), save_version (гейт редких
  кросс-фрагментных шагов), saved_at (wall-clock ms — число, <2^53, ок),
  app/build (диагностика). Парсер конверта игнорирует незнакомые ключи с
  первого дня (шов для облака/слотов).
- Фрагмент: плоский payload + **"v" инжектит шелл** (хуки фичи версию не
  трогают; генератор запрещает поле с именем "v"). Большие счётчики
  (int64) — JSON-строкой (порча double >2^53).
- Слоты: native `build/saves/<slot>.json` (+ .tmp scratch, + .bak);
  web `"<app>/save/<slot>"` (+ .bak-ключ). APP_ID — compile define.
- **Bak-ротация native, два атомарных rename без копий**: write+flush tmp
  (WRITE_THROUGH) → rename(primary→.bak) → rename(tmp→primary). Порванный
  primary ненаблюдаем; loader: primary → .bak → CORRUPT_RESET (+ карантин
  файла `.corrupt-<ts>` для форензики, статус наружу).
- Cap 1MB; hot-path печать PrintUnformatted.

## 3. Контракт фрагмента (весь ABI между L0 и фичей)

```c
typedef bool (*GameSaveMigrateFn)(cJSON *frag, char *err, int cap); /* v(i)->v(i+1) */
typedef struct GameSaveFragment {
    const char *id;            /* ключ в features + префикс символов */
    int version;               /* = steps_count + 1 */
    const GameSaveMigrateFn *steps;   /* NULL пока миграций нет (90%) */
    void  (*reset)(void);             /* дефолты (new game / absent) */
    cJSON*(*to_json)(void);           /* только данные, без "v" */
    bool  (*from_json)(const cJSON*, char *err, int cap); /* толерантный */
    void  (*reconcile)(void);         /* NULLABLE: пост-load фиксап (карантин items) */
    cJSON*(*get_path_json)(const char *sub, char *err, int cap);  /* DevAPI read */
    bool  (*set_path_json)(const char *sub, const cJSON*, char*, int); /* NULLABLE */
    cJSON*(*schema_json)(void);       /* NULLABLE у рукописных (но см. Р9) */
} GameSaveFragment;
```

- Регистрация: `game_save_register_fragment(&settings_save_fragment);` —
  одна строка на фрагмент в game_features_init(). Без авто-дискавери.
- **Рукописный settings = ~44 строки** (хедер 11 + state.c 33) поверх gsj_*.
- **Генерируемый progression = только схема (~40 строк JSON)**;
  «добавить поле» = 1 строка схемы, 0 строк C, без версий/миграций.
- Мутаторы фич сами зовут game_save_mark_dirty() — автор игры автосейв
  не пишет вообще.

## 4. Поверхность автора игры

- Compile defines: GAME_STORAGE_APP_ID, GAME_SAVE_AUTOSAVE_SLOT
  ("autosave", один rolling-слот в шаблоне; мульти-слоты = параметр,
  шов готов), GAME_SAVE_DEBOUNCE_MS (2000), GAME_SAVE_MAX_INTERVAL_MS
  (30000), GAME_SAVE_DOC_VERSION.
- main.c шаблона: init → load (статусы FRESH/LOADED/RECOVERED_BAK/
  CORRUPT_RESET — игра может показать тост) → в кадре game_save_tick();
  на вебе один раз game_save_install_web_flush() (visibilitychange→hidden
  + pagehide, НЕ beforeunload).
- Стейт самой игры = фрагмент "game" (в шаблоне рукописный — учит ручной
  путь; progression — генерируемый: по одному образцу каждого пути).
- game_save_last_saved_at() — оффлайн-Δt для идла (кламп отрицательных).

## 5. Схема v2

Типы: int, **i64** (JSON-строка, C int64_t, ридер принимает строку ИЛИ
число), float, bool, string, enum, list<string>, map<string,Object>,
object-типы через types{}. Пер-поле: default, min/max, max_length,
max_count, deprecated (читается, не пишется, редактор прячет),
**ui:{}** (сквозной: codegen игнорирует, редактор/DevAPI читают).
Топ-левел: fragment, version, string_max, reserved:[имена] (умершие имена
полей нельзя переиспользовать).

**ВЫРЕЗАНО из v1**: числовые id полей + id-диапазоны (идентичность = имя
JSON-ключа; фрагменты неймспейсят сами), collections{} (→ max_count),
lifetime, per-field devapi (всё читаемо по умолчанию), document/GameState-
байндинг (→ fragment). fields = map имя→спека (нормализация в старый
list-вид питает существующие рендереры без переписывания).

## 6. Оркестрация

LOAD: файл отсутствует → new_game (FRESH). Парс-фейл → .bak → иначе
карантин файла + new_game (CORRUPT_RESET). format незнаком/новее → как
парс-фейл (bak-путь). save_version < DOC_VERSION → кросс-фрагментные
doc-шаги по сырому features (игра владеет; фейл шага → bak-путь).
Затем пофрагментно, независимо: reset() → absent → дефолты (новая фича ≠
миграция) → v<current → чейн шагов (фейл → дефолты этого фрагмента + лог;
**никогда не all-or-nothing** — плохой фрагмент не теряет кошелёк) →
v>current → толерантный from_json + warn (без шагов) → from_json.
После всех — reconcile-проход (карантин items видит все фрагменты).

SAVE: to_json всех → шелл штампует "v" → конверт + saved_at →
PrintUnformatted → атомарная запись+ротация → clear dirty (фейл →
dirty остаётся, ретрай следующим тиком).

Dirty/debounce: mark_dirty из мутаторов; tick сохраняет при
(now-dirty_at ≥ DEBOUNCE) или (now-last_save ≥ MAX_INTERVAL — жёсткий кап
при непрерывном потоке правок). Всё на главном потоке; аллокации только
в момент фактического сейва (~каждые 2с), пик ≈ 2× размера документа.

## 7. Генератор v2

- Всегда пер-фрагментный; вход src/features/<id>/state.schema.json;
  выход <id>_state.h / <id>_state.c / <id>_state_schema.gen.h (встроенная
  схема чанками для DevAPI/редактора).
- **Неймспейс = фикс коллизий**: <id>_state_* / <Id>State / <ID>_STATE_*;
  общий GameState/GAME_STATE_* умирает.
- Реюз generic-рендереров (generate_state.py:1082-1742) с
  параметризованным префиксом; статик-хелперы (1779-1896) выносятся в
  общий game_state_json.c (дедуп: один раз, не в каждом фрагменте).
  Умирают: пре-914 половина, game_state.c.in, REQUIRED_FIELDS/
  SOURCE_MARKERS, генерируемые конверт/файловый I/O/7 DevAPI-эндпоинтов
  (всё это теперь шелл), глобальный g_game_state.
- Валидация v2: + коллизии c_ident (два пути → один C-идентификатор =
  ошибка; сегодня ловит только компилятор), + запрет поля "v",
  + уникальность fragment id, + границы i64.
- CMake: функция game_state_add_fragment(target id) — одна append-строка
  на фрагмент; каждый фрагмент зависит только от СВОЕЙ схемы (параллельная
  сборка, нет пересборки монолита).
- **События (секция `events` той же схемы фрагмента)**: генератор на каждое
  событие эмитит типизированный структ + emit-хелпер + константу type_hash +
  дескриптор {имя, тип, offset} + регистрацию имени типа в debug
  (nt_hash_register_label64) — отдельный рендерер поверх стейт-пути. Словарь
  типов полей выровнен со стейт-схемой (int/i64/float/bool/string/enum +
  hash, bytes — event-only). ИНКРЕМЕНТ E2 (ПОСЛЕ A4, расширение генератора
  v2; шелл-часть лога — E1 с A3). См.
  templates/design/event_system_design_2026-07-06.md §3/§6.
- Самотесты генератора: golden-файлы + property-проверки (стиль
  существующего generate_state_test.py) + compile+roundtrip ctest
  (настоящая сеть; golden гниют).

## 8. DevAPI (ноль правок движка)

Команды остаются теми же 7 `game.state.*` (континуитет тулинга/ботов),
регистрирует шелл один раз; внутри — диспатч по реестру:
get {path:""} → агрегат = features-map (снапшот = save-путь минус
запись); get "items.purse.gold" → голова пути = id фрагмента →
frag->get_path_json; set/patch → set_path_json владельца (+ mark_dirty,
patch транзакционно через gsj_transact); schema → агрегат схем фрагментов
(= форма для будущего редактора); save/load/reset → шелл.
Рукописный фрагмент без set_path = read-only для ботов (боты в основном
читают). Байт-совместимости со старым плоским агрегатом rb-dark НЕТ
(игра закрыта; смоук-бот шаблона переписывается на фрагментные ключи).

## 9. Тесты

Пофрагментный round-trip (reset→to_json→from_json→to_json equal +
толерантность: absent/missing/unknown/newer-v); фикстуры прошлых версий
features/<id>/tests/fixtures/v<N>.json + прогон чейна (привычка → гейт);
crash-safety ctest по настоящему game_storage (порча primary → RECOVERED,
окно между rename → last-good, порча обоих → CORRUPT_RESET + .corrupt-*);
конвертные кейсы; golden+compile+roundtrip генератора. ctest в шаблоне.

## 10. Перфоманс и швы расширения (проектируем, не строим)

Полный сейв 50KB ≈ 1-5мс раз в ~2с — пренебрежимо. Триггер инкрементала:
>4мс И чаще раза в секунду → шов: пер-фрагментные dirty-биты за тем же
mark_dirty, пересериализация только грязных (код фич не меняется).
Швы за ~ноль сейчас: bignum (провод уже строковый), облако (конверт
толерантен к незнакомым ключам), мульти-слоты (параметр slot), экспорт
сейва игроком (документ = одна самоописанная строка), аналитика стейта
(документ материализуется при каждой записи).

## 11. Анти-фичи

Авто-дискавери/constructor-регистрация (порядок недетерминирован);
рефлексия; наследование схем; кросс-фрагментные ссылки с глобальной
целостностью (dangling = забота фичи-владельца, reconcile/карантин);
generic entity-bag в сейве; бинарный формат/чексуммы (atomic+bak+parse);
возрождение глобального GameState.

## 14. ПОПРАВКИ ПО РЕВЬЮ ВОЛНЫ 2 (2026-07-06; при противоречии — §14 главнее)

Оба ревьюера (адверсариал + индустрия): «ship with fixes», ядро верно.

1. **Ротация упрощается**: ОДИН атомарный replace_file(tmp→primary)
   (MoveFileEx REPLACE|WRITE_THROUGH; primary никогда не отсутствует и не
   рвётся) + .bak пишется ОДИН раз за сессию при успешной загрузке (копия
   заведомо-парсящегося last-known-good). Двух-rename схема §2 отменена —
   у неё окно: краш между rename'ами → primary отсутствует, лоадер молча
   грузил бы СТАРЫЙ .bak при целом новом .tmp; плюс POSIX-rename ловушка
   на Windows (EEXIST).
2. **reset() расщепляется — БЛОКЕР индустрии**: reset() = нейтральные
   пустые дефолты (поздно-добавленная фича = пусто, правило «новая фича ≠
   миграция» сохранено); НОВЫЙ nullable-хук on_new_game() в дескрипторе =
   стартовый контент (100 золота, стартовые контейнеры, меч) — фрагмент
   владеет данными, шелл триггерит ТОЛЬКО при глобально-свежем сейве
   (после всех reset, до первого save). Иначе новая игра рождалась бы с
   пустым кошельком. [Р11 — ПРИНЯТО лидом 2026-07-06]
   Срабатывает ровно в 3 случаях: FRESH (файла нет), явный
   game_save_new_game(), CORRUPT_RESET. НЕ срабатывает: поздно-добавленная
   фича на живом сейве (только её reset = пусто); геймплейные ресеты
   (престиж = логика игры через API фич, не new_game сейва). Хук — поле
   дескриптора; дескрипторы регистрируются в init фич ДО первого load.
   Порядок: фрагмент game регистрируется ПОСЛЕДНИМ (самый зависимый) →
   его on_new_game/reconcile бегут последними → в нём игра сеет контент
   через API фич (items_add("gold",100,"starting")). Разделение: хук
   ФИЧИ = bootstrap своих структур из своих данных (items создаёт
   контейнеры из списка в items.json); хук GAME = контентные решения
   (сколько золота, какой меч, стартовый квест) — знание игры.
3. **Web-запись обязана уметь фейлиться вслух**: на itch общий origin
   html.itch.zone и ОБЩАЯ квота 5-10MB на все igры игрока; Safari private
   кидает на любой setItem. try/catch → статус SAVE_UNPERSISTED + probe-
   запись на старте + возможность игре показать «браузер не сохраняет».
   Web .bak-ключ ВЫРЕЗАН (setItem атомарен per-key, порванных значений не
   бывает; .bak удваивал бы расход общей квоты). Native .bak остаётся.
4. **Экспорт/импорт сейва строкой — ФУНКЦИИ в шелле с первого web-
   инкремента** [Р12 — РЕШЕНО лидом 2026-07-06]:
   game_save_export_string()/import_string() (+валидация конверта) — API
   даёт тулкит; ВЫВОДИТЬ ли в UI и куда (настройки/чит-меню/нигде) —
   решает игра (принцип «вызов — задача игры»). Плюс DevAPI-команда
   поверх тех же функций (агентам и баг-репортам UI не нужен). Контекст:
   ~30 строк, у Cookie Clicker/AD/Kittens это главный механизм
   восстановления, itch называет его единственной защитой от общей
   квоты/private-mode; бесплатный перенос между устройствами.
5. **visibility-flush = синхронный форс-сейв в обход дебаунса** (rAF-цикл
   замерзает у скрытой вкладки — tick после hidden не случится никогда);
   тест обязателен.
6. **dirty_at = момент ПЕРВОЙ пометки после чистого** (иначе непрерывный
   идл-продюсер, марающий каждый кадр, никогда не удовлетворит дебаунс и
   сейвился бы только по MAX_INTERVAL). MAX_INTERVAL остаётся (несущий
   для идла, не спекуляция).
7. **patch по фрагментам НЕ транзакционен** — заявление §8 снято: patch
   атомарен пофрагментно (gsj_transact), результат по-ключево; кросс-
   фрагментного отката нет (боты в основном читают — приемлемо).
8. **i64-провод пригвождён**: get_path эмитит строку; set_path принимает
   строку всегда, число — только если точно представимо; в DevAPI-
   контракт.
9. **Миграционные правила дописаны**: doc-шаги видят сейв КАК ЗАПИСАН
   (старые имена/версии фрагментов), бегут до всех пофрагментных чейнов;
   фрагмент без "v" → считается v1; reconcile в порядке регистрации —
   зависимости (items) регистрировать раньше зависимых (equipment);
   на каждый doc-шаг обязательна фикстура. Переименование поля =
   миграционный шаг + имя в reserved (дисциплина в скилл).
10. **Пофрагментные сбросы/карантины — в результат загрузки** (не только
    лог): игра может показать «инвентарь не загрузился», а не молча
    потерять фрагмент.
11. **schema_version диалекта — в файл схемы** (+1 assert генератора);
    **save_seq** (монотонный счётчик) — в конверт (кладётся дёшево сейчас,
    облачный шов без wall-clock-конфликтов).
12. **Template clean-break заявлен**: дев-сейвы шаблона бросаются на A3;
    межинкрементной совместимости НЕТ; на A4 схема шаблона переписывается
    руками в v2-вид. Агенту-исполнителю не строить компат-шимов.
13. **Генератор — честный скоуп**: «параметризация префиксом» = ~70
    литеральных вхождений GameState/GAME_STATE_/game_state_ по рендерерам
    + переписывание render_header/render_generic_source под новый ABI.
    Механично, но это переписывание; пакету исполнителя — явный список
    рендереров и ОБЯЗАТЕЛЬНЫЙ golden+compile+roundtrip гейт. Чарсет id
    фрагмента: [a-z_][a-z0-9_]* (он же C-префикс). fields-map: ключи =
    dotted paths как сегодня. Секция events схемы (event_system §3/§6) —
    ОТДЕЛЬНЫЙ рендерер (struct+emit+type_hash+дескриптор), инкремент E2
    ПОСЛЕ A4, вне скоупа переписывания стейт-рендереров A4 (шелл-лог E1
    идёт с A3); golden+compile+roundtrip покрывает и события.
14. **Вырезано**: .corrupt-<ts> форензик-файл (native-only, читателей
    нет — остаётся только статус CORRUPT_RESET); web .bak (п.3);
    game-side обработка RECOVERED_BAK (advisory-статус остаётся).

15. **Цепочка пост-обработки сейва** (требование лида 2026-07-06):
    упорядоченный список трансформов {id, encode, decode} в конфиге между
    PrintUnformatted и бэкендом (save: сжать→зашифровать→записать; load:
    в обратном порядке). АВТОДЕТЕКТ: плоский JSON начинается с '{';
    трансформированный файл несёт префикс "NTSV1:<ids>:" → загрузчик
    понимает оба всегда. Дефолт debug-сборки и шаблона: цепочка ПУСТАЯ →
    на ПК сейв = читаемый/редактируемый руками JSON-файл; подложенный
    плоский файл грузится даже сборкой с шифрованием. Экспорт/импорт —
    тем же путём (+base64 для бинарного). Сжатие = первый реальный
    трансформ (itch-квота; JSON режется 5-10×). Шифрование = защита от
    казуального редактирования, НЕ криптография (ключ в клиенте) —
    ожидание зафиксировано. Атомарность не затрагивается (цепочка до
    записи). Включение цепочки — решение игры.

16. **NEWER-детект: только версии; незнакомый ключ ≠ новизна** (решение
    лида 2026-07-06, закрывает гэп ревью build-спеки; правит Р7/Р13):
    новее = format незнаком ИЛИ save_version > моего ИЛИ v ЗНАКОМОГО
    фрагмента > версии его кода. Незнакомый фрагмент-ключ из детекта
    ИСКЛЮЧЁН: при версиях ≤ моих это сирота (удалённой или ещё не
    известной мне фичи) → warn + загрузка продолжается (Р7). Чтобы
    пинг-понг версий не терял данные, незнакомые ключи features{}
    ОБЯЗАТЕЛЬНО сохраняются verbatim-блобом при каждом save
    (round-trip). Обоснование старого 4-го условия («пересейв молча
    выбросил бы новые фрагменты») этим снято. Retired-списков нет.

ОТЛОЖЕНО осознанно (швы описаны): IndexedDB (>1-2MB или облако),
LZ-компрессия (по боли размера), unknown-field retention для lossless
даунгрейда, thumbnail/last-scene конверта, playtime (поле фрагмента game,
не конверт).

## 13. Решения лида (с рекомендациями синтеза)

Р7/Р13 — РЕШЕНО лидом 2026-07-06: ЖЁСТКИЙ ОТКАЗ. Сейв новее билда →
    статус NEWER, явная ошибка «обнови игру», НОЛЬ записей на диск
    (никакого clobber; экспорт-чтение доступен). Детект СОСТАВНОЙ, без
    глобального числа (его пришлось бы бампать руками и забывали бы):
    новее = format > известного ИЛИ save_version > ИЛИ v любого
    зарегистрированного фрагмента > версии его кода ИЛИ в сейве есть
    незнакомый фрагмент-ключ. Обоснование: старый билд при пересейве
    молча выбросил бы целые новые фрагменты. Обратное направление
    (новый код + старый сейв) — миграции/толерантность как было;
    сирота удалённой фичи отказ не триггерит.
    [ПОПРАВЛЕНО 2026-07-06 → §14 п.16: незнакомый ключ исключён из
    детекта (конфликтовал с сиротой-правилом); данные защищает
    verbatim-сохранение незнакомых ключей при save.]
Р8 — РЕШЕНО лидом 2026-07-06: агрегат game.state.get — фрагментные ключи;
    НИКАКИХ слоёв совместимости со старым плоским видом и фолбеков
    («чистая, новая, лучшая схема без техдолга»); смоук-бот шаблона
    переписывается.
Р9 — СНЯТ решением лида 2026-07-06: «рукописного пути» НЕТ. Путь один:
    у ВСЕХ фрагментов (включая settings и game) есть схема → стейт-слой
    (структура/дефолты/сериализация/встроенная схема) всегда генерируется;
    руками пишется только ЛОГИКА (API фичи, reconcile, on_new_game,
    миграционные шаги). Следствие для Р5: генератор --fragment двигается
    РАНЬШЕ (пререквизит первого фрагмента); settings = его первый тест.
    Дескриптор-ABI не меняется. У фрагмента game схема своя
    (state.schema.json игры).
Р10 — РЕШЕНО лидом 2026-07-06: шелл = механизм, игра = решение. При
    двойном фейле (primary+bak): шелл откладывает битый файл в .corrupt
    (ничего не уничтожает), сбрасывает фрагменты в памяти, возвращает
    статус CORRUPT с деталями и ПРИОСТАНАВЛИВАЕТ автосейв до явного
    решения игры. Игра показывает попап: «начать заново» →
    game_save_new_game() (там и срабатывает on_new_game, автосейв
    возобновляется) / «импортировать» → import_string / (дев: починить
    .corrupt руками и вернуть на место — автодетект съест). Шаблон несёт
    дефолт-обработку (тост + новая игра). Это отменяет вырезание
    .corrupt-файла из §14 п.14 — читатель есть: лид правит сейвы руками
    (native rename; web copy-в-ключ).
