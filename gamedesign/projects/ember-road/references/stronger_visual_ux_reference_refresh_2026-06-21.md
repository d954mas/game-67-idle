## Screen Translation Rules

The next Ember Road proof must be translated into native UI rules, not copied
as a poster:

1. One scene owns the fantasy: forge/workbench, lantern, blacksmith/warden,
   shard, route/depth marker, and hero live in the illustrated location.
2. Right rail summarizes equipment/result; it must not be the main action
   surface.
3. Route/depth is a map-like strip with current node, previous route, next
   node/depth, lock/open reason, and reward promise.
4. Reward/result uses at least one visual slot or object change, not only log
   text.
5. Bottom log confirms what happened; it cannot be the only evidence of the
   upgrade.
6. Player-facing labels must be final game terms. No `NEXT SLICE`, no
   implementation labels, no debug status.
7. The screen remains dense enough to read as a browser RPG, but premium/event
   clutter stays out of the first proof.

## Y-Up Non-Negotiable

Y is up. Always.

- Larger logical `y` means visually higher in game/world/UI layout.
- Reference screenshots can be discussed as images, but implementation notes,
  crop placement, route placement, and UI box layout must use Y-up semantics.
- Use words like `above`, `below`, `top`, `bottom`, `center_y`, and
  `y_from_bottom` consistently with Y-up logic.
- A Y-down number may exist only inside a named renderer/input/screenshot/
  DevAPI boundary conversion. It must never become stored layout truth.
- Any native proof for this task must include a Y-up layout audit. If it cannot
  show that route, forge, rail, log, and scene anchors are authored Y-up, it
  does not close the rejection.

## Y-Up Contract

- Game/world/UI logic is Y-up. Larger logical Y means higher on the game screen.
- `UiBox` and route/depth placement stay Y-up.
- Renderer, input, screenshot, and DevAPI rectangles may convert to Y-down only
  in named boundary adapters.
- No visual rewrite may store Y-down coordinates as layout truth.
- Any future prompt, crop manifest, or runtime layout note must preserve Y-up
  wording and reject inverted layout assumptions.

## Translation Gate

Borrow:

- scene-first location where the meaningful action lives in the world;
- quest/object markers and result feedback tied to the scene;
- route/depth state with current node, next node, lock/reason, and reward;
- item/equipment slot or object glow for the Mine Lantern result;
- bottom log as confirmation, not the only place where the event exists;
- dense fantasy chrome that supports interaction, not decorative chrome.

Avoid:

- adding Depth 2 before the town/equipment visual mismatch is solved;
- copied reference UI, exact ornaments, names, icons, characters, or layouts;
- MMO/VIP/daily/event clutter;
- action-RPG controls/camera scope;
- idle/wait-pressure framing;
- right-rail form UX as the main fantasy.

Copy-risk:

- exact War of Dragons/Dragon Eternity/AQ/DragonFable/Shakes & Fidget
  screenshots, proportions, character designs, icons, UI frames, and
  monetization surfaces.
