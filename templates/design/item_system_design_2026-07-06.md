# Item system: описание и план (принято лидом 2026-07-06, ревью pending)

Чистовой дизайн, рассчитанный на расширяемость и настройку под каждую игру.
Не наследует форму джемовой реализации rb-dark-rpg (там анти-паттерны:
арт strcmp-цепочками в обход icon_asset_id, полу-захардкоженные слоты).
Контекст архитектуры: templates/design/feature_architecture_2026-07-06.md.
Исследования: templates/design/items_feature_study_2026-07-06.md.

Правка лида 2026-07-06 (вторая итерация): валюты и опыт — ТОЖЕ ПРЕДМЕТЫ
(безлимитный стак, скрыты из инвентаря, добавление/трата = логика предметов).
Отдельная фича wallet УДАЛЕНА из дизайна.

## 1. Состав (после ревью и решения лида 2026-07-06: слияние)

```
L1  items      каталог + инстансы/стаки + КОНТЕЙНЕРЫ (вкл. валюты+purse)  (ни от кого)
L1+ equipment  слоты-как-данные                                          (items)
L2  progression — универсальные уровни (учебная фича шаблона, принято; §5)  (items)
    shop       — минимальный buy-only стаб; полноценный при первой игре с магазином
```

**Одна фича владеет всей системой владения** (решение лида по итогам
адверсариального ревью: инвариант «каждый живой инстанс ровно в одном
контейнере» должен жить в одном месте; create принимает контейнер-
назначение; пул — аллокатор, не место). Внутри папки модульность файлами:
items_catalog.c / items_instances.c / items_containers.c; публичный хедер
один. «Кошелёк» = скрытый безлимитный контейнер (purse). Порядок хранения:
контейнер = упорядоченный фикс-массив записей {id, count:int64 (большие —
строкой, §2/Р1)} (id = def_id для стака / instance_id для уника; порядок
массива = данные UI/слотов (2D-сетка НЕ кодируется 1D-массивом — claim
выкинут, synthesis §3); один стак на def_id внутри контейнера; запросы —
линейный скан, map только в пуле инстансов instance_id→структура).
Bootstrap: стартовый список контейнеров (рюкзак, purse, ...) и их
вместимости объявлены в items.json; accept-policy = enum {any,
currency_only, slot_filter, capacity_1} (synthesis §4).

## 2. items: каталог + блочная модель + инстансы/стаки

**Строка каталога = крошечное ядро + опциональные блоки.**

Ядро (фиксирует фича, одинаково во всех играх): `id, display_name,
icon_asset_id, tags[], stack {stackable, max_stack | unlimited}, base_value`.
Счётчик стака — **int64** ВЕЗДЕ (Р1: int64 сейчас во всём, валюты тоже
int64, НЕ double). Большие счётчики в JSON — строкой (cJSON-double портит
>2^53); дробное производство копит аккумулятор в game glue, не в count.
Экспоненциальный идл — будущая точка расширения (флаг/тип/фича), не строим
(synthesis §8).

Блоки (наличие блока = поведение; enum «типов» ЗАПРЕЩЁН):
- `equip` — слот + стат-блок. Стат-поля объявляет ИГРА. По ревью: стат-
  CODEGEN отложен из инкремента 1 — сначала copy-then-own рукописного
  типизированного структа; item_fields.schema.json + codegen — по триггеру
  «редактор или вторая стат-схема». Есть equip → инстансы уникальны
  (per-copy стейт: фиксированные скаляры durability/level/...; переменные
  списки аффиксов — отложенное аддитивное расширение codegen, текущий
  генератор их не выражает).
- `use` — effect_id + параметры. Фича хранит данные, НЕ исполняет эффект
  (интерпретация у игры/фич выше — items остаётся чистой L1).
- `currency` — стак безлимитный (опциональный cap с клампом на add —
  жизни/энергия в match-3), скрыт из инвентарного UI (HUD-подсказка
  отображения); в остальном обычный предмет: золото, опыт, гемы, души.
- Предмет может иметь несколько блоков (факел = equip+use) или ни одного
  (материал). Стак-дефолт: есть equip → не стакается, иначе стакается.

Категории/вкладки («оружие/зелья/материалы») — пер-геймные ДАННЫЕ
(item_kinds в items.json), витрина, не поведение.

**Владение стакуемым = счёт** (StackInstance{def_id, count:int64});
**владение уникальным = инстанс** (per-copy стейт). Пул инстансов и чистые
стак-правила живут в items (копия существует независимо от владения:
земля, сундук, buyback, почта — поэтому пул ниже контейнеров).

**Единая логика добавления/траты** — ключевая унификация лида: потратить
золото, съесть зелье, израсходовать 3 дерева на крафт, потратить опыт на
лвлап — ОДИН глагол `remove(def_id, count)` с проверкой достаточности и
одно `add(def_id, count)`. Цена/стоимость/требование где угодно в играх =
список `{def_id, count}` — магазин за золото и бартер за шкуры это один код.

API-скетч: `items_find/count/at, items_with_tag, item_core(def),
item_equip(def)?, item_use(def)?, item_is_currency(def),
items_instance_create/destroy/get, items_stack_rules`.

## 3. Контейнеры (внутри items)

Контейнер = упорядоченный набор владений (стаки и инстансы) + политика
вместимости. Рюкзак, сундук, стэш, buyback, **purse (скрытый, безлимитный,
для currency-предметов)** — ОДИН тип. API: контейнер create/destroy,
add/remove/move/count (применяет стак-правила), can_afford =
count(def_id) >= n, итерация. Правило приёма по умолчанию: блок currency →
purse, иначе рюкзак. Reconcile сирот после загрузки — внутри items (пул и
контейнеры в одной фиче). Идл-игра берёт items без UI рюкзака
(только purse-счётчики).

## 4. equipment: слоты-как-данные

Список слотов — данные игры (в items.json/схеме), enum и стейт-поля
генерятся. Никаких ручных 12 полей в схеме сейва и switch×3 (анти-паттерн
rb-dark-rpg). Слот = специализированный контейнер ёмкости 1 с фильтром по
equip.slot. equip/unequip = move между контейнерами.

## 5. progression: универсальные уровни (учебная L2-фича шаблона,
принято лидом 2026-07-06)

Трек = `{id, mode, кривая, max_level, on_level_up?}`. Треки объявляет игра
данными (tracks.json): hero, скиллы (woodcutting, magic_arrow), генераторы
идла, уровень игрока match-3.

**Одна ось — режим; режим определяет хранилище** (итерация лида
2026-07-06, снимает проблему «20 валют на 20 скиллов»):
- `threshold` — уровень НАКАПЛИВАЕТСЯ от действий (скиллы Скайрима, герой
  WoW): опыт = ВНУТРЕННИЙ счётчик трека (не предмет, не в каталоге, не в
  purse; это измеритель с одним стоком). Начисление:
  progression_add_xp(track, n) — из кода действий игры или use-эффекта
  («свиток опыта стрелы» = та же одна строка; появление свитка НЕ мигрирует
  хранилище). Лвлап автоматический в update-хуке. UI-бар читает
  progression_xp(track).
- `spend` — уровень ПОКУПАЕТСЯ (души за лвл, таланты, апгрейд генератора):
  трек не копит ничего, level_up тратит валюту из purse
  (`cost: {def_id, кривая}`); manual (кнопка) или флаг auto (идл-автобай).
  Валютами остаются только настоящие ресурсы игрока (души/золото/очки
  талантов: несколько источников И стоков, игрок оперирует как ресурсом).
- `on_level_up: [{def_id, count} | {track, xp}]` — эмиссия при лвлапе:
  валюта в purse (очки талантов ×3 за уровень героя) или xp другому треку
  (Скайрим-каскад: скиллы → xp героя). Намеренные циклы треков не
  проектировать.

**Кривая**: авторится как пресет (`{type: linear|exp|poly, params}`) ИЛИ
явная таблица; codegen допекает пресет в таблицу; рантайм всегда читает
таблицу + опциональный геометрический хвост (`tail: {growth}`) для
бесконечных уровней идла. Никакого интерпретатора формул в C. В редакторе
кривая всегда видна таблицей/графиком; операция bake: пресет → таблица для
поручной правки клеток («уровень 10 стоит ровно 500, там босс»).

**События лвлапа** (обратная связь без звонков наверх): threshold-лвлапы
и каскады резолвятся в update-хуке прогрессии раз в кадр; каждый лвлап
ЭМИТИТ событие **progression.levelup {track, old, new}** в ОБЩИЙ кадровый
лог событий (НЕ приватный буфер фичи; progression_poll_events() УМЕР).
Игра читает общий лог в своём апдейте (транспорт/ЖЦ-один-кадр/двухфазный
кадр — templates/design/event_system_design_2026-07-06.md §2/§3/§4) —
тосты/FX/пересчёт кэша статов. Производные статы предпочтительно
выводить при чтении (урон стрелы читает level в момент выстрела), кэш с
пересчётом по событию — только для тяжёлых формул. Для manual level_up
события не нужны — вызвавший знает результат.

- API: progression_level/xp/cost/can_up/level_up(track),
  progression_set_level(track, n) (кейс пролога) и progression_reset(track)
  (престиж: сброс threshold обнуляет внутренний xp) — ОБА примитива вместо
  level_down (Р6, synthesis §8); progression_add_xp(track, n, reason).
  progression_poll_events() УДАЛЁН — лвлап = событие общего лога (см. выше).
- Статы Diablo/Titan Quest = spend-треки (strength/dex/int/vit, cost =
  stat_point const 1, manual; hero.on_level_up эмитит поинты). Итоговая
  стата персонажа = база класса + level(трек) + сумма equip-статов —
  формулу собирает ИГРА (derive-on-read), прогрессия ведёт только
  бухгалтерию вложенного.
- Пререквизиты/гейты талантов — НЕ в фиче (решение лида 2026-07-06:
  «мы даём основу и базу»): поле requires у трека отсутствует осознанно.
  Дерево талантов игра строит поверх: узлы = её данные, гейт = game glue
  читает progression_level соседних треков/свои флаги перед level_up.
  Примитивов API для этого достаточно.
- Бесплатно из унификации: свиток опыта (use → add xp), опыт как награда
  квеста и как цена в магазине — без правок прогрессии.
- Апгрейд шмотки — НЕ трек (per-copy скаляр level на инстансе); треки для
  не-предметных уровней. «Levelable-предмет через каталог» отвергнут как
  god-абстракция.

Учебный набор шаблона: items + progression + HUD-бейдж (level/золото/xp —
заодно демонстрирует HUD-хинт валют). Shop в шаблон НЕ входит
(проектируется при первой реальной игре с магазином; транзакционное ядро
уже даёт items: оффер = товары + цена-список, buy = can_afford → remove →
add).

## 6. Сейв и миграции

По принятой модели «map по фичам»: `features.items` (пул инстансов +
контейнеры: рюкзак, purse, ...), `features.equipment` (слоты),
`features.progression` (треки: level, внутр. xp). Типизированные структуры
в рантайме, сериализация хуками фич.

Уточнения лида 2026-07-06 (вторая итерация): (а) стейт САМОЙ ИГРЫ — тоже
фрагмент по общему контракту (features.game: version+хуки+миграции; игра =
ещё один писатель); (б) сейв = конверт (шелл: schema-id, saved_at,
атомарность) + склад фрагментов, шелл содержимого не знает; (в) НОВАЯ
фича в обновлении ≠ миграция: нет ключа в документе → load-хук даёт
дефолты; удалённая фича = безвредный сирота-под-объект; (г) общий шаг
нужен только при переезде данных МЕЖДУ фрагментами: версия игры в
конверте, шаг по сырому JSON ДО пофрагментных load'ов, пишет игра.

Миграции — три уровня (решение лида 2026-07-06,
подтверждено: каждая фича отвечает за свои):
0. НЕ НУЖНА (частый случай, выигрыш def/instance-разреза): сейв хранит
   только id+count+per-copy скаляры; ребаланс/цены/кривые/иконки/новые
   предметы/новые поля с дефолтами сейв не трогают. Загрузчик толерантен:
   пропавший def_id → карантин, отсутствующее поле → дефолт из схемы.
1. САНТЕХНИКА (конвенция, не ручная проверка): под-объект фичи несёт
   version; load-хук фичи гонит цепочку шагов v1→v2→... по порядку;
   шелл просто зовёт load-хуки. Прецедент: features/game-state.
2. СЕМАНТИЧЕСКИЕ ШАГИ — пишет ИГРА (замена предмета, возврат за
   вырезанное, конверсия поинтов) в СВОЕЙ копии фичи
   (src/features/items/items_migrations.c) — copy-then-own делает это
   естественным, слоёвость не нарушена.
Привычка вместо машинерии: сейв-фикстура прошлой версии в tests/ фичи,
прогонять загрузку при правке схемы (учит скилл nt-game-items).

## 7. Редактор (T0316)

Уроки провала предыдущего редактора (сделан быстро, «для всего»):
универсальность достигается СХЕМОЙ (форма рендерится из схемы игры), а не
широтой скоупа; строить только после того, как модель данных живёт в
реальной игре. Предметы есть в любой игре (шмот/ресурсы/бустеры/валюты —
теперь и золото с опытом видны в том же каталоге с иконками) →
универсальный редактор оправдан.

- Студийный ВЕБ-инструмент рядом с canvas/gallery. НЕ внутриигровой
  (в игре const-таблица из codegen).
- Tool parity (решение лида 2026-07-06): единый op-слой items рождается
  ВМЕСТЕ С ФИЧЕЙ (не с редактором): list/get/validate/upsert/deprecate/
  icon-link/schema. CLI для агентов — сразу; веб — второй клиент поверх
  обкатанного API. Агенты правят каталог только через ops (не руками в
  JSON/C). deprecate вместо delete (исчезнувший def_id ломает сейвы).
- Скилл nt-game-items (мастер .codex/skills + зеркало): две половины —
  работа с базой через ops (блоки, стат-схема, иконки из галереи) и
  работа в игре через API фичи (инстанс только с контейнером-назначением,
  цены списком {def_id,count}, валюты через purse). Рождается с фичей.
- Фазы: (1) **read-only вьюер каталога с иконками** из галереи/атласа —
  «видеть предметы», самая дешёвая и нужная часть; (2) редактирование
  ядра+kinds с валидацией по схеме; (3) блочные формы (equip раскрывает
  слот+статы из схемы игры, use — параметры эффекта, currency — HUD-хинт).
- Единый источник: items.json + item_fields.schema.json. Редактор — view,
  никогда вторая модель данных. Редактор правит СТРОКИ данных, никогда
  схему (смена схемы = codegen/dev-операция) — иначе повтор «редактора для
  всего». UI-метаданные (лейблы, подсказки, виджеты, порядок секций) — в
  зарезервированном неймспейсе ui:{} схемы, codegen его игнорирует.

## 8. План

В шаблон (расширение T0327):
1. items: ядро+блоки (включая currency), codegen каталога, стаки int64 +
   пул инстансов + контейнеры + purse + bootstrap/вместимости (одна фича).
   Каталог шаблона — учебные предметы: золото (currency), зелье (use),
   меч (equip), материал (голый). Вместе с фичей: op-слой
   (list/validate/upsert/deprecate/icon-link/schema) + CLI + скилл
   nt-game-items. Событие items.txn в схеме фрагмента с рождения (§10).
2. progression: универсальные уровни (§5) — вторая учебная фича; событие
   progression.levelup в схеме; HUD-бейдж (level/золото/xp). Учебный shop =
   минимальный buy-only стаб поверх (цена = список {def_id, count});
   полноценный shop — при первой реальной игре с магазином.
3. equipment — при первой игре с экипировкой (слоты-как-данные с рождения).
4. Редактор T0316: вьюер после инкремента 1; редактирование — после того
   как схему потрогала реальная игра.

Extraction из rb-dark-rpg: машинерия grant/stack/equip — референс для
чистовой реализации (не копипаст: анти-паттерны не переносим).

## 10. Мутации, reason, экономические события (решения лида 2026-07-06; транспорт — event-док)

- **Два слоя мутаций**: API фичи = механизм+инварианты (can_afford, капы,
  стаки; про «открыт ли магазин» не знает НИКОГДА); глаголы игры =
  политика (гейты, FX, квест-триггеры). Прямая правка структур фрагментов
  запрещена (и механически затруднена: static + один хедер).
- **Обязательный reason в каждой мутации**: items_add/remove(def_id,
  count, reason) И progression_add_xp(track, n, reason) — с рождения.
  Формат verb:subject; verb из закрытого append-only списка (~20,
  src/features/items/reason_tags.h: loot, quest_reward, shop_buy,
  craft_cost, level_cost, use, starting, cheat, migration...);
  subject = существующий id. Debug-assert на verb, в релизе no-op.
  Прецедент: GameAnalytics resource events / Google Play Sources&Sinks.
- **Экономический журнал = тип события items.txn {def_id, delta, reason}**
  в ОБЩЕМ кадровом логе (НЕ кольцевой буфер в items — тот удалён; tick
  берётся из конверта события). flowType не хранится (знак delta);
  эмитится каждая смена владения (create/destroy/add/remove; equip/unequip
  — НЕТ, владение не меняется); корреляция покупки — пара (reason, tick),
  transaction-id не нужен. Тип объявлен в схеме фрагмента items → генератор
  даёт структ+emit+type_hash+дескриптор (event-док §3). Потребители дренят
  в том же кадре (транспорт — event-док §2/§4).
- **Локальная аналитика = потребитель ОБЩЕГО лога** (фильтр по items.txn),
  формат лида: агрегаты сессии (притоки/стоки по валютам) + JSONL при
  желании; native — файл в build/analytics/, web — в памяти (квоту не жжём).
  Внешний провайдер (GA) = будущий адаптер того же лога; durable-очередь +
  event_id — ТОЛЬКО если реально начнём отправлять (решение: локально-
  только). Реализация писателя — event-план E4.
- **DevAPI-потребитель**: экономический срез game.economy.* = типизированный
  фильтр поверх общей ленты game.events.tail (event-док §4; отдельного
  game.economy.journal-кольца нет): последние N транзакций + агрегаты →
  агенты-плейтестеры ассертят экономику числами (смоук: «после покупки —
  shop_buy:iron_sword, gold −50»; баланс-бот: скорость притока/стока за
  прогон).
- Лог событий пишет ВСЁ; агрегирует потребитель (не наоборот). Превью
  «нельзя купить, потому что...» — тот же сплит: can_afford у фичи,
  причины политики у игры.
- **ОБЩИЙ ПАТТЕРН ФИЧ (лид, 2026-07-06): глаголы + события-факты.** Фича =
  API-глаголы (команды с reason, инварианты, отказы) + эмиссия событий-
  фактов (транзакция items.txn; лвлап progression.levelup {track,old,new};
  xp threshold-треков журналится через progression_add_xp с reason с
  рождения) в ОБЩИЙ кадровый лог. Тип события объявляет своя фича в схеме
  своего фрагмента (генератор → структ+emit+type_hash+дескриптор);
  идентичность типа = type_hash имени, центрального реестра нет. Потребители
  (игра, локальная аналитика, DevAPI, будущий внешний адаптер) читают ОДИН
  лог, фильтруя по type_hash, — пер-фичевых потоков/курсоров/подписки-
  отписки НЕТ. Глобальный порядок = seq в конверте события (один счётчик в
  шелле, «xp, лвлап, xp» ≠ «xp, xp, лвлап»). Двухфазный кадр (react-
  фикспойнт → record), ЖЦ-один-кадр, переполнение-вслух (dev-warn при
  исчерпании арены; кольца «пропущено N» больше нет), generic-рендер по
  дескрипторам, отравление арены в debug — всё в event-доке
  (templates/design/event_system_design_2026-07-06.md §2/§4). Единого
  конверта-юниона по-прежнему нет, но универсальность даёт дескриптор
  (не адаптер на поток). События транзиентны (не сейв; истина = стейт;
  всё пропустивший ресинкается снапшотом стейта). Реакции игры на события
  могут рождать события — под капом поколений в фазе react. Сроки: items.txn
  — с рождения items (DevAPI-потребитель есть сразу); progression.levelup —
  с первой игрой с попапом.
