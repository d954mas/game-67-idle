## 0. Предпосылки (проверено по дереву HEAD 2026-07-07)

Проверенные факты, на которых стоит вся спека. Ссылки — `file:line`.

1. **Emscripten НЕ генерит HTML-шелл.** `emcc`-toolchain задаёт
   `CMAKE_EXECUTABLE_SUFFIX ".js"` (`$EMSDK/upstream/emscripten/cmake/Modules/Platform/Emscripten.cmake:281`),
   поэтому `add_executable(game …)` под EMSCRIPTEN даёт `game.js` + `game.wasm`, БЕЗ `game.html`.
   Шаблон сегодня НЕ несёт трекнутого шелла (`git ls-files templates/template` — ни одного `.html`).
   Единственный `index.html` в дереве — `templates/template/build/wasm-release/bin/index.html` —
   **ад-хок и .gitignored** (это «template wasm a3 check» localStorage-репортёр, не шелл), под
   `build/` → в новую игру НЕ едет (`new_game.mjs` SKIP-ит `build/`).

2. **Корень «file:///-стены» — АБСОЛЮТНЫЙ путь пака на web в CMake.** `main.c:71-73` уже дефолтит
   `GAME_ASSET_PACK_PATH` в ОТНОСИТЕЛЬНЫЙ `"assets/game.ntpack"`, но
   `templates/template/CMakeLists.txt:297-298` ПЕРЕОПРЕДЕЛЯЕТ его абсолютным
   `"${GAME_ASSETS_DIR}/game.ntpack"` (= `${CMAKE_BINARY_DIR}/bin/assets/game.ntpack`) ВКЛЮЧАЯ web.
   На web движок стримит пак по HTTP относительно URL страницы (`nt_resource_load_auto(s_pack_id,
   GAME_ASSET_PACK_PATH)` `main.c:424` → `nt_http`, fetch — `nt_http_web.c:21`), поэтому абсолютный
   Windows-путь = 404/запрет → **игра не грузит ассеты (не рендерит контент)**. Фикс — ЗЕРКАЛО rb-dark
   `5bbbff6f` (`games/rb-dark-rpg/CMakeLists.txt:396-405`): относительный путь на web через
   `GAME_PACK_RUNTIME_PATH`, пак кладётся плоским файлом рядом с `index.html`.
   **Уточнение причинности (ревью MED-4):** установка devapi-shim НЕ зависит от загрузки пака —
   `window.__devapi = {ready:true,…}` ставится СИНХРОННО в `nt_devapi_web_install_shim`
   (`external/neotolis-engine/engine/devapi/nt_devapi_web.c:19-21`), а fetch пака асинхронен. Живой
   round-trip требует ДВУХ независимых вещей: (а) относительный путь пака (эта карточка, для рендера) и
   (б) `--devapi <port>` в argv шелла, иначе shim вообще не запрашивается (blocker, §3.1). Карточное
   «игра не доходит до shim» = следствие того, что ад-хок страница T0328 не передавала `--devapi`, а не
   пак блокировал shim.

3. **Пак-билдер NATIVE-ONLY.** Оба CMakeLists собирают пак под `if(NOT EMSCRIPTEN)`
   (`templates/template/CMakeLists.txt:96-127`). На web пак НЕ строится → его копируют из native-сборки
   (rb-dark `tools/package_web.sh:13-14` копирует `build/native-debug/bin/assets/game.ntpack`).

4. **ГОЧА клоббера (критично).** Движок пишет статик-архивы в
   `${CMAKE_SOURCE_DIR}/build/engine/${NT_PRESET_NAME}` (`external/neotolis-engine/cmake/nt_module.cmake:44-45`;
   `CMAKE_SOURCE_DIR` = топ-левел = `templates/template`). Каталог ШАРИТСЯ всеми build-директориями
   с одинаковым `NT_PRESET_NAME`. Пресет-логика шаблона проверяет Release ПЕРВОЙ
   (`CMakeLists.txt:19-26`): **Release + DevAPI молча даёт `NT_PRESET_NAME=wasm-release`** (ветка Release
   выигрывает у devapi). Существующий `web_persistence_check.py` конфигурит именно так
   (`-DCMAKE_BUILD_TYPE=Release -DGAME_DEVAPI_ENABLED=ON`, `web_persistence_check.py:267-272`, build-dir
   `build/wasm-devapi-check`) → его engine-архивы с devapi-символами уезжают в `build/engine/wasm-release`
   и **КЛОББЕРЯТ** чистую человеческую `wasm-release`. Это и есть причина, по которой ночью не строились
   новые пресеты. Спека ОБЯЗАНА развести имена (см. §5) и убить Release+DevAPI-комбо (§2.1, guard).

5. **`web_persistence_check.py` сегодня НЕ доказывает shim — он СКИПАЕТ.** Он ищет `game.html`
   (`web_persistence_check.py:283,350-352`), которого emscripten не делает (п.1) → `raise Skip("no
   game.html shell")` → exit 2. И даже с шеллом он не проходил: (а) абсолютный путь пака (п.2) не давал
   игре рендерить, (б) шелл без `Module.arguments=['--devapi',…]` не запрашивал shim (blocker §3.1).
   Что из него РЕЮЗ: stdlib-`ThreadingHTTPServer` (`:288-293`), RFC6455 `MiniWebSocket`
   (`:113-189`), CDP-клиент `Cdp` c `wait_for_devapi`/`devapi_submit` (`:212-258`), `find_chrome`/
   `launch_chrome` (`--headless=new` + `--use-gl=swiftshader`, `:296-308`), `quit_chrome` (`Browser.close`,
   `:311-342`). **Именно этот скаффолдинг переиспользует headless-проба T0333** (§4). Что было доказано
   раньше («web-персистентность») — это ад-хок localStorage-репортёр (п.1), а НЕ devapi-shim round-trip;
   T0333 закрывает именно shim.

6. **Санитайзер-фикс уже в шаблоне.** `nt_set_sanitizer_flags(${GAME_TARGET})` (`CMakeLists.txt:318`) +
   web-devapi host-экспорты (`:250-253,266-271`) починили wasm-devapi ЛИНК (2af7a6cf5; INSTALL.md History
   `features/game-state/INSTALL.md:262-274`). **`wasm-debug` давно не собирался** (дерева нет:
   `build/wasm-debug/bin/` пуст; `build/engine/` не содержит `wasm-debug`). Санитайзер-фикс общий для всех
   Debug-модулей → должен покрыть и `wasm-debug`; спека ОБЯЗУЕТ проверить его линк первым же прогоном (§2.4).
   **Явно (ревью MED-15):** человеческий `wasm-debug` НЕСЁТ ASan/UBSan (`nt_set_sanitizer_flags` →
   `external/neotolis-engine/cmake/warnings.cmake:33-41`) — бандл большой и медленный на старте. Память:
   `nt_set_sanitizer_flags` (`warnings.cmake:43-49`, зовётся `CMakeLists.txt:318` ПОСЛЕ игрового
   link-блока `:254-265`) ставит `-sINITIAL_MEMORY=512MB` на Debug / `64MB` на Release; будучи ПОЗЖЕ на
   emcc-строке, 512MB перекрывает игровой безусловный `64MB` (`:264`) на Debug → Debug-wasm стартует с
   512MB (+`ALLOW_MEMORY_GROWTH`), ASan-shadow OOM не наступает (подтверждено обоими ревью). Это ожидаемо;
   выпил санитайзера из `wasm-debug` = ОТДЕЛЬНОЕ решение лида, в T0333 НЕ делаем. Лид не должен удивляться

7. **Память/heap уже починены в шаблоне** (`CMakeLists.txt:263-264`: `ALLOW_MEMORY_GROWTH=1` +
   `INITIAL_MEMORY=64MB` на линке экзешника). Это НЕ трогаем — уже перенесено из rb-dark.

8. **`new_game.mjs` копирует всё, кроме `build/_cache/node_modules/.git`** (`games/new_game.mjs:45`).
   Значит любые новые ТРЕКНУТЫЕ файлы вне `build/` (`web/`, `tools/`, `tests/`) едут в новую игру

9. **Готовые серверы (оценка реюза, директива «не плодить новый»):**

## 1. Goal / Non-goals

**Goal (done-when карточки T0333):** шаблон собирается под web и грузит паки по HTTP БЕЗ ручной
настройки, в трёх пресетах — человеческие `wasm-release` и `wasm-debug` (чистые, без devapi) и
агентский `wasm-devapi-debug`; одной командой сборки и одной командой сервера лид открывает игру в
браузере (пак грузится, нет file:///-ошибок); headless-проба доказывает `window.__devapi.submit/poll`
round-trip (`command.describe` отвечает) на devapi-пресете; всё это переезжает в новую игру через
`new_game.mjs` из коробки.

**Non-goals (явно ВНЕ T0333, не тянуть):**
- Zip-для-itch, web-load-smoke (console-error + first-frame pixel-health), `devapi/scenarios.py` skeleton —
  это **T0323** (`package_web.sh` + smoke + scenarios). См. §6 (граница скоупов).
- Web-бот как полноценный driver (site+agent parity браузерных прогонов) — отдельная будущая карточка.
- Правки движка. Движок READ-ONLY; фикс относительного пути — на СТОРОНЕ ШАБЛОНА (движок уже умеет
  относительный HTTP-стрим, п.2). Если исполнителю кажется, что нужна правка движка — СТОП, доклад лиду.
- Смена уже стоящих флагов памяти/санитайзера/экспортов (§0 п.6-7).

## 2. CMake (`templates/template/CMakeLists.txt`)

### 2.1 Относительный путь пака на web + guard от клоббера

Заменить блок `target_compile_definitions(... GAME_ASSET_PACK_PATH="${GAME_ASSETS_DIR}/game.ntpack" ...)`
(`CMakeLists.txt:297-302`) на ЗЕРКАЛО rb-dark (`games/rb-dark-rpg/CMakeLists.txt:396-408`): ввести
`GAME_PACK_RUNTIME_PATH` — относительный на web, абсолютный на native:

```cmake
if(EMSCRIPTEN)
    # На web движок стримит пак по HTTP относительно URL страницы -> путь ОТНОСИТЕЛЬНЫЙ,
    # пак ships плоским файлом рядом с index.html (tools/build_web.sh копирует из native).
    set(GAME_PACK_RUNTIME_PATH "assets/game.ntpack")
else()
    set(GAME_PACK_RUNTIME_PATH "${GAME_ASSETS_DIR}/game.ntpack")
endif()
target_compile_definitions(${GAME_TARGET} PRIVATE
    GAME_ASSET_PACK_PATH="${GAME_PACK_RUNTIME_PATH}"
    NT_INTROSPECT_ENABLED=$<BOOL:${NT_INTROSPECT_ENABLED}>
    NT_INTROSPECT_WRITE_ENABLED=$<BOOL:${NT_INTROSPECT_WRITE_ENABLED}>
    _CRT_SECURE_NO_WARNINGS
)
```

**Guard от Release+DevAPI-клоббера (§0 п.4).** ЕДИНЫЙ web-only блок ПОСЛЕ всей цепочки
`if(EMSCRIPTEN)…elseif…else…endif` (т.е. после `CMakeLists.txt:33`, НЕ между 26 и 27 — там внутренний
`endif` цепочки, вставка сломает `if`; и обязательно с обёрткой `if(EMSCRIPTEN …)`, иначе сработает и на
native, что противоречит §9 Q3). Фатал на комбо, которая молча алиасит `NT_PRESET_NAME=wasm-release` и
клобберит чистые человеческие engine-архивы:

```cmake
if(EMSCRIPTEN AND CMAKE_BUILD_TYPE STREQUAL "Release" AND GAME_DEVAPI_ENABLED)
    message(FATAL_ERROR
        "wasm Release+DevAPI не поддержан: NT_PRESET_NAME алиасится в wasm-release и клобберит "
        "чистые engine-архивы build/engine/wasm-release. Для devapi-web используйте Debug "
        "(пресет wasm-devapi-debug).")
endif()
```
(Тот же латентный футган есть и на native-стороне, но он вне скоупа T0333 — только отметить в §9 Q3.)

### 2.2 Доставка шелла в bin/ через `configure_file` + подстановка devapi-args (из коробки)

Под EMSCRIPTEN (рядом с EMSCRIPTEN-линк-блоком, напр. после `CMakeLists.txt:271`) генерить ОДИН
исходник шелла (§3.1) в `bin/`, подставляя ДВЕ переменные: заголовок `GAME_TITLE` (`CMakeLists.txt:34`,
дефолт `"Template"`; игра переопределяет `-DGAME_TITLE=…`) и **`GAME_DEVAPI_MODULE_ARGS`** —
`Module.arguments` для web-шима (см. §3.1 / blocker-1):

```cmake
if(EMSCRIPTEN)
    # blocker-1: web-shim ставится только если main() получил `--devapi <port>` в argv (main.c:135-136,
    # 331-332). На web argv задаётся ТОЛЬКО через Module.arguments. Один ИСХОДНИК шелла -> вывод
    # различается по пресету: devapi несёт флаг, человеческие пусты (guard §2.1 не даёт Release+devapi).
    if(GAME_DEVAPI_ENABLED)
        set(GAME_DEVAPI_MODULE_ARGS "arguments: ['--devapi', '17890'],")   # порт валиден 1..65535 (main.c:128);
                                                                            # на web TCP не поднимается (main.c:149-157)
    else()
        set(GAME_DEVAPI_MODULE_ARGS "")
    endif()
    # Трекнутый шелл -> bin/ на configure-time; заголовок из GAME_TITLE, args по пресету.
    configure_file("${CMAKE_CURRENT_SOURCE_DIR}/web/index.html.in"
                   "${GAME_OUTPUT_DIR}/index.html" @ONLY)
endif()
```
`@ONLY` — подставляются только `@GAME_TITLE@` и `@GAME_DEVAPI_MODULE_ARGS@`, JS-шаблонные `${…}` внутри
HTML не трогаются. `configure_file` идемпотентен, перегенерит при реконфиге. `GAME_OUTPUT_DIR` =
`${CMAKE_BINARY_DIR}/bin` (`CMakeLists.txt:90`). Шелл ссылается на `game.js`, который кладётся в тот же
`bin/` линком экзешника (`RUNTIME_OUTPUT_DIRECTORY`, `CMakeLists.txt:319`) — совпадают.
(`debug.html` вырезан из T0333 — ревью LOW-9, см. §11 Q6; console-smoke = дом T0323.)

### 2.3 Негативный инвариант «пресеты не смешиваются» (проверяется гейтом, не CMake-кодом)

Существующая логика уже корректна: `GAME_WEB_EXPORTS` не含 devapi без флага (`CMakeLists.txt:250-253`),
`nt_devapi_web` линкуется только под `GAME_DEVAPI_ENABLED` (`:281-292`), `NT_DEVAPI_ENABLED=0` в
человеческих (`:310-312`). НИЧЕГО менять не надо — но §9 (G6) добавляет НЕГАТИВНЫЙ гейт (grep exports + «нет `--devapi` в шелле»), чтобы
это доказать на артефакте.

### 2.4 Проверка линка `wasm-debug` (давно не собирался, §0 п.6)

Не правка CMake, а обязательный ПЕРВЫЙ прогон исполнителя: сконфигурить+собрать `wasm-debug`

## 3. Шелл (`templates/template/web/`) — новые ТРЕКНУТЫЕ файлы

### 3.1 `web/index.html.in` (человеческий шелл, зеркало rb-dark, генерализован)

Основан на проверенном на джеме `games/rb-dark-rpg/build/wasm-release/bin/index.html` (тот НЕтрекнут —
берём его как эталон и трекаем генерализованную версию). Заголовок — плейсхолдер `@GAME_TITLE@` (не
«Дракон…»). Статус «Загрузка…» — DOM, движковый text-renderer к нему НЕ применяется (инвариант про
шрифты не задет — это HTML-оверлей, не игровой текст).

```html
<!doctype html>
<html lang="en">
<head>
<meta charset="utf-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<title>@GAME_TITLE@</title>
<style>
  html, body { margin: 0; padding: 0; height: 100%; background: #0d0b09; overflow: hidden; }
  canvas { display: block; width: 100vw; height: 100vh; outline: none; }
  #status { position: absolute; top: 50%; left: 0; right: 0; text-align: center;
            color: #b8a68a; font: 16px sans-serif; pointer-events: none; }
</style>
</head>
<body>
<div id="status">Loading…</div>
<canvas id="canvas" oncontextmenu="event.preventDefault()" tabindex="-1"></canvas>
<script>
  var statusEl = document.getElementById('status');
  var Module = {
    canvas: document.getElementById('canvas'),
    @GAME_DEVAPI_MODULE_ARGS@
    onRuntimeInitialized: function () { statusEl.style.display = 'none'; },
    setStatus: function (t) { if (statusEl && t) statusEl.textContent = t; }
  };
</script>
<script src="game.js"></script>
</body>
</html>
```
`id="canvas"` + `Module.canvas` — контракт web-платформы движка (rb-dark так и делает).

**Blocker-1 (ревью HIGH-1) — web-shim требует `--devapi` в argv.** `nt_devapi_web_install_shim`
(`main.c:167`) зовётся ТОЛЬКО из `devapi_start()`, который ранним `return` выходит при
`!s_devapi_requested` (`main.c:135-138`); `s_devapi_requested` ставится ТОЛЬКО парсингом `--devapi <port>`
из argv (`main.c:331-332`). На web argv пуст, если шелл не задал `Module.arguments`. Поэтому строка
`@GAME_DEVAPI_MODULE_ARGS@` (§2.2) на devapi-пресете разворачивается в `arguments: ['--devapi', '17890'],`
(shim встаёт), на человеческих — в ПУСТО (shim не запрашивается). Один ИСХОДНИК шелла, вывод различается
подстановкой по пресету — «единый шелл» лида соблюдён; семантика зеркалит native (лаунчер передаёт флаг).
Осторожно: в non-devapi билде `--devapi` ушёл бы в `#else`-ветку `return 2` (`main.c:337-341`) → пустая
страница — поэтому человеческие пресеты обязаны разворачиваться в ПУСТО (гейт G6, §9, grep «нет `--devapi`
в сгенерированном человеческом `index.html`»). Пустое разворачивание оставляет валидный JS (просто нет
ключа `arguments`). Devapi-shim (`window.__devapi.ready=true`) ставится СИНХРОННО при инсталле
(`nt_devapi_web.c:19-21`), не зависит от загрузки пака.

**Почему `web/`, а не `tools/` или `src/`:** трекнутый источник вне `build/` → едет в новую игру
(`new_game.mjs`, §0 п.8); отдельная от кода директория — очевиден смысл; не под `build/` → не путается с
ад-хок артефактом (§0 п.1).

### 3.2 `debug.html` — ВЫРЕЗАН из T0333 (ревью LOW-9)

## 4. Команда сервера + команда сборки (`templates/template/tools/`) — новая ТРЕКНУТАЯ директория

Шаблон сейчас НЕ имеет `tools/` (проверено; T0323 это тоже отмечает). T0333 заводит `tools/` с ДВУМЯ
командами — «одна сборка, один сервер» из done-when. Обе едут в новую игру (§0 п.8).

### 4.1 `tools/serve_web.mjs` — команда сервера (Node, self-contained)

Тонкий статик-сервер (~50 строк, stdlib `node:http`), bind `127.0.0.1`. **MIME — ИНЛАЙН-таблица
(~12 строк) в самом `serve_web.mjs`** (ревью MED-8, перевёрнутый дефолт): `.html`, `.js`, `.wasm`
(`application/wasm` — критично для стрим-компайла), `.data`/`.ntpack` (`application/octet-stream`),
`.json`, `.png`, `.css`, `.wav`/`.mp3`/`.ogg`, дефолт `application/octet-stream`. Почему инлайн, а не
импорт `ai_studio/core_harness/tool_lib/mime.mjs`: игра должна быть SELF-CONTAINED и портируемой (дух
«из коробки») — импорт вшил бы в каждую копию зависимость на внутренности `ai_studio` (путь резолвится,
§0 п.8, но связность нежелательна). ~12 строк дубля MIME дешевле связности. Дефолт вынесен в §11 Q1
(ждёт лида, оба аргумента). Интерфейс:

```
node tools/serve_web.mjs [--preset wasm-release|wasm-debug|wasm-devapi-debug] [--port N] [--dir <bin>]
```
- `--preset` (дефолт `wasm-release` — человеческий по умолчанию для лида). Резолв директории:
  `<gameDir>/build/<preset>/bin`, где `<gameDir>` = `dirname(dirname(fileURLToPath(import.meta.url)))`
  (`tools/` → вверх на игру). `--dir` переопределяет.
- **Порт по пресету, чтобы человек и агент не конфликтовали одновременно:** дефолт `8080` для
  `wasm-release`/`wasm-debug`, `8081` для `wasm-devapi-debug`; `--port` переопределяет. (Дефолт-разведение
  портов = решение лида «разные порты или флаг».)
- Безопасность обхода директории (как `serve.mjs`/`serve_gallery.mjs`: `normalize`+containment check
  `startsWith(root+sep)`), 404 на выход за root.
- При старте: если `bin/assets/game.ntpack` отсутствует — печатать WARN «run tools/build_web.sh --preset
  <preset> first» (пак кладёт build-команда, §4.2), но всё равно поднять сервер.
- `Content-Type` из `mimeType(path)`, дефолт `application/octet-stream`. `/` → `/index.html`.

Стартовый лог: `serve_web: http://127.0.0.1:<port>/ (preset <preset>, dir <bin>)`.

### 4.2 `tools/build_web.sh` — команда сборки (bash/Git Bash, зеркало rb-dark минус zip)

Один прогон = собрать пресет и положить пак → сервер сразу отдаёт рабочую игру. Основан на
`games/rb-dark-rpg/tools/package_web.sh` (EMSDK `source /c/develop/emsdk/emsdk_env.sh`), но БЕЗ zip/itch
(это T0323). Интерфейс: `bash tools/build_web.sh [--preset wasm-release|wasm-debug|wasm-devapi-debug]`
(дефолт `wasm-release`). Пресет → флаги configure:

| preset | флаги configure |
|---|---|
| `wasm-release` | `-DCMAKE_BUILD_TYPE=Release` |
| `wasm-debug` | `-DCMAKE_BUILD_TYPE=Debug` (devapi по дефолту OFF, `CMakeLists.txt:10-12`) |
| `wasm-devapi-debug` | `-DCMAKE_BUILD_TYPE=Debug -DGAME_DEVAPI_ENABLED=ON` |

Шаги (все идемпотентны):
1. `GAME_DIR="$(cd "$(dirname "$0")/.." && pwd)"`. **EMSDK — переносимый поиск (ревью LOW-14):** уважать
   сначала `$EMSDK` (если выставлен — `source "$EMSDK/emsdk_env.sh"`), иначе `/c/develop/emsdk/emsdk_env.sh`
   (зеркало rb-dark), иначе положиться на `emcc` в `PATH`; все три под `… >/dev/null 2>&1 || true`.
   (Хардкод пути из rb-dark оставлен как fallback, но не единственный — помечено portability-wart.)
2. **Свежий native-пак (пак native-only, §0 п.3).** Если `build/native-debug` не сконфигурен —
   `cmake -S "$GAME_DIR" -B "$GAME_DIR/build/native-debug" -G Ninja -DCMAKE_C_COMPILER=clang -DCMAKE_BUILD_TYPE=Debug`;
   затем ВСЕГДА `cmake --build "$GAME_DIR/build/native-debug" --target game_asset_packs` (инкрементально →
   пак всегда свежий, снимает «не забудь dev_rebuild» из rb-dark).
3. **Web-сборка.** Если `build/<preset>` не сконфигурен — `emcmake cmake -S "$GAME_DIR" -B
   "$GAME_DIR/build/<preset>" -G Ninja <флаги-пресета>`; затем `cmake --build "$GAME_DIR/build/<preset>"
   --target game`.
4. **Копия пака плоским файлом** (зеркало `package_web.sh:13-14`): `mkdir -p
   "$GAME_DIR/build/<preset>/bin/assets"`; `cp "$GAME_DIR/build/native-debug/bin/assets/game.ntpack"
   "$GAME_DIR/build/<preset>/bin/assets/game.ntpack"`.
5. Печать: `built <preset>; serve with: node tools/serve_web.mjs --preset <preset>`.

Когда пересобирать пак: шаг 2 инкрементально пересобирает `game_asset_packs` при каждом прогоне, так
что после изменения ассетов достаточно снова запустить `build_web.sh`.

**Граница с T0323:** `build_web.sh` (сборка+копия пака) и `serve_web.mjs` — ОБЩИЙ ФУНДАМЕНТ. T0323
`package_web.sh` ВЫЗЫВАЕТ `build_web.sh --preset wasm-release` и добавляет zip+itch; T0323

T0333 их НЕ реализует.

## 5. Имена build-директорий и `NT_PRESET_NAME` (анти-клоббер, §0 п.4)

**Инвариант:** имя build-директории == `NT_PRESET_NAME` == имя `build/engine/<preset>`. Флаги configure
подбираются так, чтобы пресет-логика (`CMakeLists.txt:19-33`) дала именно это имя.

| build-dir | флаги | `NT_PRESET_NAME` | engine-архивы | devapi |
|---|---|---|---|---|
| `build/wasm-release` | `-DCMAKE_BUILD_TYPE=Release` | `wasm-release` | `build/engine/wasm-release` | нет |
| `build/wasm-debug` | `-DCMAKE_BUILD_TYPE=Debug` | `wasm-debug` | `build/engine/wasm-debug` | нет |
| `build/wasm-devapi-debug` | `-DCMAKE_BUILD_TYPE=Debug -DGAME_DEVAPI_ENABLED=ON` | `wasm-devapi-debug` | `build/engine/wasm-devapi-debug` | да |

**Не бьют существующие:** `NT_PRESET_NAME` выводится из build-type+devapi, а НЕ из имени build-директории
(ревью LOW-12). native-презеты (`native-debug`/`native-release`/`devapi-debug`) резолвятся в свои
`build/engine/<preset>`. `feature-review-*` НЕ создают новых engine-каталогов: `feature-review-devapi`
(Debug+devapi) делит `build/engine/devapi-debug` с `devapi-debug`; `feature-review-no-state` (Debug,
devapi ON) — туда же; `feature-review-release` (Release) — `build/engine/native-release`. Это КОРРЕКТНО
(одинаковые флаги → идентичные архивы, не клоббер). Три wasm-имени разведены между собой и с native. Ни
одна пара с РАЗНЫМИ engine-флагами не делит каталог — единственным нарушителем был Release+devapi
(`wasm-devapi-check` → `wasm-release`), убран здесь.

**Убрать источник клоббера:** ретайрить `build/wasm-devapi-check` (Release+devapi → `wasm-release`,
клоббер). headless-проба (§7) и `web_persistence_check.py` теперь строят **Debug+devapi →
`wasm-devapi-debug`** (§0 п.6 разрешает: санитайзер-фикс дал Debug-wasm линк) в `build/wasm-devapi-debug`
— свой engine-каталог, ноль клоббера. Guard §2.1 не даст случайно вернуть Release+devapi.

## 7. Headless-проба devapi (закрывает хвост T0328) — `templates/template/tests/web_devapi_check.py`

Новый скрипт, advisory (как `web_persistence_check.py`: exit 0=PASS / 1=FAIL / 2=SKIP при отсутствии

`web_persistence_check.build_wasm_devapi` КАНОНИЧЕСКИМ билд-хелпером и импортировать его в
`web_devapi_check.py` — обе пробы зовут ОДНУ функцию, не копипастят. Хелпер (после фикса ниже):
конфигурит `build/wasm-devapi-debug` c `-DCMAKE_BUILD_TYPE=Debug -DGAME_DEVAPI_ENABLED=ON` (НЕ Release —
пресет `wasm-devapi-debug`, свой engine-каталог, §5), `cmake --build --target game`; затем гарантирует
пак — собрать `game_asset_packs` в `build/native-debug` (native) и скопировать
`build/native-debug/bin/assets/game.ntpack` → `build/wasm-devapi-debug/bin/assets/game.ntpack`; вернуть
`bin_dir`. (Остаётся ОДИН bash-путь `build_web.sh` для людей/CI и ОДИН python-путь `build_wasm_devapi`
для обеих проб — два, не три; bash↔python-дубль ~5 строк допустим: у них разные вызыватели.) Шелл кладёт
`configure_file` (§2.2) при configure, с `arguments:['--devapi','17890']` (devapi-пресет).

**Сценарий (done-when «`command.describe` отвечает»; ФИКС ревью HIGH-2 — реальный контракт движка):**
1. `bin_dir = build_wasm_devapi(...)`; `serve_dir(bin_dir, port)`; `launch_chrome(chrome,
   "http://127.0.0.1:<port>/index.html", cdp_port, profile)` (грузим `index.html`, НЕ `game.html`, §0 п.5;
   шелл сам передаёт `--devapi` через `Module.arguments`, blocker-1/§3.1).
2. `cdp.wait_for_devapi()` — ждёт `window.__devapi.ready`. **Причинность (ревью MED-4):** `ready=true`
   ставится СИНХРОННО при инсталле shim (`nt_devapi_web.c:19-21`), НЕ после загрузки пака — проба
   доказывает ТОЛЬКО живой транспорт shim, а НЕ pack-over-HTTP (его доказывает человеческий G1).
3. **Два дешёвых шага против РЕАЛЬНОГО контракта discovery** (`external/neotolis-engine/engine/devapi/
   nt_devapi_discovery.c`): `command.describe {}` вернул бы `bad_params "missing params.method"`
   (`:53-60`), и `command.describe` отдаёт 7-полевой СЛОВАРЬ-дескриптор, НЕ список — исполнителю НЕ чинить
   движок, это by design. Поэтому:
   - `r1 = devapi_submit("endpoints", {})` → ассерт `r1["ok"] is True` И `r1["result"]["commands"]`
     непуст (`endpoints` — источник СПИСКА команд, `:43-48,105-114`).
   - `r2 = devapi_submit("command.describe", {"method": "endpoints"})` → ассерт `r2["ok"] is True` И в
     `r2["result"]` есть поля дескриптора (`method`/`group`/`summary`/`params_shape`/… `:69-75`). Это
     буквально закрывает карточное «`command.describe` отвечает».
   Оба идут через `window.__devapi.submit` (тот же транспорт, что `poll` дренит) → round-trip доказан.
4. **Дёшево (ревью MED-4): собрать консоль через CDP** (`Log.enable`/`Runtime.consoleAPICalled`) и
   ассертить, что в devapi-прогоне НЕТ `404`/`file:///` по URL пака — быстрый вторичный сигнал, что
   относительный путь/копия не сломаны (не заменяет G1, но ловит регресс дёшево).
5. PASS/FAIL + `quit_chrome`. Профиль временный (не нужен persistent — это НЕ тест persistence).

**Таймауты под ASan-Debug (ревью MED-5).** `wasm-devapi-debug` несёт ASan/UBSan (§0 п.6) → старт
медленный. `wait_for_devapi` и CDP-таймауты пробы поднять до **60-90 s** (дефолт персистенс-чека 20 s мал
для Debug-wasm). SKIP по таймауту — известный риск стенда, не FAIL.

**Гейтинг:** advisory (как персистентность — headless капризен). НЕ в `ctest`. Документируется в README и
INSTALL.md как «одна команда — сигнал». Прогон закрывает done-when «живой shim» и хвост T0328 (ссылкой на

**Заодно ПОЧИНИТЬ `web_persistence_check.py` (убрать клоббер + оживить):** переписать `build_wasm_devapi`
(`:262-285`) на КАНОНИЧЕСКИЙ хелпер выше — Debug+devapi в `build/wasm-devapi-debug` (было Release в

Негативные (пресеты НЕ смешиваются, §2.3):
- **G6 (человеческий чист от devapi + НЕТ `--devapi` в шелле):** (а) в `build/wasm-release/bin/game.js` и

  **(ревью HIGH-1) в СГЕНЕРИРОВАННОМ `build/wasm-release/bin/index.html` НЕТ подстроки `--devapi`**

- **G7 (агентский НЕСЁТ devapi):** в `build/wasm-devapi-debug/bin/game.js` присутствует

- **G8 (анти-клоббер):** после СБОРКИ всех трёх пресетов (engine-архивы материализуются на СБОРКЕ, а не на

## 11. Open questions (с дефолтами)

- **Q0 — как web-шелл запрашивает devapi-shim (blocker-1, ревью HIGH-1). Ждёт РАТИФИКАЦИИ лида.**
  ДЕФОЛТ (вариант А, §2.2/§3.1): вторая подстановка `configure_file` `@GAME_DEVAPI_MODULE_ARGS@` — на
  devapi-пресете шелл несёт `Module.arguments = ['--devapi','17890']`, на человеческих пусто. Один
  ИСХОДНИК шелла (лид «один шелл» соблюдён), вывод различается по пресету; зеркалит native-семантику
  (лаунчер передаёт флаг). Гейт G6 стережёт «нет `--devapi` в человеческом index.html». Альтернатива Б
  (ОТКЛОНЕНА дефолтом): авто-`s_devapi_requested=true` на web при `NT_DEVAPI_ENABLED` в template `main.c`
  (без argv-флага). Почему Б отклонена: правит `main.c` шаблона (шире, чем шелл), размывает «devapi
  включается флагом» инвариант, и на человеческих пресетах `NT_DEVAPI_ENABLED=0` всё равно не даст shim —
  т.е. Б не проще А, но инвазивнее. **Вынесено на лида, т.к. это правка семантики бута.**
- **Q1 — serve_web: ИНЛАЙН MIME vs импорт `mime.mjs` (ревью MED-8, дефолт ПЕРЕВЁРНУТ). Ждёт лида (LOW).**
  ДЕФОЛТ: **инлайн** ~12-строчная MIME-таблица в `serve_web.mjs` → игра self-contained и портируема
  (дух «из коробки»). Аргумент ЗА импорт `../../../ai_studio/core_harness/tool_lib/mime.mjs`: DRY + уже
  тестирован; путь резолвится и в шаблоне, и в копии (§0 п.8). Аргумент ЗА инлайн (выбран): импорт вшивает
  в КАЖДУЮ копию игры зависимость на внутренности `ai_studio` (ломается, если игру вынесут из репо;
  связность нежелательна для «из коробки»). ~12 строк дубля дешевле связности. **Реальный tradeoff → лид.**
- **Q2 — serve_web на Node, проба на Python (два рантайма).** ДЕФОЛТ: да. serve_web = Node (гарантирует
  `application/wasm` инлайн-таблицей, единая с `new_game.mjs` экосистема); проба = Python (реюз готового
  CDP-скаффолдинга `web_persistence_check.py`, один self-contained вызов для CI). Python-`http.server` в
  самой пробе (`serve_dir`) отдаёт `.wasm` через `mimetypes` — на HEADLESS-пробе fallback-компайл терпим;
  для ЖИВОГО лида важнее Node-MIME. **Не спорно → дефолт.**
- **Q3 — guard от Release+DevAPI только на web.** ДЕФОЛТ: FATAL только под `if(EMSCRIPTEN)` (§2.1). Тот же
  латентный клоббер есть на native (Release+devapi → `native-release`), но native так никто не собирает и
  это ВНЕ скоупа T0333. **Ждёт лида (LOW):** распространять ли guard на native (одна строка). Дефолт —
  не трогать native в этой карточке.
- **Q4 — порты по умолчанию 8080/8081.** ДЕФОЛТ: `wasm-release`/`wasm-debug`→8080, `wasm-devapi-debug`→8081
  (человек и агент не конфликтуют одновременно; `--port` переопределяет). **Не спорно → дефолт.**
- **Q5 — чинить ли `web_persistence_check.py` в рамках T0333.** ДЕФОЛТ: да, минимально (§7) — его текущий
  Release+devapi build АКТИВНО клобберит `build/engine/wasm-release` (§0 п.4), т.е. это баг в зоне
  поражения T0333, а не отдельная работа. Правка мелкая (build-type + грузить index.html + копия пака +
  таймауты). Оба ревью подтвердили: фикс в зоне T0333, НЕ scope-creep. **Не спорно → дефолт.**
- **Q6 — `debug.html`: в T0333 сейчас или отдать T0323 (ревью LOW-9, дефолт ПЕРЕВЁРНУТ). Ждёт лида (LOW).**
  ДЕФОЛТ: **вырезано из T0333, передано T0323** (§3.2) — console-оверлей не привязан ни к одному done-when
  T0333, а console-error-smoke = дом T0323; lean-фильтр. Если лид хочет дешёвый ручной диагностик-шелл
  прямо в T0333 — вернуть `web/debug.html.in` + второй `configure_file` (одна строка).
