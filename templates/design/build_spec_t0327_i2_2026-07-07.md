## 1. Goal / Non-goals

**Goal.** В `templates/template` появляется ОДНА фича `items` (L1) —
`src/features/items/` с одним публичным `items.h` (`// feature-layer: L1`):
каталог (ядро + блоки equip/use/currency, данные-only, items НИКОГДА не исполняет
эффекты) с компайл-тайм const-таблицей из `content/items.json`; владение — стаки
int64 + пул уник-инстансов + контейнеры с инвариантом «инстанс ровно в одном
контейнере» в ОДНОМ месте; purse = скрытый безлимитный контейнер валют
(gold/xp = currency-предметы, HUD-hint флаг); bootstrap контейнеров и стартовых
валют из данных (accept-policy enum); сейв-фрагмент `items` (третий) через
генератор `--fragment items`, толерантный load + карантин сирот + `on_new_game`;
типизированное событие `items.txn` с рождения; read-only op-слой (list/validate/
schema) как Python-CLI + скилл `nt-game-items`. Предваряет всё под-инкремент **И2-0**:
ось `FEATURE_GAME_STATE` снимается целиком (решение лида 2026-07-07, §2.4/§5) —
стейт всегда on, items = одна версия. Сборка native+web зелёная, ctest (9 + новые)
зелёный, live smoke зелёный, items виден в `game.state.schema/get`.

**Non-goals — явно ВНЕ И2 (не тянуть, констатировать в коде/доке):**
- **progression, resource_panel, coin_fx, ЛЮБОЙ HUD/UI** — И3+. items в И2 —
  API-фича БЕЗ участия в кадровых фазах (нет `<id>_draw_ui/update`), без арта
  (только `art_needs`-декларация, когда появится реальный потребитель).
- **shop** — вне шаблона (карточка: «магазин слишком специфичен»).
- **equipment (фича)** — при первой игре с экипировкой. Слоты-как-ДАННЫЕ в

## 2. Архитектурное решение: модель владения (ЯДРО спеки — читать первым)

Единственное нетривиальное решение И2. Всё остальное — механика вокруг него.

### 2.1 Проблема

Дизайн §1/§3 хочет: (а) пул уник-инстансов `instance_id → {per-copy скаляры}`;
(б) НАБОР контейнеров (рюкзак, purse, …), КАЖДЫЙ — упорядоченный массив записей
`{id, count:i64}`; (в) набор контейнеров и вместимости — из ДАННЫХ (items.json),
не из схемы. Генератор `--fragment` (§0 п.3) объектные типы держит ТОЛЬКО из
скаляров — «контейнер как объект со вложенным массивом записей» НЕ выразим, а
«по map-полю на контейнер» делает набор контейнеров СХЕМНЫМ (компайл-тайм), что
рушит «контейнеры из данных».

### 2.2 Решение — ПЛОСКАЯ таблица владения (generator-fit, ноль хирургии генератора)

**Сейв-фрагмент `items` хранит ОДНУ плоскую карту владения `owned:
map<string,ItemOwned>`, где `ItemOwned` несёт поле `container` (id контейнера как
строка). Контейнер — это НЕ строка сейва, а РАНТАЙМ-вид (фильтр `owned` по
`container`) поверх const-определений контейнеров из каталога. «Инстанс ровно в
одном контейнере» = у каждой записи ровно одно значение `container`, и менять его
может ТОЛЬКО `items_move()` — единственная точка (инвариант в одном месте).**

Почему это корректно и лучше альтернатив:
- **Разрез def/instance держится** (дизайн §2/§6, миграции уровень-0): сейв = только
  `id + count + per-copy скаляры + container-тег`. Ребаланс/цены/вместимости/новые
  предметы сейв не трогают.
- **Набор контейнеров = ДАННЫЕ.** Определения контейнеров (`id, capacity,
  accept_policy`) — const-таблица каталога (`content/items.json`, §0 п.10), НЕ
  схема сейва. Новый контейнер (сундук/стэш) = строка каталога, БЕЗ правки схемы.
- **purse** = контейнер каталога с `accept_policy=currency_only, capacity=∞`; его
  currency-записи живут в той же `owned` с `container="purse"`. Валюта — обычный
  предмет (add/remove тем же глаголом), i64 count.
- **Переиспользует готовый образец** `ItemInstance`
  (`game_state.schema.json:16-24`) — риск формы уже снят ревью.
- **Карантин** = поле `quarantined:bool` на записи (персистентно, переживает
  round-trip, «возвращается при реставрации def» — reconcile снимает флаг, когда
  def снова в каталоге; запись сохраняет свой `container`-дом). Дешевле параллельного
  `list<string>` limbo (нечего синхронизировать).
- **items.txn** типизированное событие получаем БЕСПЛАТНО из той же схемы (секция
  `events`).

Что теряем и почему это ОК в И2: явный ПОРЯДОК записей внутри контейнера (порядок =
данные UI/слотов, дизайн §1). В И2 нет UI → порядок не нужен; это забота И3
(resource_panel/inventory). Byte-stable round-trip держится: `to_json` идёт по
слотам массива 0..N, load аллоцирует в порядке документа в те же слоты (§0 п.3,
`render_collection_helpers` map-путь).

### 2.3 Ключевая конвенция (единственная «магия», задокументировать в items.h)

Ключ карты `owned`:
- уникальный инстанс (есть `equip`-блок → per-copy стейт): `key = instance_id`
  (аллоцируется `items_instance_create`, формат `<def_id>#<seq>`). **ПОЛЕ
  `.container` авторитетно** (ключ контейнер не кодирует).
- стак (стакуемый/валюта): `key = "<container>/<def_id>"` — гарантирует «один стак
  на def_id внутри контейнера» уникальностью ключа; запросы `count(container,def)` —
  прямой lookup/линейный скан. **КЛЮЧ авторитетен: поле `.container` ДОЛЖНО совпадать
  с префиксом ключа (M2).** Следствие: `items_move` стака = СТРОГИЙ remove+add с
  ре-кеингом (сменить только поле — ключ рассинхронится); `items_add` всегда пишет
  `.container = target`. Инвариант «field == key-prefix» покрыт тестом (G8).

Ключ детерминирован (round-trip-стабилен). `string_max` фрагмента (64) вмещает
`<container>/<def_id>` — op-слой ЖЁСТКО линтует `len(container)+1+len(def_id) ≤ 63`
(§8.3, L5/code-review L1).

### 2.4 Ось `FEATURE_GAME_STATE` УМЕРЛА (решение лида 2026-07-07)

Лид: «игра без стейта невозможна» → ось `FEATURE_GAME_STATE` снимается ЦЕЛИКОМ
(под-инкремент **И2-0**, §5, лендится ПЕРЕД И2a). Стейт всегда включён. Следствия
для items: **items — ОДНА версия, БЕЗ единого `#if FEATURE_GAME_STATE`**; каталог +
владение + фрагмент компилируются БЕЗУСЛОВНО. Прежний дизайн «catalog-only под
`!FEATURE`» и OQ4 ОТМЕНЕНЫ этим решением (см. §12 OQ4-note). `GAME_DEVAPI_ENABLED`-
гейтинг НЕ трогаем (остаётся). Раскладка папки зеркалит settings, но без
FEATURE-гейта.

Рукописные типы каталога (структ, НЕ codegen — дизайн §2) + каталог-API + владение-API

```c

#define FEATURES_ITEMS_H
// feature-layer: L1
/* Единственный публичный хедер фичи items — вся публичная поверхность; остальное
   в папке static. L1 foundation: зависит только от L0-шелла (game_save-toolkit +
   gsj_ + движок), НЕ от других фич. Владение ведётся в int64 ВЕЗДЕ (валюты тоже
   int64, НЕ double; большие счётчики в JSON — строкой, §14 п.8). Дробное
   производство копит аккумулятор в game glue, НЕ в count (Р1; см. README). */
#include <stdbool.h>
#include <stdint.h>

/* ---- Каталог (const-таблица из content/items.json; компилируется ВСЕГДА) ---- */
typedef enum { ITEM_ACCEPT_ANY = 0, ITEM_ACCEPT_CURRENCY_ONLY, ITEM_ACCEPT_SLOT_FILTER,
               ITEM_ACCEPT_CAPACITY_1 } item_accept_policy_t;

typedef struct { const char *effect_id; /* плоские use-params — по мере нужды */ } item_use_block_t;
typedef struct { const char *slot; } item_equip_block_t;
typedef struct { const char *hud_hint; int64_t cap; /* 0 = безлимит */ } item_currency_block_t;

typedef struct game_item_def_t {
    const char *id;            /* <namespace>.<slug>; НИКОГДА не ключуемся по display_name */
    const char *display_name;  /* витрина; НЕ ключ (греп-гейт §10) */
    const char *icon_asset_id; /* логический id; хендл арта даёт игра (И3) */
    const char *kind;          /* витрина-категория (item_kinds) */
    const char *const *tags; int tag_count;
    int64_t base_value;
    bool stackable; int64_t max_stack; bool unlimited;
    const item_equip_block_t *equip;      /* NULL если нет блока */
    const item_use_block_t *use;          /* NULL если нет блока */
    const item_currency_block_t *currency;/* NULL если нет блока */
} game_item_def_t;

typedef struct game_container_def_t {
    const char *id; int64_t capacity; /* 0 = безлимит */ item_accept_policy_t accept_policy; bool hidden;
} game_container_def_t;

const game_item_def_t *item_core(const char *def_id);         /* NULL если неизвестен */
const game_item_def_t *item_at(int index); int items_def_count(void);
int  items_with_tag(const char *tag, const game_item_def_t **out, int out_cap);
bool item_is_currency(const game_item_def_t *def);
const game_container_def_t *item_container_def(const char *container_id);

/* ---- Владение (поверх генерируемого фрагмента items_state) ----
   Единый глагол add/remove: потратить золото / съесть зелье / израсходовать
   3 дерева / потратить опыт — ОДИН код. reason обязателен с рождения (§10):
   формат verb:subject, verb из закрытого списка reason_tags.h (debug-assert).
   L1-нота: для СТАКОВ per-copy поля level/durability — игнорируемые дефолты
   (плоская форма под генератор); смысловы только для УНИКОВ (equip-блок). */
bool    items_add(const char *container_id, const char *def_id, int64_t count, const char *reason);
bool    items_remove(const char *container_id, const char *def_id, int64_t count, const char *reason);
int64_t items_count(const char *container_id, const char *def_id);
bool    items_can_afford(const char *container_id, const char *def_id, int64_t n);
bool    items_move(const char *from, const char *to, const char *entry_key, int64_t count, const char *reason);
const char *items_instance_create(const char *container_id, const char *def_id, const char *reason); /* уник -> instance_id */

/* purse-удобства: валюты по умолчанию едут в purse (accept-policy). */
int64_t items_purse(const char *def_id);

#endif /* FEATURES_ITEMS_H */
```

Точные сигнатуры уточняются исполнителем по потреблению тестов; ключевое — API

### 6.5 `state/items.schema.json` (фрагмент; `--fragment items`)

```json
{
  "schema": "game_seed.items",
  "schema_version": 2,
  "fragment": "items",
  "version": 1,
  "string_max": 64,
  "hooks": { "on_new_game": true, "reconcile": true },
  "types": {
    "ItemOwned": {
      "kind": "object",
      "fields": {
        "def_id":      { "type": "string", "max_length": 63 },
        "container":   { "type": "string", "max_length": 63 },
        "count":       { "type": "i64", "default": 1, "min": 0, "max": 9000000000000000000 },
        "level":       { "type": "int", "default": 1, "min": 1, "max": 9999 },
        "durability":  { "type": "float", "default": 1, "min": 0, "max": 1 },
        "quarantined": { "type": "bool", "default": false }
      }
    }
  },
  "fields": {
    "owned": { "type": "map<string,ItemOwned>", "max_count": 64 }
  },
  "events": {
    "txn": {
      "fields": {
        "def_id": { "type": "string" },
        "delta":  { "type": "i64" },
        "reason": { "type": "string" }
      }
    }
  }
}
```

Заметки: `count.min=0` (толерантно к транзиентному/идл-0, без churn слотов;
`items_remove` до 0 освобождает слот, чтобы не копить cruft — рантайм-политика, не
схема). `version=1` + НЕТ секции `migrations` = скелет (генератор требует
`version==len(migrations)+1`, §0 п.3). Хуки `on_new_game/reconcile=true` → генератор

## 7. И2b — владение + контейнеры + purse + bootstrap + карантин + событие

### 7.1 `items_containers.c` — владение над `items_state` (линейный скан по owned[])

Оперирует генерируемым инстансом `items_state` (тип `ItemsState`, `items_state.h`).
**КРИТИЧНО (code-review #12/M3): генерируемые `find_owned`/`alloc_owned` — `static`
ВНУТРИ `items_state.c`** (`render_collection_helpers`, §0 п.3) → из другого TU
НЕДОСТУПНЫ. `items_containers.c` СКАНИРУЕТ `items_state.owned[]` НАПРЯМУЮ: тип записи
`ItemsItemOwned` (генератор: `{Pascal}{TypeName}`), массив `[ITEMS_STATE_MAX_OWNED]`,
поля `.used / .key[ITEMS_STATE_STRING_MAX] / .def_id / .container / .count(i64) /
.level / .durability / .quarantined` (точные имена — из сгенерированного
`items_state.h`). Аллокация = найти первый `!used` слот, заполнить key/def_id/
container/count. Освобождение = `memset(&slot,0,…)` (used→false).

Правила:
- **Ключ и авторитет (§2.3, M2):** стак — `key = "<container>/<def_id>"`, **KEY
  АВТОРИТЕТЕН** (поле `.container` ДОЛЖНО совпадать с префиксом ключа — инвариант,
  тест G8); уник — `key = instance_id`, **ПОЛЕ `.container` авторитетно** (ключ не
  кодирует контейнер). Один хелпер строит ключ (R3).
- **accept_policy (L6):** `currency_only` (purse) отвергает не-currency; `any`
  (backpack) берёт всё. **`slot_filter` и `capacity_1` в И2 ИНЕРТНЫ** — объявлены в
  enum, но `items_add` трактует их как `any` до эпохи equipment.
- **capacity — REJECT, НЕ clamp (M1):** `container.capacity` = максимум РАЗЛИЧНЫХ
  записей владения в контейнере (число слотов, НЕ сумма count; `capacity=0` =
  безлимит). Enforcement ТОЛЬКО в `items_add` при аллокации НОВОЙ записи: если
  контейнер уже держит `capacity` различных записей И это НОВЫЙ `(container,def)` →
  **return false (REJECT)**, НИКОГДА не клампить. Рост существующего стака (тот же
  ключ) вместимостью НЕ ограничен.
- **currency.cap — CLAMP (M1/M3):** `currency.cap` из каталога (`cap>0`) КЛАМПИТ
  СУММУ count при add валюты: add сверх cap → `count=cap` (НЕ reject). `cap=0` =
  безлимит.
- **add(container,def,count,reason):** дефолт-контейнер по accept-policy (currency →
  purse, иначе backpack — дизайн §3), явный `container` уважается; reason_check;
  accept_policy → capacity(new record) REJECT → find-or-alloc → `count += count` →
  currency.cap CLAMP → `game_save_mark_dirty()` → `items_emit_txn(def, +Δ, reason)`
  (Δ = фактически добавленное после клампа).
- **remove(container,def,count,reason):** `can_afford` (count>=n) иначе false;
  вычесть; при 0 — освободить слот; mark_dirty; emit `txn(def, −count, reason)`.
- **move(from,to,key,count,reason):** ЕДИНСТВЕННАЯ точка смены контейнера (инвариант
  §2.2). Для стака = СТРОГИЙ remove+add с ре-кеингом (ключ несёт container → менять
  только поле НЕЛЬЗЯ, ключ рассинхронится; M2); для уника = смена поля `.container`.
  НЕ эмитит txn (владение def не меняется). mark_dirty.
- **instance_create(container,def,reason):** аллоцирует `<def>#<seq>` (seq —
  static-счётчик, в тесте сидировать для детерминизма), `count=1`, per-copy дефолты;
  emit `txn(def, +1, reason)`. Уник существует независимо от контейнера (пул ниже
  контейнеров — дизайн §2).
- **purse(def):** `count("purse", def)`.
- **can_afford/count:** чистое чтение (const state), карантинные записи ИСКЛЮЧЕНЫ,
  без mark_dirty.

### 7.3 `items_bootstrap.c` — on_new_game + reconcile (Р9 exercise; ТЕЛА в И2b, стабы в И2a)

**Точная семантика reset vs on_new_game (Р9):**
- `frag_reset` → `items_state_init_defaults(&items_state)` (генерится): memset →
  `owned` ПУСТ. Нейтральный, без контента. Зовётся: свежий-load ДО on_new_game;
  плохой/отсутствующий фрагмент при load (падает в пустой, остальные грузятся);
  `--fresh-state`. Семантика: «пустой инвентарь, ноль стартового».
- `items_on_new_game(void)` (рукописный, дескриптор зовёт по hook): ТОЛЬКО на
  генуинно свежем сейве (файла нет) ПОСЛЕ reset, через `game_save_new_game`/
  fresh-load (§0 п.2). Сеет СТАРТОВЫЙ контент демо: `items_add("purse","tmpl.gold",
  50,"starting:new_game")` + `items_add("backpack","tmpl.potion",1,"starting:
  new_game")`. Семантика: «стартовый инвентарь новой игры». Возвращающийся игрок
  идёт через `from_json` (не on_new_game) → стартовое НЕ переприменяется. Числа
  демо — §OQ5 (минорно).
- `items_reconcile(void)` (рукописный, hook): пост-load фиксап (§0 п.2, порядок
  load = version-steps → reconcile → карантин, §4 синтеза). Скан `owned`: если
  `item_core(rec.def_id)==NULL` (def исчез из каталога) → `rec.quarantined=true`
  (не удалять — порча сейва иначе, дизайн §9). Если ранее карантинная запись снова
  валидна (`item_core != NULL`) → `quarantined=false` (реставрация def, §4). Запись
  хранит свой `container`-дом весь цикл. Карантинные записи ИСКЛЮЧЕНЫ из
  count/can_afford (живые запросы), но переживают round-trip (`quarantined:bool`
  персистентен).

## 8. И2c — op-слой CLI + скилл (read-only, tool parity)

### 8.1 `tools/items_ops.py` — read-CLI (Python; §OQ3)

Единый op-слой, на котором СИДИТ будущий веб-редактор T0316 (tool parity). read-v1
(LEAN §3: сначала read; upsert/deprecate — редакторская эра). Интерфейс:

```
py -3.12 tools/items_ops.py list     [--catalog content/items.json] [--json]
py -3.12 tools/items_ops.py validate [--catalog content/items.json] [--schema content/item_fields.schema.json] [--baseline content/items.lock.json] [--json]
py -3.12 tools/items_ops.py schema   [--schema content/item_fields.schema.json] [--json]
```
- `list` — печатает def_id + display_name + блоки + kind; `--json` = машинный вывод
  (для веб-клиента subprocess-парити).
- `schema` — эхо нормализованной схемы полей (тот же источник, что валидация).
- exit 0 = OK, 1 = validation FAIL, 2 = usage/IO error.

### 8.2 `content/items.lock.json` — baseline rename-guard

Список отгруженных def_id (append-only). Начальный = def_id текущего items.json.
`validate --baseline` фейлит, если id из lock ИСЧЕЗ из каталога без пометки
deprecated (rename = delete+create = сироты в сейвах, §4 синтеза). deprecate-op —
редакторская эра, поэтому в И2 guard ТОЛЬКО детектит (не чинит).

### 8.3 Правила `validate` (из §4 синтеза + дизайн §6/§9)

1. **Неймспейс-конвенция:** каждый `id` матчит `<namespace>.<slug>`, `namespace`
   == `items.json.namespace`.
2. **rename-guard:** `id ∈ baseline && id ∉ каталог` → FAIL.
3. **Линт «код не ключуется по display_name»:** grep C-исходников фичи на
   `strcmp(...display_name...)`/ветвление по display_name → FAIL (страховка урока
   арт-strcmp rb-dark, дизайн §6). Формально это греп-гейт (§11 G-neg), op-слой его
   ДУБЛИРУЕТ как advisory.
4. **Композит-ключ ≤ string_max-1 (L5/code-review L1) — ЖЁСТКИЙ FAIL:** для каждого
   стака `len(container) + 1 + len(def_id) ≤ 63` (ключ `"<container>/<def_id>"` в
   `owned` при `string_max=64`); проверяется по ВСЕМ парам (contain × стакуемый def
   с совместимой accept_policy). Не advisory — переполнение ключа = тихая порча.
5. **Форма:** блоки соответствуют item_fields.schema.json; стак-консистентность
   (equip ⇒ не unlimited); accept_policy/container форма; уникальность id.
