## 2. Целевая структура фичи settings

### 2.1 Папка (feature_architecture §«Папка фичи в игре»)

```
templates/template/src/features/settings/
  settings.h           ЕДИНСТВЕННЫЙ публичный хедер (// feature-layer: L2)
  settings.c           persist-логика фрагмента (FEATURE_GAME_STATE-only): геттеры/сеттеры над settings_state
  settings_screen.c    экран (гир+панель) + состояние открытия + не-persist фолбэк-геттеры; не-static ТОЛЬКО символы из settings.h (settings_draw_ui, open/close/is_open, !FEATURE-геттеры), остальное static
```

**Почему два файла, а не один** (task предлагал выбор): раскладка зеркалит
конвенцию `<id>.c` (логика) + `<id>_screen.c` (экран, всё static)
(feature_architecture §«Папка фичи»), и — решающий довод — сохраняет ТЕКУЩЕЕ
разделение компиляции: `settings.c` живёт только под `FEATURE_GAME_STATE`
(включает генерируемый `settings_state.h`), `settings_screen.c` компилируется
ВСЕГДА (как `sys_settings.c` сейчас). Слить в один файл нельзя без того, чтобы
persist-часть тащилась в `!FEATURE`-сборку под `#if` — раскладка 1:1 переносит
уже проверенную границу конфигов и делает И1 near-verbatim move, а не редизайн.
Состояние `s_open` живёт в `settings_screen.c` (тот же TU, что публичные
`settings_open/close/is_open`) → никаких shared-static между TU, никакого
внутреннего хедера.

### 2.2 Проводка кадра

`game_features_draw_ui(World*)` становится владельцем ui_runtime-кадра и рисует
фичи по одной строке (список = z-order). `settings_draw_ui(ctx,w)` получает ctx
от агрегатора. `main.c` теряет прямой `ui_runtime_begin/sys_settings_ui/end` →
один вызов `game_features_draw_ui(&s_world)`. `draw_world` остаётся прямым
вызовом шелла (`render_mesh_draw`) — заглушка `game_features_draw_world` НЕ
зовётся (§8 OQ5). Фаза `update` settings не нужна (§0 п.2).

### 2.3 Навигация (правило)

Фича экспортирует `settings_open/close/is_open` + `settings_draw_ui`. КОГДА
открыть — решает игра: `main.c` зовёт `settings_open()` в ветке `--settings`.
Гир-кнопка внутри панели — собственный аффорданс фичи (не саморегистрация в
общий нав; общего нава в шаблоне нет). Саморегистрации фичи в какую-либо
нав/UI-композицию нет.

## 3. План файлов: код фичи

### 3.1 НОВЫЙ `templates/template/src/features/settings/settings.h`

Полное содержимое (единственный публичный хедер, вся публичная поверхность
фичи). Сеттеры объявлены под `#if FEATURE_GAME_STATE` — они существуют
(определяются в `settings.c`) и вызываются (как commit-колбэки экрана) ТОЛЬКО в
этом конфиге; под `!FEATURE` экран передаёт NULL-commit и сеттеры не нужны
(§0 п.3).

```c
#ifndef FEATURES_SETTINGS_H
#define FEATURES_SETTINGS_H
// feature-layer: L2
/* Единственный публичный хедер фичи settings — вся её публичная поверхность;
   всё остальное в папке static. Слой L2 (surface): зависит только от L0-шелла
   (ui_runtime / theme / World / game_save), не от других фич. */
#include "ui/nt_ui.h"    /* nt_ui_context_t */
#include "world/world.h" /* World */

#include <stdbool.h>

/* Фазовая отрисовка. Зовётся из game_features_draw_ui() внутри ui_runtime-кадра;
   ctx даёт агрегатор. Рисует гир-кнопку всегда + панель, когда is_open. */
void settings_draw_ui(nt_ui_context_t *ctx, World *w);

/* Навигация: фича экспортирует open/close/is_open; КОГДА открыть решает игра
   (main.c, флаг --settings). Гир-кнопка панели — собственный аффорданс фичи. */
void settings_open(void);
void settings_close(void);
bool settings_is_open(void);

/* Текущие громкости 0..1 — игра подключает их к своему аудио-микшеру. */
float settings_master(void);
float settings_music(void);
float settings_sfx(void);

#if FEATURE_GAME_STATE
/* clamp [0,1] -> запись в фрагмент settings_state -> game_save_mark_dirty().
   Существуют только под FEATURE_GAME_STATE (persist-путь); экран передаёт их
   как commit-колбэки. Под !FEATURE_GAME_STATE commit = NULL (сессионный store). */
void settings_set_master(float value);
void settings_set_music(float value);
void settings_set_sfx(float value);
#endif

#endif /* FEATURES_SETTINGS_H */
```

## 5. `templates/template/src/features/README.md` (НОВЫЙ — конвенция в шаблоне, Part B)

Нормативный чек-лист (не эссе). Полное содержимое:

```markdown
# src/features — конвенция фич в шаблоне

Игра = набор фич. Здесь описано, как фича устроена ВНУТРИ этого шаблона/игры.
Библиотечный контракт переиспользуемых фич — в корневом `features/README.md`.

## Что такое фича

- Фича = папка `src/features/<id>/` с ОДНИМ публичным хедером `<id>.h`
  (первая строка-комментарий `// feature-layer: L1|L2`); всё, что НЕ объявлено в
  `<id>.h`, — `static`, поэтому компилятор физически не даёт залезть в чужие кишки.
- Все экспортируемые символы с префиксом `<id>_`.
- Тест «фича или шелл»: «если бы две таких сосуществовали в одной игре, каждая
  захотела бы свою копию?» → фича. «Решает, какие фичи есть и в каком порядке,
  или владеет глобальным неймспейсом (тема, кадр, id полей)?» → шелл (L0, живёт
  вне `features/`).

## Слои (включать строго ВНИЗ)

- **L0 шелл** — не фича: `main.c`-кондуктор, `game_features.{c,h}`, `world.h`
  (заморожен как handle-bag), `ui_runtime`, `theme`, render bootstrap, codegen,
  persistence toolkit (`game_save` реестр/оркестрация + `game_storage` бэкенд +
  JSON-хелперы + транзакционный swap + codegen-инструмент), build_packs-каркас,
  DevAPI-обвязка.
- **L1 foundation** — дают API, не знают экранов и фич выше (items, dialogue-
  runtime). (`game-state` — библиотечный codegen/contract-пак, поставляющий
  L0-toolkit, а не in-game L1-фича.)
- **L2 surface** — вертикальные слайсы (settings, progression; journal — пример
  вне шаблона), используют L1 и L0.
- Кросс-фичевый include — ТОЛЬКО `features/<x>/<x>.h` слоя строго НИЖЕ.
  L2→L2 запрещено: зависимую-от фичу опускаем в L1, боковое ребро не разрешаем.

## Кадр: явный агрегатор, НЕ hook-таблица

- `game_features.c` — 7 фазовых функций (`init/update/react/record/draw_world/
  draw_ui/shutdown`), явные вызовы по ОДНОЙ строке на фичу. Никаких hook-таблиц,
  реестров, шедулера, `__attribute__((constructor))`.
- Порядок вызовов в `game_features_draw_ui()` = z-order (читается сверху вниз).
- Новая фича = ОДНА добавленная строка в соответствующий фазовый список; правок
  в середине файла нет (критерий бесконфликтности параллельных агентов).

## Навигация: показ решает ИГРА

- Фича экспортирует только API: `<id>_open()/<id>_close()/<id>_is_open()/
  <id>_draw_ui()`. Фича НЕ регистрируется ни в каком наве/UI-композиции сама.
- Форму и состав нава строит игра (композиционный слой) — код игры, не фичи.
  bottom_nav как код НЕ строится (это «просто набор кнопок») — правило вместо кода.
- Виджет-фичи получают entries/config ОТ игры (пример-принцип: resource_panel
  (появится в И3) — entries с геттерами от игры).

## Стейт

- Транзиентный стейт — `static` внутри фичи + аксессоры. НОВЫХ полей в `World`
  НЕ добавлять (handle-bag заморожен).
- Персистентный стейт — сейв-фрагмент через реестр `GameSaveFragment`: своя
  `<id>_state_t`, свои `to_json/from_json/reset` (генератор `--fragment <id>`
  для крупных, руками над хелперами для мелких), версия+миграции ЛОКАЛЬНЫ фиче.
  Идентичность фрагмента = имя JSON-ключа (числовых id-диапазонов нет).
  Фрагмент регистрируется ОДНОЙ строкой в шелле (`main.c`, ср. `main.c:397-398`
  `game_save_register_fragment(&<id>_state_fragment)`) — это легальный include
  ВНИЗ к L0-persistence-toolkit (напр. items → `game_save_register_fragment`).
- Кросс-фичевый доступ к стейту — только через API владельца, не через чужой
  под-объект.

## Ассеты (декларативная модель)

- Фичи НИКОГДА не пишут в паки. `build_packs.c` — 100% код игры.
- Фича ДЕКЛАРИРУЕТ потребности арта: `art_needs {slot, kind, hint}` в
  `feature.json` + рецепт в README. Код фичи берёт арт ТОЛЬКО как хендлы в
  конфиге (`<id>_init/<id>_config`), с graceful-фолбэком без арта.
- Рецепт подключения арта: скопировать пример/свои ассеты в `assets/` игры с
  тегом `feature=<id>` → добавить строки в свой `build_packs.c` → передать
  хендлы в `<id>_init/config`.
- Example-ассеты фичи живут в библиотечной копии `features/<id>/assets/`
  (с license + provenance).

## Кастомизация (три ступени)

1. Конфиг — без правок кода (рескин, entries, хендлы арта).
2. Правка своей копии — под нужное поведение.
3. Промоут обобщаемого улучшения в `features/` ДО закрытия игры.

Copy-then-own: игры = fixtures, багфиксы НЕ текут назад автоматически;
дисциплина одна — обобщаемое промоутить вверх.

## Чек-лист новой фичи

- [ ] папка `src/features/<id>/`, один публичный `<id>.h` (`// feature-layer: L1|L2`)
- [ ] все экспортируемые символы `<id>_*`, остальное `static`
- [ ] include только вниз по слоям
- [ ] проводка = по строке на фичу в нужных фазах `game_features.c`
- [ ] нав/показ — через `open/close/is_open`, саморегистрации нет
- [ ] стейт: транзиент = static+аксессоры (не в World); персист = фрагмент
- [ ] арт: `art_needs` в feature.json + хендлы-в-конфиге + graceful-фолбэк
```

Эталон конвенции — папка `settings/` (L2): один публичный `settings.h`,
`settings.c` (persist-фрагмент) + `settings_screen.c` (экран; не-static только
символы из `settings.h`), проводка одной строкой в `game_features_draw_ui`,
показ через `settings_open()` из игры.
