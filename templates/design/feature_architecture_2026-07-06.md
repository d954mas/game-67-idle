# Feature-based game architecture — синтез двух независимых Opus-проектов (2026-07-06)

Цель лида: игра = набор фич; фича = код + ассеты + стейт; фичи извлекаются из
игры в общую библиотеку `features/` и реюзаются; фичи могут использовать друг
друга строго сверху вниз (shop → inventory, никогда наоборот). Template-first.

## 0. Рефрейм (важный)

Боль джема — не отсутствие реюза, а **коллизии параллельных агентов на общих
«серединах»**: game_actions.c (1769 строк, все фичи), bottom_nav.c (каждая
ветка гасит все соседние экраны), game_content.h, один schema.json c общим
пространством числовых id. Критерий успеха архитектуры: **каждая общая точка
= одна ДОБАВЛЕННАЯ строка на фичу, никогда правка в середине**. Реюз в
`features/` выпадает почти бесплатно, как побочный эффект самодостаточности.

## 1. Что уже есть в репо

- `features/README.md` — контракт есть: фича = copyable capability (code,
  assets, state schema, migrations, UI, DevAPI hooks, tests); copy-then-own;
  плагин-менеджер явно запрещён. Один реальный пак: `features/game-state`
  (codegen schema→C). `feature.json` schema `ai_studio.feature.v1` готова к
  расширению.
- `templates/TEMPLATE.md` — уровни реюза уже описаны: engine (linked) /
  features (copy-then-own) / template shell / game-only.
- Внутри игры границы фичи НЕТ: shop = 11+ файлов в 6 слоях.
- Движок: осознанно без реестра систем, один коллбек кадра — композиция
  целиком на шаблоне. DevAPI command table = принятый C17-идиом реестра.

## 2. Целевая конвенция

### Слои (зависимости строго вниз)

- **L0 ядро-шелл** (НЕ фича, живёт в шаблоне): main.c-кондуктор,
  game_features.{c,h} (агрегатор/композиция), world.h (заморожен как
  handle-bag: время + камера + entity-хендлы, НИКАКИХ полей фич), ui_runtime,
  theme, game_storage (бэкенд), render bootstrap, codegen-механизмы,
  build_packs-каркас, DevAPI-обвязка. (bottom_nav — НЕ ядро, а реюзабельная
  фича-виджет; см. «Навигация».)
- **L1 foundation-фичи**: items (каталог+инстансы+контейнеры, ВКЛ. валюты и
  purse — отдельные wallet/inventory слиты в items, item_system §1),
  dialogue-runtime, game-state — дают API, не знают экранов и фич выше.
- **L1+ над items**: equipment (слоты-как-данные, API без своего экрана —
  item_system §1/§4; НЕ L2).
- **L2 surface-фичи**: shop, combat, journal, world-map, settings, progression
  (учебная, API-уровни; item_system §5) — вертикальные слайсы, используют L1
  и ядро.
- Тест «фича или шелл»: «если бы две таких сосуществовали в одной игре, каждая
  хотела бы свою копию?» → фича. «Решает, какие фичи есть и в каком порядке,
  или владеет глобальным неймспейсом (id полей, тема, кадр)?» → шелл.
- L2→L2 зависимость = запах: зависимую-от фичу опускаем в L1, а не разрешаем
  боковое ребро.

### Папка фичи в игре

```
src/features/<id>/
  <id>.h              ЕДИНСТВЕННЫЙ публичный хедер (// feature-layer: L1|L2)
  <id>.c              фазовые функции: <id>_init/_update/_draw_ui/_draw_world/
                      _shutdown/_register_devapi (+опц. <id>_react/_record,
                      см. «Проводка кадра»); вызов показа — задача игры
                      (см. «Навигация»)
  <id>_screen.c       экран(ы), всё static
  <id>_actions.c      мутации GameState (game_actions.c умирает, режется сюда)
  <id>_content.h      контент-структуры этой фичи
  state.schema.json   схема фрагмента стейта (идентичность = имя JSON-ключа;
                      числовых id полей и блоков диапазонов НЕТ — генератор v2,
                      state doc §5/§7)
  feature_packs.c     <id>_pack_assets(ctx)/<id>_pack_atlas(ctx) (если есть арт)
  tests/  README.md
```

Все экспортируемые символы с префиксом `<id>_`; внутренности static →
компилятор физически не даёт залезть в чужие кишки. Кросс-фичевые include —
только `features/<x>/<x>.h` слоя строго ниже.

### Проводка кадра: явный агрегатор, НЕ hook-таблица

`src/features/game_features.c` — СЕМЬ фазовых функций с явными вызовами по
одной строке на фичу: пять базовых (init/update/draw_world/draw_ui/shutdown)
+ ДВЕ новых для двухфазного кадра событий — game_features_react() и
game_features_record() (Q1, event-док §7). Кадр событий: update эмитит →
react гонит реакторов-потребителей до фикспойнта (перепроход лога, пока не
родится новое; кап поколений 16, dev-warn) → record — чистые рекордеры
(аналитика/лог/DevAPI) одним проходом → шелл сбрасывает лог кадра (event-док
§2, страж 1). Список в `game_features_draw_ui()` — это и есть z-order
(читается сверху вниз). Игра добавляет потребителя события ОДНОЙ строкой в
react- или record-список. Аргументы против таблицы: движок явно запрещает
шедулер; сигнатуры фаз разные (react/record несут World* как остальные
пять); конфликт-поверхность та же (обе модели = append одной строки в один
файл); отладка порядка отрисовки по одному месту.

### Навигация: вызов показа — задача ИГРЫ (решение лида, 2026-07-06)

Фича НЕ регистрируется в нав сама (это предполагало бы, что фича знает про
нав). Фича экспортирует только API: `shop_open(id)/shop_is_open()/
shop_close()/shop_draw_ui()`. **bottom_nav — сам реюзабельная фича-виджет**:
`bottom_nav_ui(ctx, entries[], count)` рисует что дали. Соединяет игра
(композиционный слой): строит entries из указателей на API фич, решает что в
наве / что со сцены / что с карты. Одна добавленная строка на фичу в файле,
которым владеет игра — append-only критерий сохраняется. Стрелки строго вниз:
игра → nav-фича, игра → shop; nav и shop друг про друга не знают. Взаимное
исключение full-screen'ов — цикл по entries в композиции игры. bottom_nav со
switch'ем на все экраны умирает.

### Стейт

- Транзиентный стейт фичи — file-static внутри `<id>.c` + аксессоры (прецедент
  sys_settings). В World полей не добавлять — он уже стал god-struct в
  rb-dark-rpg.
- Персистентный — **map по фичам в сейве, структуры в рантайме** (решение
  лида, 2026-07-06): документ слота = один JSON, внутри под-объект на фичу по
  её строковому id (`{"features": {"shop": {...}, "inventory": {...}}}`).
  Ключ = имя папки → уникален по построению, глобальное пространство числовых
  id и таблица диапазонов НЕ НУЖНЫ. В рантайме — типизированная
  `<id>_state_t g_<id>_state` в папке фичи (не map, без hash-lookup); хуки
  save/load фичи сериализуют её в свой под-объект, шелл обходит фичи → один
  атомарный документ на слот. Кросс-фичевый доступ только через API владельца
  (`wallet_get_gold()`), не через чужой под-объект. Миграции локальны фиче
  (версия внутри её фрагмента), едут при извлечении вместе с папкой.

### Ассеты

Библиотека и игра — асимметричны, и это и есть граница copy-then-own:
- в `features/<id>/` (библиотека) ассеты лежат самодостаточно в
  `features/<id>/assets/`;
- в игре ассеты остаются в общем `assets/`-дереве, помеченные полем
  `feature=<id>` в assets.jsonl (pull.mjs, provenance-сканер и галерея знают
  только game/template asset-roots — ноль переделок тулинга);
- у каждой фичи свой атлас-блок / своя `<id>_pack_assets(ctx)`, вызываемая из
  build_packs.c одной строкой; общий UI-кит (panel/button/slider) остаётся в
  шелловском атласе `ui`.
- (Отвергнутая альтернатива: assets внутри src/features/<id>/ в игре — ломает
  видимость для pull/gallery/provenance; вернёмся, если extraction докажет
  боль.)

### Контент

`game_content.h` остаётся ОДНИМ, но генерируемым: данные по фичам в
`design/data/<feature>.json` (append-friendly), merge делает codegen, руками
хедер не правится → нет конфликтов.

## 3. Жизненный цикл фичи

Родилась самодостаточной в игре → закалилась → извлечена в `features/<id>/`
(почти folder-move) → скопирована в следующую игру (copy-then-own).

Извлекаемость = ссылаться только на: движок, World-handle-bag, theme,
GameState через API game-state, публичные хедеры фич ниже слоем. Extraction —
чеклист в features/README (не скилл): собрать feature-tagged ассеты с
провенансом, забрать schema/content-фрагменты, сгенерить INSTALL.md из
feature.json (+ новые поля: layer, provides, registers, assets_tag;
state_id_range убран — числовых id-диапазонов больше нет, state doc §5).

Багфиксы под copy-then-own НЕ текут назад — и это ок (игры = throwaway
fixtures). Дисциплина одна, вверх: «улучшил скопированную фичу и это
обобщается — промоутни в features/ до закрытия игры».

## 4. План миграции templates/template (инкременты, каждый shippable)

1. **features/ + агрегатор + settings как первая фича** (~5 файлов): перенос
   sys_settings → features/settings/, game_features.{c,h}, main.c-кадр
   сворачивается до 4 вызовов, CMake. Поведение идентично.
2. **Документ конвенции** src/features/README.md (слои, one-public-header,
   фазовый контракт, no-World-fields, стейт/ассет-правила).
3. **Стейт per-feature как конвенция**: сейв-документ получает под-объект на
   фичу; новая фича пишет свою `<id>_state_t` + save/load хуки руками через
   game_storage JSON-хелперы (генератор per-fragment — только когда докажет
   боль, см. п.6).
4. **bottom_nav как фича-виджет** (entries от игры) + ассет-шов: build_packs.c
   вызывает `<id>_pack_assets`.
5. **Образцовые фичи items (L1) + progression (L2, учебная) + HUD-бейдж** —
   учат правило «сверху-вниз» примером; shop = минимальный buy-only стаб
   (item_system §5/§8); валидируют extraction на живом слайсе.
6. **Только при первом реальном промоуте во 2-ю игру**: schema-merge tool
   (генератор учится --schema-dir, assert на дубли id/path),
   feature.json→INSTALL генератор, extract-feature скрипт для ассетов.

## 5. Что НЕ строим (отвергнуто обоими)

Плагин-менеджер / dependency solver / версии; hook-таблица кадра / шедулер;
фрагментация рантайм-GameState или сейв-формата; per-feature nt_ui
контексты/атласы; авто-дискавери ассетов/glob в CMake (только явные
append-списки); `__attribute__((constructor))` саморегистрация (порядок
недетерминирован) — только явные вызовы; 4-й source-kind в галерее;
schema-auto-merge до первой реальной боли.

Advisory include-check скрипт (~40 строк: слои из тегов, back edge = флаг) —
в очередь ПОСЛЕ конвенции, только если ревью пропустит связность.

## 6. Решения лида (2026-07-06)

1. Вызов показа экрана — задача игры; bottom_nav = реюзабельная фича-виджет,
   состав и форму нава решает игра. Фичи в нав НЕ саморегистрируются.
2. Стейт: map по фичам в сейв-документе (ключ = id фичи), типизированные
   структуры в рантайме. Числовые id-диапазоны отменены (идентичность = имя
   JSON-ключа; генератор v2 пер-фрагментный, без глобального пространства id
   — state doc §5/§7).
3. Copy-then-own подтверждён: копия фичи в игре полностью правится лидом;
   дисциплина — обобщаемые улучшения промоутить в features/ до закрытия игры.

Закрыто (item_system §5/§8/§9, лид 2026-07-06): учебная пара шаблона =
items (L1) + progression (L2) + HUD-бейдж; shop = минимальный buy-only стаб
(не полноценный магазин).
