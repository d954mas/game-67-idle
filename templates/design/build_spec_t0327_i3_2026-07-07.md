## 2. Архитектурное ядро: модель прогрессии + слоёвость (читать первым)

### 2.1 Хранилище трека = плоская map (generator-fit, как owned у items)

**Сейв-фрагмент `progression` хранит `tracks: map<string, TrackState>`, ключ карты =
`track_id`, `TrackState = { level:int, xp:i64 }`.** «Кривая», `max_level`, `mode`,
`currency_def`, `on_level_up` — НЕ в сейве, они в const-каталоге треков
(`content/progression.json`, компайл-тайм). Разрез «определение/состояние» держится
как у items (миграции уровень-0): сейв несёт только `level + xp` на трек, ребаланс
кривых/добавление треков сейв не трогают.

- **`level`** — текущий уровень трека (все режимы). Индекс в таблицу стоимости.
- **`xp`** — ВНУТРЕННИЙ счётчик, СМЫСЛОВ ТОЛЬКО для `threshold`-режима (аккумулятор
  действий). Для `manual`/`auto` поле `xp` = игнорируемый дефолт 0 (xp живёт в
  purse как currency-предмет — L2→L1 связь). Ровно как per-copy `level/durability`
  у items смысловы только для уников (плоская форма под генератор, §6.4 i2).

Трек БЕЗ записи в `tracks` → уровень 0, xp 0 (ленивый дефолт). Запись аллоцируется
при первой мутации (`level_up`/`add_xp`/`set_level`). Скан `progression_state.tracks[]`
НАПРЯМУЮ (генераторные find/alloc — `static`, как у items). ОДИН хелпер строит/ищет
ключ = `track_id` (ключ прост, НЕ композит — в отличие от items; линт длины не нужен,
но `snprintf`-усечение проверяется).

### 2.2 Три режима (одна ось `mode`; хранилище определяет режим — дизайн §5)

| mode | xp-источник | лвлап | тик |
|---|---|---|---|
| `manual` | purse (`currency_def`) | `progression_level_up()` тратит `cost(level)` из purse | НЕ тикает |
| `auto` | purse (`currency_def`) | `progression_update()` авто-покупает пока хватает, HARD-кап | тикает |
| `threshold` | внутренний `xp` (`progression_add_xp`) | `progression_update()` авто-лвлапит пока `xp>=cost`, HARD-кап | тикает |

**Слоёвость (ЯДРО арки).** `manual`/`auto` читают и тратят purse через `items.h`:
`items_purse(currency_def)` (сколько xp у игрока), `items_remove("purse",
currency_def, cost, "level_cost:<track>")` (потратить). ЭТО и есть include
`features/items/items.h` в `progression.h`/`progression.c` — единственное разрешённое
фича-ребро, доказательство слоёвости (греп-гейт §8). Обратного ребра НЕТ: items
НИКОГДА не включает `progression.h` (греп-гейт §8, доказывает направление вниз).

### 2.3 Кривая = int64-таблица, запечённая в codegen (ноль float в C)

Кривая авторится в `content/progression.json` пресетом; в И3 поддержан РОВНО ОДИН
тип — `exp` (LEAN-порез B: `linear`/`table`/`poly` в codegen были бы мёртвы и
непокрыты тестом → валидатор отвергает их как `unknown curve type` ГРОМКО).
`generate_progression_tracks.py` ВЫЧИСЛЯЕТ кривую в int64-таблицу стоимостей
`cost[0..max_level-1]` (Python-арифметика на codegen-этапе, **округление вниз —
FLOOR** — к int64, §5.2) и эмитит как `static const int64_t`-массив. РАНТАЙМ читает
ТОЛЬКО таблицу (`cost[level]` = цена перехода `level→level+1`). Никакого
интерпретатора формул в C, никакой float-математики в xp/level (синтез: «рантайм
всегда читает таблицу»). `display`-интерполяция бара — float, но она ВНУТРИ
resource_panel (не в математике уровней). Запечённые значения ПОКРЫТЫ golden-тестом
`test_progression_curve` (§5.8/§5.9d). Добавление `table` (design-санкционированный
escape-hatch «поручная правка клеток: уровень 10 стоит 500, там босс») — тривиальное
расширение генератора СО СВОИМ identity-тестом, когда реальной игре понадобится.

### 2.4 HARD-капы тика (синтез §1 T5 — БАЗОВЫЙ инвариант, не опция)

`progression_update()` резолвит auto/threshold-треки. Два вектора зависания за кадр:
(1) auto-трек с `on_level_up`-эмиссией ≥ своей цены (self-refund) → бесконечный
while; (2) каскад A→B→A (трек кормит xp другой трек) внутри кадра. Оба ЗАКРЫТЫ:

```c
#define PROGRESSION_MAX_LEVELUPS_PER_TRACK 64  /* T5: per-track per-frame while-кап */
#define PROGRESSION_MAX_CASCADE_DEPTH      8   /* T5: глубина xp-to-track рекурсии */
```

Достижение любого капа = `nt_log_warn(...)` (drop-with-warn) + выход. Тесты
доказывают ТЕРМИНАЦИЮ (не зависание) и точное число лвлапов == кап. Алгоритм — §5.4.

## 5. И3a — progression (L2)

### 5.1 `content/progression.json` (демо-каталог; copy-then-own, НЕ шарится)

Один демо-трек `hero` (auto-режим, кормится валютой `tmpl.xp` из каталога items).
Форма зеркалит items.json (`namespace` + список объектов).

**РЕШЕНИЕ (created/lock — #8, явно):** `content/progression.json` НЕ несёт поле
`created` и НЕ имеет lock-гарда (в отличие от `items.json`/`items.lock.json`).
Основание: у items орфан-def = ПОТЕРЯ количества в сейве (нужен деструктивный гард +
карантин + миграция); у progression орфан-трек-запись ИНЕРТНА (`progression_update`
итерирует только каталог-треки, лишняя запись игнорится). Цена решения: удаление
трека из каталога ТЕРЯЕТ заработанные уровни игрока БЕЗ компенсации и БЕЗ красного
гейта — ПРИНИМАЕТСЯ для шаблона-fixture; забота реальной игры (при первой игре с
престижем/переносом — свой lock+миграция трека). Задокументировать это в
`progression/README.md`.

```json
{
  "schema": "game_seed.progression_tracks",
  "namespace": "tmpl",
  "tracks": [
    {
      "id": "hero",
      "mode": "auto",
      "currency_def": "tmpl.xp",
      "max_level": 20,
      "curve": { "type": "exp", "base": 50, "growth_num": 3, "growth_den": 2 }
    }
  ]
}
```

Правила формы (валидирует генератор, §5.2): `id` = `<namespace>.<slug>` ИЛИ голый
slug — **ДЕФОЛТ: id трека = голый slug `hero` (не `tmpl.hero`)**, т.к. трек-id —
внутренний идентификатор прогрессии, не пересекается с items-неймспейсом; генератор
валидирует непустоту + уникальность (namespace-конвенцию к трекам НЕ применяем).
`mode ∈ {manual,auto,threshold}`; `currency_def` ОБЯЗАТЕЛЕН для manual/auto (должен
существовать в items — валидатор проверяет наличие в `content/items.json` и что это
currency-предмет), для threshold — отсутствует/игнор; `1 ≤ max_level ≤ 9999` (**кап
9999 обязателен — #12: сейв-поле `level` капнуто схемой 9999 (§5.3); `max_level>9999`
дал бы тихий кламп на сериализации; валидатор отвергает громко**); `curve.type` = ТОЛЬКО
`exp` (иначе FAIL, §5.2). `on_level_up` в JSON — **ОТВЕРГАЕТСЯ валидатором** (LEAN-порез
A: codegen его не пекёт; рантайм-поддержка есть, но авторится не через JSON в И3 —
см. §5.2). Демо-трек `on_level_up` не имеет.

**Кросс-зависимость progression.json → items.json (#9, «падать громко»).**
`currency_def` (напр. `tmpl.xp`) ссылается на items-def. Удаление этой валюты по
деструктивному воркфлоу items (И2c: items.lock.json + гард) пройдёт items-гард ЗЕЛЁНЫМ
(items про progression не знает — и не должен, §2.2), но ОБРУШИТ codegen progression
(`--items`-проверка не найдёт def → `SystemExit`). Это ЖЕЛАЕМОЕ поведение (красная
сборка ≠ тихая порча). План-файл И3a добавляет ОДНУ advisory-строку в
`src/features/items/README.md` раздел удаления def («валюта, на которую ссылается трек
progression, не удаляется без синхронной правки `content/progression.json`») — это
единственная правка items-артефакта в И3 и она ДОКОВАЯ (не код; items-код не трогается).

### 5.2 `tools/generate_progression_tracks.py` (контент-codegen)

Наследник `tools/generate_items_catalog.py` (те же хелперы `c_str/c_bool/c_i64/
c_ident`, `write_if_changed` атомарно, идемпотентно, детерминированный порядок ==
порядок JSON, `SystemExit`-валидация с префиксом `progression tracks validation:`).
Интерфейс:

```
py -3.12 tools/generate_progression_tracks.py \
    --catalog content/progression.json \
    --items content/items.json  \  # для проверки currency_def существует и is-currency
    --out-dir <dir>
```

Выход `progression_tracks.gen.h`:
```c
#ifndef PROGRESSION_TRACKS_GEN_H
#define PROGRESSION_TRACKS_GEN_H
/* Generated by tools/generate_progression_tracks.py from content/progression.json. Do not edit. */
#include "features/progression/progression.h"
#define PROGRESSION_TRACK_COUNT <N>
extern const progression_track_def_t k_tracks[];
extern const int k_tracks_count;
#endif
```
Выход `progression_tracks.gen.c`: per-track `static const int64_t COST_<SYM>[] = {…LL,…}`
(запечённая кривая, длина == `max_level`), затем `const progression_track_def_t
k_tracks[] = {…}` (с `.on_level_up = NULL, .on_level_up_count = 0` ВСЕГДА, см. ниже) +
`const int k_tracks_count = …;`.

**Запекание кривой (Python, int64-выход) — ТОЛЬКО `exp`, округление FLOOR:**
- `exp {base, growth_num, growth_den}` → **`cost[L] = floor(base * (growth_num /
  growth_den) ** L)`** как int64. Считать через целочисленную арифметику
  `(base * growth_num**L) // (growth_den**L)` (чистый int, без float-погрешности) —
  это и есть FLOOR по построению; переполнение int64 → `SystemExit`. Демо: `base=50,
  gn=3, gd=2` → 50, 75, 112 (=112.5 floor), 168 (=168.75 floor), … (герой-идл-кривая).
- Любой другой `curve.type` (`linear`/`table`/`poly`/…) →
  `SystemExit("progression tracks validation: unknown curve type '<type>' (only 'exp'
  supported in И3)")`. LEAN-порез B: не-exp типы отвергаются громко.
- **golden-тест обязателен** (§5.9d `test_progression_curve`): ассертит запечённые
  значения демо-кривой (50/75/112/168/…) на РЕАЛЬНОМ `progression_tracks.gen.c` —
  закрывает риск FLOOR-vs-ROUND и корректность struct-разводки. Опц. дополнить
  `generate_progression_tracks.py --selftest` (быстрый юнит бэйк-функции), не обязателен
  при наличии C-golden.

**`on_level_up` — LEAN-порез A (codegen НЕ пекёт):** в И3 у on_level_up НОЛЬ
shipped-потребителей (демо-трек его не имеет) и НОЛЬ покрытия codegen-ветки (T5-тесты
идут через РУКОПИСНЫЙ тест-каталог §5.7). Поэтому генератор его НЕ парсит и НЕ пекёт —
всегда эмитит `.on_level_up = NULL, .on_level_up_count = 0`; наличие `on_level_up` в
JSON = `SystemExit` (не тихо игнорить). **РАНТАЙМ on_level_up ОСТАЁТСЯ** (`progression_emit_t`,
`apply_on_level_up`, каскад, self-refund — §5.4/§5.6, доказаны тест-каталогом). Когда
реальной игре понадобятся authored-каскады — вернуть ветку генератора СО СВОИМ тестом.
Ноль правок `generate_state.py` (это ДРУГОЙ codegen; тронуть стейт-генератор → СТОП,
§Риски R1).

`progression_update`, см. §5.4. Генератор даёт `progression_emit_levelup(const char

### 5.4 `src/features/progression/progression.h` (публичная поверхность, L2)

Рукописные каталог-типы (struct, НЕ codegen — как game_item_def_t у items) + API.
**ВКЛЮЧАЕТ `features/items/items.h` — ЕДИНСТВЕННОЕ фича-ребро (слоёвость).** Скетч:

```c
#ifndef FEATURES_PROGRESSION_H
#define FEATURES_PROGRESSION_H
// feature-layer: L2
/* Единственный публичный хедер фичи progression — вся публичная поверхность;
   остальное в папке static. L2 surface: зависит от L0-шелла (game_save-toolkit +
   gsj_ + движок) И от L1-фичи items (единственное фича-ребро — доказательство
   слоёвости: manual/auto-режимы читают+тратят purse через items API). Обратного
   ребра нет: items НИКОГДА не включает progression.h. Вся математика уровней/xp —
   int64; кривая запечена в int64-таблицу на codegen (ноль float-формул в C). */
#include "features/items/items.h"   /* L2->L1: items_purse / items_add / items_remove */
#include <stdbool.h>
#include <stdint.h>

typedef enum progression_mode_t {
    PROGRESSION_MODE_MANUAL = 0,
    PROGRESSION_MODE_AUTO,
    PROGRESSION_MODE_THRESHOLD,
} progression_mode_t;

/* on_level_up-эмиссия: РОВНО одна из форм (другой указатель NULL / поле игнор):
   currency (def_id!=NULL) -> items_add(purse, def_id, amount);
   xp-to-track (to_track!=NULL) -> progression_add_xp(to_track, amount) (каскад). */
typedef struct progression_emit_t {
    const char *def_id;    /* валюта в purse, либо NULL */
    const char *to_track;  /* трек-получатель xp (каскад), либо NULL */
    int64_t amount;
} progression_emit_t;

typedef struct progression_track_def_t {
    const char *id;
    progression_mode_t mode;
    const char *currency_def;              /* purse-валюта manual/auto; NULL для threshold */
    int max_level;
    const int64_t *cost;                   /* запечённая таблица [0..max_level-1]; cost[L] = цена L->L+1 */
    int cost_count;                        /* == max_level */
    const progression_emit_t *on_level_up; /* NULL если нет */
    int on_level_up_count;
} progression_track_def_t;

const progression_track_def_t *progression_track_def(const char *track); /* NULL если неизвестен */

/* ---- Запросы (чистые чтения) ---- */
int      progression_level(const char *track);        /* 0 если трек без записи/неизвестен */
int      progression_max_level(const char *track);
int64_t  progression_xp_current(const char *track);   /* manual/auto: items_purse(currency); threshold: внутр. xp */
int64_t  progression_xp_needed(const char *track);    /* cost(level); 0 если на max */
bool     progression_can_level_up(const char *track); /* xp_current >= xp_needed && level<max */

/* ---- Мутации (reason обязателен, дизайн §10; формат verb:subject) ---- */
bool     progression_level_up(const char *track, const char *reason); /* MANUAL: тратит purse; true если поднял */
void     progression_add_xp(const char *track, int64_t n, const char *reason); /* THRESHOLD: копит внутр. xp (резолв в update) */
void     progression_set_level(const char *track, int level, const char *reason); /* Р6: пролог (сильный герой) */
void     progression_reset(const char *track, const char *reason);   /* Р6: престиж (level=0 И внутр. xp=0) */

/* ---- Тик: auto/threshold авто-лвлапы, T5-капы, эмит progression.levelup ---- */
void     progression_update(void);

#endif /* FEATURES_PROGRESSION_H */
```

Точные сигнатуры уточняются по потреблению тестов; ключевое — API отдаёт/принимает
строки+int/i64, НЕ генерируемые типы (`progression.h` не включает
`progression_state.h`).

### 5.5 `src/features/progression/progression.c` (логика над `progression_state`)

Includes: `"features/progression/progression.h"`, `"progression_tracks.gen.h"`
(extern `k_tracks`/`k_tracks_count`), `"progression_state.h"` (генерируемый:
`ProgressionState progression_state; ProgressionTrackState; PROGRESSION_STATE_MAX_TRACKS`),
`"progression_state_events.gen.h"` (`progression_emit_levelup`), `"game_save.h"`
(`game_save_mark_dirty`), `"log/nt_log.h"` (T5 warn), `<assert.h>/<string.h>`.

**Скан состояния (как items_containers.c):**
```c
static ProgressionTrackState *find_track(const char *id) {
    for (int i=0;i<PROGRESSION_STATE_MAX_TRACKS;++i)
        if (progression_state.tracks[i].used && strcmp(progression_state.tracks[i].key,id)==0)
            return &progression_state.tracks[i];
    return NULL;
}
static ProgressionTrackState *find_or_alloc_track(const char *id); /* find; иначе первый !used, key=id, level/xp=дефолт */
```
**reason-контракт (лёгкий, без items-verb-списка):** inline `progression_reason_check`
= debug-assert `reason!=NULL && strchr(':') && subject непуст` (формат `verb:subject`,
дизайн §10); в release no-op. НЕ включает `features/items/reason_tags.h` (это items-
internal; не тянуть чужие кишки). Спенды в purse передают reason В `items_remove/add`,
где срабатывает ПОЛНЫЙ items-verb-чек — так verb-список валют не дублируется.

**Запросы:**
- `progression_level(track)` → `find_track` ? `->level` : 0.
- `progression_xp_current(track)`: def=`progression_track_def`; если mode∈{manual,auto}
  → `items_purse(def->currency_def)`; если threshold → `find_track?->xp:0`. **Здесь
  живёт include items.h — слоёвость.**
- `progression_xp_needed(track)`: `level=progression_level`; `level>=max?0:def->cost[level]`.
- `progression_can_level_up(track)`: `level<max && xp_current>=xp_needed`.

**Мутации:**
- `progression_level_up(track, reason)` [MANUAL — на auto/threshold возвращает false,
  их двигает update]: `progression_reason_check`; def; если `level>=max` → false; если
  `items_purse(currency)<cost` → false; `items_remove("purse", currency, cost,
  reason)` (передаёт reason, items-verb-чек); `st=find_or_alloc; st->level++;`
  `apply_on_level_up(def, 0)` (та же (def,depth)-сигнатура, что в §5.6; при cut A
  demo/shipped on_level_up пуст → no-op); `game_save_mark_dirty()`; **НЕ эмитит
  levelup** (дизайн §5: manual — вызвавший знает результат); true.
- `progression_add_xp(track, n, reason)` [THRESHOLD]: reason-чек; `n<=0`→return; def;
  `st=find_or_alloc; st->xp += n` (клам к schema-max); mark_dirty. **Резолв лвлапа —
  в update, не здесь** (копит; update авто-лвлапит). **Для не-threshold трека — ТИХИЙ
  no-op (xp теряется, #17):** это происходит только при каскаде `on_level_up{to_track}`
  в manual/auto-трек; поведение осознанное (кормить xp можно только threshold-трек) —
  задокументировать в README, чтобы авторы каскадов не удивлялись.
- `progression_set_level(track, level, reason)` [Р6 пролог]: reason-чек; clamp `level`

### 5.6 `progression_update()` — тик с HARD-капами (§2.4; ЯДРО алгоритма)

```c
static void apply_on_level_up(const progression_track_def_t *def, int depth); /* fwd */

static void resolve_track(const progression_track_def_t *def, int depth) {
    if (def == NULL) return;
    if (def->mode == PROGRESSION_MODE_MANUAL) return;         /* manual двигает игра */
    if (depth > PROGRESSION_MAX_CASCADE_DEPTH) {
        nt_log_warn("progression: cascade depth cap at '%s' (dropped)", def->id);
        return;                                              /* T5: каскад A->B->A */
    }
    /* #6: ЛЕНИВО. НЕ аллоцируем запись на входе (иначе кадр 1 создаёт нулевые записи
       всех auto/threshold-треков -> противоречит §2.1/OQ3 «свежая игра = пустые треки»).
       Читаем уровень через find_track (NULL -> 0); alloc ТОЛЬКО перед реальным лвлапом. */
    ProgressionTrackState *st = find_track(def->id);          /* может быть NULL */
    int level = st ? st->level : 0;
    int iters = 0;
    while (iters < PROGRESSION_MAX_LEVELUPS_PER_TRACK) {
        if (level >= def->max_level) break;
        int64_t cost = def->cost[level];
        if (def->mode == PROGRESSION_MODE_AUTO) {
            if (items_purse(def->currency_def) < cost) break; /* purse пуст -> записи НЕ создаём */
        } else { /* THRESHOLD */
            if ((st ? st->xp : 0) < cost) break;              /* st==NULL -> xp 0 -> не лвлапит */
        }
        if (st == NULL) st = find_or_alloc_track(def->id);    /* ЛЕНИВАЯ аллокация: только при реальном лвлапе */
        if (def->mode == PROGRESSION_MODE_AUTO)
            items_remove("purse", def->currency_def, cost, "level_cost:auto"); /* reason<=32 */
        else
            st->xp -= cost;
        int old_level = st->level;
        st->level += 1;
        level = st->level;
        progression_emit_levelup(def->id, old_level, st->level); /* факт-событие (§10) */
        apply_on_level_up(def, depth);                        /* рекурсия каскада внутрь глубины */
        iters += 1;
    }
    if (iters == PROGRESSION_MAX_LEVELUPS_PER_TRACK)
        nt_log_warn("progression: per-track levelup cap at '%s' (dropped rest this frame)", def->id);
    if (iters > 0) game_save_mark_dirty();
}

/* Cut A: shipped/demo on_level_up ВСЕГДА пуст (генератор эмитит NULL,0) -> этот цикл =
   no-op в шаблоне. Рантайм-путь остаётся и покрыт ТОЛЬКО тест-каталогом (§5.7). */
static void apply_on_level_up(const progression_track_def_t *def, int depth) {
    for (int i = 0; i < def->on_level_up_count; ++i) {
        const progression_emit_t *e = &def->on_level_up[i];
        if (e->def_id != NULL)   items_add("purse", e->def_id, e->amount, "loot:levelup");
        if (e->to_track != NULL) { progression_add_xp(e->to_track, e->amount, "convert:cascade");
                                    resolve_track(progression_track_def(e->to_track), depth + 1); }
        /* #17: если to_track — НЕ threshold, add_xp = тихий no-op (xp теряется) — осознанно. */
    }
}

void progression_update(void) {
    for (int i = 0; i < k_tracks_count; ++i)
        resolve_track(&k_tracks[i], 0);
}
```

Ключевое: (а) self-refund auto-трек (`on_level_up` возвращает `def->currency` ≥ cost
в purse) → while перезаполняет purse → терминация на per-track капе 64. (б) каскад
A→B→A (`on_level_up.to_track`) → `resolve_track(…, depth+1)` → терминация на
depth-капе 8. Оба = `nt_log_warn` (drop-with-warn), кадр всегда завершается.
Использованные reason-verb'ы (`level_cost`,`loot`,`convert`) — из закрытого items-
списка (`reason_tags.h`), items-чек их пропускает.

```c
    const char *label;                     /* имя ресурса ("Gold"/"Hero") */
    resource_panel_kind_t kind;
    const nt_atlas_region_ref_t *icon;     /* ОПЦ. хендл арта от игры; NULL -> graceful фолбэк (rect/без иконки) */
    void *ud;                              /* прозрачный, в каждый геттер */
    int64_t (*value)(void *ud);            /* ОБЯЗАТЕЛЕН. counter: число; bar: текущий прогресс (xp) */
    int64_t (*max)(void *ud);              /* bar: знаменатель (xp до след. уровня); NULL для counter/при неизвестном max */
    int64_t (*level)(void *ud);            /* bar ОПЦ.: уровень для подписи; NULL -> без уровня */
} resource_panel_entry_t;
/* Контракт NULL: `value` ОБЯЗАТЕЛЕН (панель зовёт его всегда). `max`/`level` —
   опциональны: панель ГАРДИТ их на NULL перед вызовом. `max==NULL` -> форма bar без
   знаменателя рисуется как счётчик-с-меткой (без заливки) ИЛИ игнорится (см. §6.3 п.1). */

/* Рисует entries по строке (порядок = вертикальный z/layout). Вызывается КАЖДЫЙ кадр
   игрой (из game_features_draw_ui через композицию). Внутри: poll геттеров -> diff ->
   count-up displayed -> акцент при смене -> render (счётчик/бар). */
void resource_panel_ui(nt_ui_context_t *ctx, const resource_panel_entry_t *entries, int count);

/* Шов под будущую coin_fx: экранная позиция (центр иконки/значения) последнего
   отрисованного entry по id. false если id не рисовался в этом кадре. Транзиентно
   (пересчитывается каждый кадр). ЕДИНСТВЕННЫЙ аксессор шва (карточка). */
bool resource_panel_anchor(const char *entry_id, float *out_x, float *out_y);

#endif /* FEATURES_RESOURCE_PANEL_H */
```

**Обоснование entry-struct против «панель без фича-хедеров» (греп-гейт):** entry несёт
СТРОКИ + указатели-на-функции + `void*ud` + движковые типы (`nt_atlas_region_ref_t`,
`nt_ui_context_t`) — НИ ОДНОГО типа из items/progression. Игра-композиция (demo_hud.c)
поставляет геттеры, замыкающие на `items_purse`/`progression_*`. Панель переиспользуема
в игре БЕЗ items (любой геттер `int64_t(void*)`). Три опц. геттера (value/max/level)
покрывают обе визформы: counter = value; bar = value/max + level-подпись.

### 6.3 `src/features/resource_panel/resource_panel.c` (поведение v1 — ратифицировано)

Includes: `"features/resource_panel/resource_panel.h"`, `"game_format.h"` (L0-abbrev,
легальный вниз), `"clay.h"`, `"ui/nt_ui_label.h"`, `"ui/nt_ui_image.h"`, `"ui/theme.h"`
(`g_theme` для стилей), `"app/nt_app.h"` (`g_nt_app.dt`), `<math.h>/<string.h>`.
НИ ОДНОГО `features/*` include (греп-гейт).

**Транзиентное состояние (static, keyed by entry.id — НЕ в сейве, не в World):**
```c
typedef struct { char id[32]; bool seen; double displayed; int64_t last_logical;
                 float accent; /* 0..1 таймер акцента */ float anchor_x, anchor_y; bool drawn; } rp_slot_t;
static rp_slot_t s_slots[RESOURCE_PANEL_MAX_SLOTS]; /* напр. 16 */
static rp_slot_t *slot_for(const char *id); /* find-or-alloc по id */
```

**Поведения v1 (РОВНО эти, ничего сверх — карточка):**
1. **displayed≠logical + count-up ease-out.** Каждый кадр: `logical=entry.value(ud)`
   (`value` ОБЯЗАТЕЛЕН, §6.2). **Порог снапа (#10 — max может быть NULL у counter!):**
   `snap_ref = (entry.kind==BAR && entry.max) ? max(1, entry.max(ud)) : max(1,
   llabs(logical))` — для бара база = знаменатель, для счётчика (max отсутствует) =
   |текущее значение| (или абсолют `SNAP_ABS≈1000`, что больше). Снап если `!slot->seen`
   ИЛИ `llabs(logical - last_logical) > snap_ref/4` (≈25% базы) ИЛИ первый кадр →
   `displayed=logical`; иначе экспон. ease-out `displayed += (logical - displayed) *
   (1 - expf(-dt / TAU))`, `TAU≈0.12` (визуально ~0.3–0.8s). **Ретаргет-не-рестарт:**
   цель — ВСЕГДА текущий `logical` (нет перезапуска с нуля). `slot->seen=true`. НИКОГДА
   не вызывать `entry.max(ud)`/`entry.level(ud)` без NULL-гарда.
2. **Акцент gain/spend.** Если `logical != last_logical` → `slot->accent=1.0` +
   знак (gain=зелёный/spend=красный тинт). Каждый кадр `accent = max(0, accent -
   dt/ACCENT_SECS)` (ACCENT_SECS≈0.35). Рендер: цвет лейбла = lerp(нормальный,
   акцент-цвет, accent). **punch (#14) — БЕЗ reflow:** punch-scale НЕ через шрифт-размер
   (двигает соседей 0.35с), а масштабом ВНУТРИ ФИКСИРОВАННОЙ ячейки лейбла (Clay-бокс
   фиксированного размера, контент центрирован и слегка увеличен — `scale = 1.0 +
   0.12*accent`), либо цвет-only-акцент если фикс-масштаб дорог. `last_logical=logical`.
3. **poll+diff.** Значение читается ТОЛЬКО геттером раз/кадр; сравнение с
   `last_logical` — сам diff (авто-коалесинг идл-спама: панель не читает event-лог,
   не занимает курсоры).
4. **Форматирование.** Отображаемое число = `game_format_i64_abbrev((int64_t)
   llround(displayed), buf, sizeof buf)` (L0-хелпер). Бар-подпись: `"Lv %lld  %s/%s"`
   (level, abbrev(value), abbrev(max)); на max_level (`max<=0`) → `"Lv %lld  MAX"`.
5. **Счётчик (COUNTER).** Иконка (или фолбэк-rect если `icon==NULL`) + abbrev-число,
   `nt_ui_label` со стилем `g_theme.label`/локальным `label_style`. Clay-id
   `resource_panel/<entry.id>`.
6. **Бар (BAR).** Форма `top_hud_bar` (rb-dark `first_screen_hud.c:165-191`,
   портировать локально): Clay-rect трек (`backgroundColor`+`border`+`cornerRadius`)
   + плавающий fill-child шириной `(track_w - inset) * ratio`, `ratio =
   max>0 ? clampf(displayed/max, 0, 1) : 1`. **displayed используется для fill (плавно
   ползёт), logical — для diff/акцента.** Подпись через `shadowed_label` (портировать
   `first_screen_hud.c:82-97` локально — ui-kit не даёт тень штатно; минимально).
   Clay-id `resource_panel/<entry.id>/bar`.
7. **graceful no-art.** `icon==NULL` → рисуем плоский rect-плейсхолдер ИЛИ пропускаем
   иконку (панель = ui-kit rect + лейбл). Панель НЕ падает и НЕ требует атласа. Если
   `icon!=NULL` → `nt_ui_image` с копией ref (ref мутируется резолвом — копировать в
   локальную mutable). ДЕМО передаёт `icon=NULL` (арта нет — И3 не добавляет ассеты).
8. **anchor (best-effort — #11).** immediate-mode: ТОЧНАЯ экранная позиция известна
   только ПОСЛЕ `ui_runtime_end` (EndLayout), а `nt_ui_context_layout_size` даёт лишь
   размер канваса — поэтому точный центр «во время build» НЕ вычислим. В И3 ратифицировано
   ТОЛЬКО НАЛИЧИЕ шва. Реализация — best-effort: сохранить ГРУБУЮ аппроксимацию
   (позиция строки entry по её порядковому индексу в известном layout панели, напр.
   top-left угол панели + i*row_h) в `slot->anchor_x/y`, `slot->drawn=true`;
   `resource_panel_anchor(id,&x,&y)` возвращает её (false если id не рисовался этот кадр).
   **Помечено «приблизительно; точную позицию по Clay-id уточнит coin_fx»** (пост-layout
   запрос `Clay_GetElementData(hash)` по `resource_panel/<id>` — забота будущей фичи, не И3).

Текст — ТОЛЬКО `nt_ui_label`/движковый рендерер (инвариант). НИКАКОГО `hud_text`/
handmade. НЕТ потребления событий, НЕТ персист-стейта, НЕТ полёта/floater/звука.

**Латентное (#16, задокументировать):** транзиент `s_slots` (keyed by entry.id) НЕ
инвалидируется при смене сейв-слота в СЕССИИ (напр. load другого профиля без рестарта).

```c
static int64_t get_hero_xp(void *ud){ (void)ud; return progression_xp_current("hero"); }
static int64_t get_hero_needed(void *ud){ (void)ud; return progression_xp_needed("hero"); }
static int64_t get_hero_level(void *ud){ (void)ud; return progression_level("hero"); }

/* Демо idle-доход xp: float-аккумулятор в game glue -> флаш в i64 (паттерн Р1;
   ДРОБНОЕ производство копит в glue, НЕ в count). Кормит purse tmpl.xp; progression
   auto-трек hero авто-покупает уровни -> живой count-up + levelup для глаза лида. */
#define DEMO_XP_PER_SEC 8.0F
static float s_xp_accum;
void demo_hud_update(float dt) {
    s_xp_accum += DEMO_XP_PER_SEC * dt;
    if (s_xp_accum >= 1.0F) {
        int64_t whole = (int64_t)s_xp_accum; s_xp_accum -= (float)whole;
        items_add("purse", "tmpl.xp", whole, "loot:demo_idle");
    }
}
void demo_hud_draw_ui(nt_ui_context_t *ctx) {
    const resource_panel_entry_t entries[] = {
        { .id="gold", .label="Gold", .kind=RESOURCE_PANEL_COUNTER, .value=get_gold },
        { .id="hero", .label="Hero", .kind=RESOURCE_PANEL_BAR,
          .value=get_hero_xp, .max=get_hero_needed, .level=get_hero_level },
    };
    resource_panel_ui(ctx, entries, 2);
}
```

## 9. Open questions (с дефолтами — лид ратифицирует быстро, one-at-a-time)

- **OQ1 — треки как ДАННЫЕ (content/progression.json + codegen) vs рукописный C-конфиг.**
  ДЕФОЛТ: **данные + codegen** (`generate_progression_tracks.py`, зеркало items-каталога).
  Основание: дизайн §5 явно «треки объявляет игра данными (tracks.json)»; кривая-пресет
  «допекается codegen'ом в таблицу, рантайм читает таблицу» ТРЕБУЕТ Python-запекания
  (exp/poly в C = запрещённый интерпретатор формул); арка строится ПО конвенции, а items
  уже поставил контент-codegen. Отвергнуто: рукописная C-таблица (либо руками пекут ту же
  кривую = дубль, либо тянут формулы в C = нарушение дизайна). Тест-каталог — рукописный
  (контролируемые/патологические треки), это ОРТОГОНАЛЬНО (тест владеет своими входами).

- **OQ2 — порядок регистрации фрагментов.** ДЕФОЛТ: `settings → items → progression →
  game` (вставить progression МЕЖДУ items и game). Основание: progression (L2) зависит
  от items (L1) — регистрировать после зависимости; game «last, most dependent».
  Нейтрально для LOAD (progression без hooks, from_json читает только свои tracks; purse
  читается лишь на тике, после полной загрузки). Минорно.

- **OQ3 — hooks фрагмента progression (on_new_game/reconcile).** ДЕФОЛТ: **НЕТ hooks**
  (reset = пустые tracks). Основание: свежей игре прогрессия НЕ нужна засеянной (уровень
  0 = ленивый дефолт; кейс лида «сильный герой в прологе» — это game-код зовёт
  `progression_set_level`, НЕ on_new_game); сирота-трек-запись безвредна (`update`
  итерирует только каталог-треки, orphan игнорится) — restoration-заботы items тут нет,
  reconcile не нужен. Отвергнуто: on_new_game-сид (нечего сеять) / reconcile-prune
  (YAGNI — orphan не вредит, `max_count=32` c запасом).

- **OQ4 — расположение общего форматтера.** ДЕФОЛТ: `src/game_format.{c,h}` (L0-лист,
  как `game_log`/`game_save`; вне `features/`, включается фичами вниз). Основание:
  конвенция README — L0-шелл в `src/`; 2 потребителя (resource_panel, будущая coin_fx) →
  общий, не per-feature. Отвергнуто: `src/features/resource_panel/` (coin_fx тогда
  включала бы чужую фичу — нарушение) / `src/ui/` (не UI-специфичен, чистый int64→string).

- **OQ5 — где демо-биндинг resource_panel.** ДЕФОЛТ: `src/ui/demo_hud.{c,h}` (game-owned
  композиция, как `hud.c`); `game_features.c` зовёт `demo_hud_update`/`demo_hud_draw_ui`
  ОДНОЙ строкой каждую (агрегатор остаётся тонким — «по строке на фичу»). Основание:
  композиция игры (геттеры, связывающие items+progression с виджетом) — это код игры, не
  фичи; main.c — thin-conductor, не место для биндинга. Отвергнуто: биндинг прямо в
  `game_features.c` (раздувает агрегатор геттерами+entries — против «одна строка на фичу»)
  / в main.c (нарушает thin-main).

- **OQ6 — режим и «живость» демо-трека.** ДЕФОЛТ: `hero` = **auto**, `currency_def=
  tmpl.xp`, + демо idle-xp-доход в game glue (demo_hud). Основание: делает слоёвость
  ВИДНОЙ в демо (xp_current = `items_purse` — реальный include-путь), а не латентной;
  idle-доход + auto-автопокупка = живой count-up/levelup для глаза лида (тема студии —
  idle) БЕЗ кнопок (панель display-only, кнопок в v1 нет); статический гейт = корректность
  рендера (не требует точного значения). Отвергнуто: threshold-демо (include items.h был
  бы латентным, демо не показал бы purse-путь) / manual-демо (нужна кнопка, панель её не
  даёт). Числа демо (`DEMO_XP_PER_SEC=8`, кривая base=50/×1.5, max_level=20) — минорный
  дефолт, лид может занулить/поменять; лид может и вовсе убрать idle-доход (тогда бар
  статичен на 0/cost — гейт всё равно проходит по рендеру).

Свёрнуто в дефолты (НЕ форки): **curve-тип = ТОЛЬКО `exp`** (LEAN-порез B: linear/table/

**Возражений к фикс-листу НЕТ** (все H/M/L приняты; LEAN-порезы A/B/C приняты как
