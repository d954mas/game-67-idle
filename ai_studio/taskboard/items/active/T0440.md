## What

`DevApiClient` opened its socket file as `self.sock.makefile("rwb", buffering=0)`
(`ai_studio/runtime_automation/devapi_client.py:51`). With `buffering=0`,
`readline()` reads **one byte per syscall**. A `capture.frame` response for a
visually dense frame (e.g. a private game's 400-walker crowd/game testbed scene
at 1280x720) is ~700 KB of base64 → the client drains
it for ~3.7 s. The engine's writer gives up after
`NT_DEVAPI_NET_SEND_TIMEOUT_MS` = 2000 ms
(`external/neotolis-engine/engine/devapi/nt_devapi_net.c:63`, `send_all` :167,
deferred drain → `close_client` :413-419) and **drops the client connection**
mid-response-stream. The client sees a truncated stream / EOF ("empty
response"), which looks like an engine crash but the engine process stays
alive.

Confirmed 2026-07-17 while capturing reel footage: 6/6 reproducible EOFs on the
first large capture in the crowd/game scenes; the walker scene (1 body, small
PNG) survived 240+ captures. A buffered reader (`recv(65536)` chunking) reads
the same 700 KB line in ~0.085 s and the identical capture sequence completes
(360 frames, 0 errors). Diagnostics scripts + full root-cause analysis:
deep-reasoner session artifacts `diag.py`/`diag2.py`/`diag3.py` (scratchpad,
2026-07-17); reel capture workaround in
`capture_reel.py::FastFile` (session scratchpad).

Why existing tests never caught it: smoke bots and iterate.py capture a single
proof PNG of a nearly-empty template scene — the response is small enough that
even the byte-at-a-time reader drains it well under 2 s. The failure needs a
dense frame (big PNG) or `capture.region` of a busy area, which no current
check exercises.

## Done when

- [x] `DevApiClient` reads responses through a buffered path (e.g. keep writes
      on `sock.sendall`, reads via `sock.makefile("rb")` or an explicit
      `recv()` line splitter) — full-frame dense-scene reads take ~0.1 s, not
      ~3.7 s.
- [x] No regression for interleaved read/write callers (batch, deferred
      frame.wait, recording helpers) — the runtime_automation unit suite stays
      green.
- [x] A regression check exercises a LARGE capture response (dense frame or
      synthetic big payload) so a future slow-reader/timeout interaction fails
      loudly instead of masquerading as an engine crash.
- [x] Decide whether `NT_DEVAPI_NET_SEND_TIMEOUT_MS` (2 s) should also grow or
      log a distinguishable error server-side when it drops a client
      mid-response (currently the client just sees EOF).

## Decision

- Keep the 2 s engine timeout unchanged. A distinguishable server-side timeout
  log may be proposed through a separate engine issue/PR; it is not required to
  correct this client-side root cause.
