## What

Build the first commercial-factory slice: define reusable publish-target adapters, a universal platform SDK wrapper, and a compact analytics scorecard so each web game can be packaged, checked, published or queued for a target, and judged by the same metrics.

Target platforms are `local`, `itch`, `poki`, `yandex`, and `playgama`. Platform SDK adapters are `mock`, `poki`, `yandex`, and `playgama`. `mock` is not a target: `local -> mock` and `itch -> mock` because itch has no required runtime game SDK. The old JS template path `C:\projects\ai_games\game-ai-template\src\features\core\platform` is a reference for shape only, not the source of truth for this repo. Playgama Bridge is one platform SDK adapter, not the universal wrapper.

This should sit above existing template web delivery work, not duplicate `tools/build_web.sh`, `serve_web.mjs`, or T0323's package/smoke implementation. It should consume those outputs and define target-specific acceptance.

The first runtime implementation belongs in this same task: a mock platform SDK adapter and template debug/test controls that make ad flows verifiable before real portal SDK adapters are wired.

The old JS reference method set was: `whenReady`, `gameLoadingFinished`, `gameReady`, `gameplayStart`, `gameplayStop`, `commercialBreak`, `showRewarded`, `onPause`, `onResume`, `onAudioToggle`, `onVisibilityChange`, `measure`, `loadData`, `saveData`, and `destroy`. Use that as behavior reference only; keep the new API aligned with `features/platform-sdk/references/contract.md`.

This task also owns the bridge from platform SDK events into the L0
`features/game-events` event/analytics feature pack that T0328 must extract
from the current template-root `game_events`/`game_analytics` files. Do not
create a new event system for SDK telemetry: SDK lifecycle and ad-flow events
should land in the reusable event spine so the local scorecard can mix
game-originated and SDK-originated events from the same NDJSON/export fixture.
Runtime identity (`target`, `platformSdk`, capabilities) stays available through
the wrapper API and/or session/export context; it is not duplicated into every
event payload.

- [x] A CLI or documented command path can produce a scorecard from a local NDJSON/export fixture without requiring a live portal account.
- [x] Validation evidence is recorded: taskboard item validates, and any new docs/commands pass their narrow checks.
- [x] `features/game-events` exists as an L0 feature pack, installed in the template in-place like `game-state`; T0339 must not close the SDK event bridge against anonymous template-root event files.
- [x] Platform SDK events are bridged into `features/game-events`, not a parallel system: at minimum `platform.ready` (`ready`), `game.loading_finished` (no payload), `gameplay.start` (no payload), `gameplay.stop` (no payload), `ad.interstitial.request/result` (`placement`, `supported`, `shown`, `reason`), and `ad.rewarded.request/result` (`placement`, `supported`, `shown`, `rewarded`, `reason`). `placement` values are finite stable IDs. Do not duplicate `target`, `platformSdk`, `source`, or timestamps into each event payload; those belong to runtime/session/export context. Do not add platform-sdk events for capability reads, `game.ready`, destroy cleanup, session end, save writes, gameplay-start guard failures, pause/resume/audio/visibility callbacks, storage load/save calls, or analytics forwarding (`track`/`measure`); those are portal SDK method calls, app/game lifecycle, runtime callback, storage API, warning, or analytics-sink concerns.
- [x] `platform_sdk_init()`, `platform_sdk_game_loading_finished()`, and `platform_sdk_game_ready()` are idempotent C facade lifecycle calls; `gameReady()` emits no analytics event, and later asset-pack loads are owned by the asset/loader feature and must not re-emit `gameLoadingFinished()`. `platform.ready` and `game.loading_finished` are emitted by the C facade through `features/game-events`.
- [x] JS no longer owns platform-sdk scorecard emissions or runtime guards: `gameplay.start.blocked`, `platform.destroy`, `save.write`, `input.first`, lifecycle/ad-flow events, and per-event `target`/`platformSdk`/`source`/timestamp duplication are not emitted by a JS local event sink.
- [x] The JavaScript backend layer does not expose a public `track(event)` event path; the stable game event source remains `features/game-events`, and any future portal analytics forwarding must subscribe through that bridge.
- [x] Scorecard fixture/tests include SDK-originated events and prove the scorecard can read a mixed game + platform SDK NDJSON/export stream without a live portal account.

## Open questions

- Should the first runtime implementation be added directly under `features/platform-sdk`, or prototyped in a JS template and promoted once the adapter API is stable?
- Should mock overlay support a visible countdown only, or also explicit skip/close for interstitial?
- Should `itch -> mock` ever allow debug overlay in non-release preview builds, or always no-op to mirror production?
- Where should the template debug/test UI live so it is available to copied games but gated out of release builds?
- Which Yandex SDK path is the default for builds: archive-hosted `/sdk.js`, custom-domain absolute SDK URL, or both as target modes?
- Do we need a separate `self_host` target now, or is `itch` plus `local` enough until the first published game?
