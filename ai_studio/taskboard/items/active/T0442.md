## What

Any game built on the template + platform-sdk local mock target boots to a
loading overlay stuck at "Running... 45%" forever, while the game itself runs
and renders perfectly underneath (proven by force-hiding the overlay and by
in-page devapi: full scene draws, pack state ready). The 45% is index.html's
own onRuntimeInitialized value; ZERO progress/hide calls from the C side ever
reach JS (pre-load hooks counted only the 4 shell-side calls).

Confirmed defect #1 — `features/platform-sdk/src/platform_sdk.c`
`platform_sdk_game_loading_progress`: the monotonic filter sets
`loading_progress_sent` even when NO backend is installed yet. The template
main calls progress every frame from boot; the web backend installs later
(inside the game-runtime start barrier), so the first post-install call with
an equal value is swallowed forever. Fix candidate: mark sent only when the
value was actually forwarded to a backend.

Open defect #2 — even after forcing `Module._platform_sdk_web_complete_init(1)`
(status -> BOOT_READY), `after_frame_present` never emits
loading_finished/hide on web; the readiness/lifecycle chain needs pinning.

Latent, not a regression: `portal_evidence` releaseLevels() honestly records
local-mock as "unverified — Package verification does not run or record a
local mock simulator session". This browser path was never proven for any
game; it surfaced on the first full headless browser run (2026-07-17, a
private depth-3 game; the full session write-up with capture paths lives in
that game's design/knowledge).

Repro (any template game): `node tools/build_web.mjs && node
tools/serve_web.mjs --preset wasm-release --target local`, open in Chrome.

Introspection method that cracked it (promote to ai_studio/runtime_automation):
raw CDP over node's builtin WebSocket (no puppeteer — Avast-proof): console +
exception capture, pre-load JS hooks via Page.addScriptToEvaluateOnNewDocument,
and in-page engine devapi on a wasm-debug build with GAME_DEVAPI_ENABLED=ON
(stale caches keep NT_DEVAPI_GROUP_* FORCE-OFF — pass CORE/TIME/INPUT/DISCOVERY
explicitly) + a Module setter hook injecting `arguments: ['--devapi', ...]`,
then `window.__devapi.submit(...)`.

## Done when

- [ ] Local mock target hides the overlay through the normal C lifecycle
      (progress reaches JS, loading_finished fires, no force-hide needed).
- [ ] The platform-sdk progress filter cannot swallow the first real
      forwarded value (regression test in the feature's test suite).
- [ ] Defect #2 (hide chain after forced READY) is root-caused and fixed.
- [ ] A headless browser check exists (runtime_automation) so local-mock web
      runs are verifiable instead of "unverified" in portal evidence.
