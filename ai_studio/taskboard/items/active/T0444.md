## What

Close the post-merge Items Runtime v2 correctness gaps that are hidden when
`NT_ASSERT_MODE=0`: payment fingerprint construction must not depend on assert
side effects, public transaction inputs must fail safely, partial same-container
moves must preserve unique slots, and bound catalog costs must remain valid for
runtime payment semantics.

## Done when

- [ ] Assert-off regression coverage proves paid payment/acquire/upgrade initialize
      every catalog ref used by transaction fingerprints.
- [ ] Partial same-container AUTO moves either produce one valid split slot or
      return a normal refusal without mutating state.
- [ ] Invalid/oversized payment scopes, stale refs, non-stackable cost resources,
      and overflowing cost totals fail before state mutation or audit emission.
- [ ] Focused Items runtime/package tests, the full mandatory verify, and
      Ubuntu/Windows CI pass with QTECH evidence recorded.
