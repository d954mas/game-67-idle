# Visual Proof

Load this when producing visual GDD deliverables, fake gameplay screens,
generated art direction, prepared asset proofs, or review packets.

## Tiers

- Reference: mood, tone, competitor, meme, UI density, or art target. Not enough
  for implementation.
- Fake shot: composed gameplay screen showing what the player sees, does,
  wants, and can press.
- Runtime proof: running-game screenshot or capture proving the implemented
  screen matches the accepted direction.
- Prepared asset proof: separate production-oriented images with manifest,
  dimensions, transparency expectations, and composed proof.

## Fake Shot Checklist

A fake shot must show:

- player avatar or main object;
- current status/progression number;
- 2-4 top currencies/stats;
- primary action button or active activity;
- one upgrade or choice;
- next unlock or goal;
- feedback/result from the last action;
- environment, home, or status visual state;
- enough UI density to imply the real product genre.

Reject posters, splash art, isolated characters, generic landing pages, and mood
boards when the user asked to see gameplay.

## Runtime Asset Proof

Prepared game-use visuals need:

- separate character/object images where compositing is needed;
- transparent backgrounds for sprites/UI when needed;
- backgrounds separate from characters and UI;
- UI frames/buttons/icons as separate files or implementation-native shapes;
- manifest with id, file, dimensions, usage, and fallback;
- composed screen proving separate assets recreate the target;
- ignored raw sheets, failed generations, and temporary composites.

## Review Packet

After the first strong fake shot or art direction, stop and report:

- image path(s);
- visual tier;
- what gameplay is visible;
- what is intentionally placeholder;
- what should be accepted, changed, or regenerated;
- next gate if accepted.

Ask the user to choose: keep direction, adjust specific elements, or regenerate
from a different reference.
