## Strengths (verified, credit where due)

- **Parity discipline is real and near-total.** Every mutating op exported by `ops.mjs` is wired
  to BOTH a CLI verb and an `api.mjs` route (I diffed all 60 exports at ops.mjs:402-4863 against
  cli.mjs:255-892 and api.mjs:194-905). The CLI and API are thin marshallers over the identical op
  layer — no logic fork. This is the invariant holding.
- **Remedy-naming errors are genuinely good in the high-value spots.** undo/redo/history-jump
  (cli.mjs:845,852,877) don't just say "missing flag" — they say *"the project may be live; run
  history-list to read the current head, then retry with it"*. promote-recipe/promote-style name
  "run extract first". The Python-missing error names the exact one-shot setup command
  (bridge.mjs:133-143). This is the right pattern; it is just not applied uniformly (see F5).
- **The `--preview` split is a strong pattern.** quantize/denoise with `--preview` compute + write
  the PNG + print the report with **zero journal writes** (cli.mjs:578-584), apply-mode without it.
  "Cancel is free" is a real ergonomic win and is correctly mirrored on both clients.
- **Singular-vs-plural IS an intentional, learnable system.** `element-set` (one, by flag) vs
  `elements-set` (batch, by --json); `node-reorder` vs `nodes-reorder`. This is consistent and
  should be documented as the rule, not "fixed".
- **Batch verbs already collapse the hot multi-object paths into one op + one undo** (add-images,
  elements-set, elements-remove, nodes-move/-reorder/-delete/-duplicate, alpha --elements,
  groups-set). This is the correct answer to per-call overhead for the common cases.
- **Actor attribution + error trail**: the CLI stamps every mutation as agent-made
  (setOpsActor("agent"), cli.mjs:919) and logs op failures to `<project>/errors.jsonl`
  (cli.mjs:905). Solid observability.

## Findings ranked

### F1 (HIGH) — `list` dumps every project in full; this is the 73KB/95KB overflow
**WHAT** `list` returns the complete project object for every project (all elements incl. their
`regions` arrays, all groups, `tool_runs`). **EVIDENCE** cli.mjs:257 `listProjects(repoRoot)` →
store.mjs:167-181 pushes `readProjectFile` (the whole project) per folder. Measured: 2 projects =
**95,332 bytes** on stdout. It scales with total canvas content across ALL projects, unbounded.
**WHY** A project picker needs id/title/updated/counts — not every element's pixels-metadata. This
is the buffer overflow the lead already hit, and it will only get worse. **FIX (S)** `list` returns
a summary `[{id,title,updated,elements:N,groups:N,history_seq}]` by default; add `--full` for the
current behavior. `show` stays full. Low blast radius: no found consumer parses element data out of
`list` (the skill routes id-resolution through `show`, SKILL.md:29-32).

### F2 (HIGH) — Every mutation echoes the entire project; no `--quiet`/`--summary`
**WHAT** All ~45 mutating verbs `print(...)` the full project the op returns (group-create →
`{project, group}`, move → `{project, element}`, etc.). There is no compact mode. **EVIDENCE**
cli.mjs:153-155 `print` = `JSON.stringify(value)`; ops return the whole project (e.g.
patchElement, ops.mjs:565). A single move on a busy project re-emits every element. **WHY** For an
agent loop this is the *dominant* cost — context tokens, not the 85ms process start. The agent
actually needs only: the new id(s) and the new head (for the next `--expect-head`). **FIX (M,
staged)** Add an opt-in `--quiet` that prints `{ok:true, op, ids:[new ids], head, duration_ms}`;
teach the skill to prefer it. Do NOT flip the default yet — the full-project return is a tested,
load-bearing contract (cli.test.mjs:46-99 asserts `.project.elements[0].x`, `.project.history_seq`;
the chat agent is told to run `show` for full state, agent.mjs:70-73). Own section below.

### F3 (HIGH) — No single-element resize verb (`--w`/`--h`); parity gap vs API/UI
**WHAT** You can *move* one element (`move --x --y`) and set name/visible/content/style/rotation/
flip (`element-set`), but nothing on a single element sets `w`/`h`. **EVIDENCE** cli.mjs:321-352:
`move` wires only x/y; `element-set` wires name/visible/content/style/rotation/flip — neither
touches w/h. Yet the op accepts w/h (patchElement → sanitizeTransformPatch, ops.mjs:556) and the
API PATCH route passes `{x,y,w,h,...}` (api.mjs:53,876). To resize ONE element from the CLI today
you must hand-author a JSON file and use the `elements-set --json` batch path. **WHY** Resize is a
first-class inspector gesture on the page; the agent can't do it ergonomically → tool-parity dent.
**FIX (S)** Add `--w`/`--h` (and `--x`/`--y`) to `element-set`, forwarded to patchElement. Fold
`move` into it as an alias, or keep `move` as a shorthand.

### F4 (MED) — Verb grammar has four competing patterns; a fresh agent guesses several wrong
**WHAT** The surface mixes (a) bare project CRUD (list/create/show/rename/delete), (b) verb-noun
minting (add-image/add-text), (c) scoped noun-verb (group-*/element-*/nodes-*/recipe-*), and (d)
bare element-transforms that still require `--element` (move/slice/alpha/quantize/denoise/extract).
**EVIDENCE** usage banner cli.mjs:188. The high-probability wrong guesses:
- `move` is the ELEMENT move, but the GROUP move is `group-move` → an agent symmetric-guesses
  `element-move`. (cli.mjs:321 vs 676)
- Three words for delete: project `delete`, group `group-delete`, nodes `nodes-delete`, but element
  `element-remove` → agent guesses `element-delete`. (remove vs delete split, cli.mjs:353 vs 746)
- Minting splits create/add: `create`(project) / `group-create` / `add-image` / `add-text` → agent
  guesses `element-create` or `create-group`.
- Renaming: project `rename`, element/group via `*-set --name` → agent guesses `element-rename`.
- Element transforms (`alpha`,`quantize`,`denoise`,`slice`,`extract`) have no scope prefix though
  they're element-scoped → agent guesses `element-alpha`.
**WHY** Every wrong guess is a wasted CLI round-trip + a usage-dump the agent must re-read. **FIX
(S, additive)** Don't rename anything. (1) Add the 3-4 highest-value aliases: `element-move`→move,
`element-delete`→element-remove, `element-alpha`→alpha (same dispatch, one `case` fallthrough
each). (2) Document the rule in grouped help (F6): *"bare verbs = project CRUD + element-transform
pipeline (operate on `--element`); everything else is `<scope>-<verb>`."*

### F5 (MED) — `--visible` is silently coerced, with OPPOSITE garbage-defaults across verbs
**WHAT** `--visible` accepts non-boolean junk silently, and the junk resolves to *false* in
`element-set` but *true* in `group-set`. **EVIDENCE** element-set: `patch.visible =
flags.visible === "true"` → `--visible maybe` = false (cli.mjs:334). group-set: `args.visible =
flags.visible` raw string; op does `!(v===false||v==="false")` → `--visible maybe` = true
(cli.mjs:689, ops.mjs:1453). Meanwhile `--clip` (cli.mjs:701) and `groups-set --visible`
(cli.mjs:714) DO validate strictly ("must be true or false"). So the *same flag* has three
behaviors. **WHY** An agent that passes `--visible 1` or `--visible yes` gets a silent, direction-
dependent wrong result — the worst kind of failure (no error, wrong state). **FIX (S)** Route all
boolean flags through one strict helper that errors on anything but true|false (the `--clip`
pattern), everywhere.

### F6 (MED) — Help is a 64-line flat wall; no grouping, no discoverability of preconditions
**WHAT** `usage()` (cli.mjs:187-252) is one alphabetical-ish block of ~58 verbs. Preconditions that
gate success (extract-before-promote; `--expect-head` on undo/redo/jump; "run history-list first")
live inline per line and are easy to miss at scale. **EVIDENCE** the pipe-separated header line 188
lists 55 verbs on one line; the body has no section breaks. **WHY** An agent scanning for "how do I
resize / recolor / cut out" has to linear-read 58 lines each session. **FIX (S)** Group the help
into ~8 headed sections (Projects · Elements · Nodes/z-order · Groups · Alpha/cleanup · Recipe/Style
· Export · History) and add a one-line "preconditions" callout (expect-head; extract→promote).
Purely a print-formatting change.

### F7 (LOW) — Boolean flag style is documented two ways (`--dither` vs `--flip-h true|false`)
**WHAT** Presence-style booleans (`[--dither]`, `--clear`) coexist with explicit-value booleans
(`--flip-h true|false`, `--visible true|false`, `--clip true|false`). **EVIDENCE** quantize doc
`[--dither]` (cli.mjs:34,221) vs element-set `[--flip-h true|false]` (cli.mjs:16,200). Both actually
*accept* bare presence (parseFlags gives "true"), so the docs disagree with each other, not the
code. **WHY** Minor, but an agent copying the documented form pointlessly varies its calls. **FIX
(S)** Pick one documented convention (recommend explicit `true|false` for state toggles, bare
presence only for one-shot actions like `--clear`/`--preview`) and align the doc strings.

### F8 (LOW) — Exit code is `1` for both usage errors and op failures
**WHAT** Argument-validation `fail()` and op-throw both exit `1` (tool_lib/cli.mjs:14-17; the CLI's
two error paths both funnel to `fail`). **EVIDENCE** cli.mjs:263 `fail("show requires <id>")` and
cli.mjs:920 `.catch(...=>fail(...))` — same exit. **WHY** An agent can't distinguish "I called it
wrong" (retry with fixed args) from "the op legitimately failed" (don't retry) by exit code; it must
parse stderr text. **FIX (S)** Usage/validation → exit 2, op failure → exit 1 (Unix convention).
Additive; nothing keys on exit 1 vs 2 today.

### F9 (LOW) — `ops-stats` is a hidden verb; `historyFlags` op has no CLI verb
**WHAT** `ops-stats` works (cli.mjs:881) but is absent from the usage banner and the README CLI
list → undiscoverable. `historyFlags` (ops.mjs:3069, used by api.mjs:208 to fold canUndo/canRedo
into every mutation response) has no CLI verb. **WHY** ops-stats is the agent's self-profiling
window (it's in the API at api.mjs:789) — parity says it should be listed. historyFlags is a
strict subset of `history-list` (which prints head + full spine), so the gap is cosmetic. **FIX
(S)** Add `ops-stats` to the banner + README. Leave historyFlags unexposed (document that
history-list is its superset).

## Do-differently #1 — The output contract (argued)

**Today.** Every mutation prints the full project (F2); `list` prints every project in full (F1).
The full-project return is not incidental — it is a *tested contract* (cli.test.mjs:46-99 reads
`.project.elements[...]`, `.project.history_seq`) and the page's API twin deliberately folds even
MORE in (duration_ms + history flags, api.mjs:203-214) so the browser re-renders from the response
with no reload GET. So the full payload earns its keep **on the API/page side**.

**On the CLI side it does not.** The CLI's consumer is an LLM agent (codex in the chat panel,
agent.mjs:70), and its system prompt already says *"For the full project state … run … show
<projectId>"* (agent.mjs:73). So the agent is architecturally expected to fetch full state on
demand — it does NOT need every mutation to re-dump it. What it needs after a mutation is tiny:
the id(s) just minted, and the new head for the next `--expect-head`.

**Recommendation (two moves, different risk):**
1. **`list` → summary by default + `--full`.** Do now. This is the acute overflow, the summary is
   strictly more useful for the "which project" decision, and I found no consumer that parses
   element data out of `list`. Low risk.
2. **Mutations → additive `--quiet`** printing `{ok, op, ids, head, duration_ms}`. Ship opt-in,
   teach the skill to prefer it, and treat a *default flip* to quiet (with `--full` as the escape
   hatch) as a SEPARATE, later step — it breaks cli.test.mjs's contract and any implicit reliance,
   so it needs a coordinated update of cli.test.mjs + the SKILL doc + the chat agent prompt. Flip
   only after `--quiet` proves out. What would break on a naive flip: (a) cli.test.mjs assertions;
   (b) any agent flow that reads `.project.elements` straight from a mutation's stdout instead of
   `show` (none found in-repo, but the chat agent's freeform behavior can't be fully audited — an
   opt-in avoids betting on that).

Net: fix the read path (`list`) aggressively; make the write path opt-in-lean and migrate the
default deliberately.

## Do-differently #2 — Startup cost (argued, with numbers)

**Measured on this box (v24.15.0):**
- bare `node -e 0`: **~60 ms**
- CLI metadata/geometry verb (show, group-create, move): **~80-90 ms** (node 60 + module import ~25)
- CLI **image verb** (detect-regions), 3 *separate* invocations: **1141 / 1025 / 1025 ms**

**The premise needs correcting.** Node startup is ~85 ms, not 0.5-0.7 s. The 0.5-0.7 s (and here, ~1
s) is **Python cold-start inside image verbs**, and it does NOT amortize across CLI calls. The warm
Python worker (T0202) is a *per-node-process* module singleton (worker.mjs:30-31, spawned lazily,
killed on process `exit` at worker.mjs:42) — so a long-lived server pays the ~165-278 ms interpreter
+ numpy/PIL floor once, but a one-shot CLI invocation pays it *every* image verb (detect-regions,
slice, alpha, quantize, denoise, export, render-group). The 3× ~1025 ms above (three fresh
processes, no drop on #2/#3) is the proof.

**So the real cost structure is:**
- Light verbs: ~85 ms each. A 50-command layout session = ~4.3 s of process overhead. Fine.
- Image verbs: ~1 s each, cold every time, because the warm worker can't survive the CLI process.
- Codex/agy verbs (recipe-generate, extract, expand, alpha-dual-generate): real minutes — inherent,
  not addressable here.

**Options, honestly:**
- *Accept the 85 ms (light verbs).* Yes — simplicity law wins. A REPL to save 85 ms/call is not
  worth a stateful long-lived process, and the store is re-read per call anyway so a REPL saves
  little I/O. The existing batch verbs already collapse the multi-object light paths.
- *The image-verb ~1 s is the one worth attacking* — and a persistent-worker CLI mode is the ONLY
  option that helps it, because the win is keeping the *Python* worker warm across ops (1 s → ~0.2 s
  for calls 2..N), not keeping node warm. Two framings: (a) a `--batch <file>` mode that runs N
  verbs in ONE node process (worker stays warm for the run) — cheap, additive, no daemon,
  reuses the existing singleton; (b) a stdin REPL — more complex, stateful, and only pays off for
  very long interactive agent sessions. Recommend (a) if/when an agent is observed doing many
  image ops serially; extend the batch-op pattern (alpha already batches via `--elements`) to
  detect-regions/slice/export first — that's the same win without any new transport.
- There's already a bench harness (tests/bench.mjs) that baselines these exact ops incl. "raw
  process-spawn overhead in isolation" — wire the decision to its numbers before building anything.

Bottom line: don't optimize the 85 ms. Either accept the image-verb second-cost or, if profiles
show serial image loops, add `--batch` (worker-warm-for-the-run) — argued by the 1025 ms number,
not by node startup.
