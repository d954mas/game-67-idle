# Canvas-based AI image workspaces and game-asset pipelines — UX/architecture research for the local 2D conveyor (2026-07-02)

## Summary

Across ~20 products, the industry has converged on one core primitive for per-region regeneration: **`(image, region_mask, optional_prompt) → N candidates`, written to a non-destructive layer/slot bound to that region, with a variants rail to swap/accept/discard**. Photoshop Generative Fill is the reference implementation; Ideogram Canvas has the cleanest formalization ("generation windows" you drag over sub-regions of "image frames"); InvokeAI's Unified Canvas is the best open-source implementation; Scenario adds the game-specific twist (per-region model choice + influence slider); Canva's Magic Grab is the best "promote region to independent layer + heal the hole" mechanic — which is nearly identical to the lead's slice step.

On build-vs-adopt: **no open-source canvas is worth embedding wholesale**. InvokeAI and ComfyUI+Krita are excellent but tightly coupled to a local diffusion GPU stack the lead doesn't run (his generator is remote gpt-image via Codex CLI); tldraw's SDK now requires a paid production license or a watermarked hobby license; Excalidraw is MIT but shape-whiteboard-shaped, not image-region-shaped. The strongest precedent for his exact architecture is **openOutpaint (MIT)** — a vanilla-JS canvas thin client over a local HTTP image API — which proves his existing stage-view approach scales to a full conveyor. Recommendation: **extend the existing thin canvas, copy the Photoshop/Ideogram/InvokeAI interaction patterns, and lift MIT code (openOutpaint, PixelRefiner) for specific pieces**.

## Entries

### Recraft (recraft.ai) — the lead's named reference
- **Interaction model:** true infinite canvas with Figma-like frames, pan/zoom, multiplayer. The "Layers" panel is deliberately shallow — a **flat, read-only selection/visibility index** over floating images (no reorder/rename/hierarchy). Also: Layer Alignment (snap images into rows/grids), and a Chat Mode for iterating on-canvas.
- **Per-region regen:** classic drawn-mask + reprompt ("Modify Area"/inpainting, retrained to handle "even the smallest regions"), plus **AI Eraser** (remove + heal) and a **lasso modify** tool — three thin UIs over one masking code path. Outpaint/expand for extension.
- **Style:** custom style = one base style (photo/illustration/vector/icon) + **up to 5 reference images**, saved as a reusable token; brand-color pinning. Exposed in the API (`create-style`).
- **Real vs marketing:** canvas, inpaint/erase/lasso, vectorize, custom styles all real and documented. Native SVG generation (V3) is genuinely differentiated. The weak spot is the layer system (not a compositing stack). Official canvas docs URL 404'd; canvas mechanics corroborated via reviews + Recraft's blog.
- **Licensing:** closed SaaS; public API (~$0.04/raster, $0.08/vector). Free tier outputs are **public with no commercial license**.
- **Copy:** flat layers-as-index over a canvas of floating images (cheap, useful); one mask engine with eraser/lasso/modify presets; ≤5-reference reusable style token; grid-snap alignment for slices.

### Scenario (scenario.com)
- **Interaction model:** API-first; "Canvas" is a **non-destructive, layer-based retouch editor** (duplicate/hide/merge/delete layers, per-layer effects), not a whiteboard; plus a node workflow builder for batch pipelines.
- **Per-region regen (best-in-class for this use case):** define region by **mask OR sketch** → **Influence slider** (obey guidance vs free generation) → **choose a model per region** (keep the original model for consistency, or swap) → prompt → generate N variants → pick one to keep.
- **Style:** deepest consistency stack — custom LoRA training (~10-50 images), model merging, Character Reference (single-image identity), and clean separation of reference intent: structure (ControlNet) / identity / style.
- **Game assets:** sprite sheet = one grid PNG you slice yourself; Pixelate; direct Unity/Unreal export. Real but the sprite path is manual.
- **Licensing:** closed SaaS; free tier non-commercial; paid plans (~$10-50/mo) grant IP ownership.
- **Copy:** mask-or-sketch + influence slider pairing; **per-region model dropdown**; generate-N-pick-one as the atomic loop; separating structure/identity/style reference channels.

### Layer.ai
- **Status:** alive — $6.5M seed May 2025, GDC 2026 presence, 200+ studios claimed.
- **Interaction model:** team workspace with Sessions + a searchable **Activity Feed** of every generation; advertises a "Realtime Live Canvas" and node workflows. "Forge" is a generation mode, thinly documented.
- **Per-region regen:** documented as **prompt-driven regional edit** ("edit or regenerate a specific region of a texture map") + reference-a-previous-asset. **A drawn-mask UI could not be verified from public docs** — flagged unverified.
- **Style:** custom training + **40+ prebuilt game-style models** + the standout cheap trick: **prompt prefix/suffix framing lock** (prefix = view, suffix = art style; artist types only the asset name). Batch Refine (upscale/remove-bg/vectorize across many assets at once).
- **Licensing:** closed SaaS, consumption pricing from $10/mo; studio-oriented, not solo-API-oriented.
- **Copy:** prefix/suffix framing lock (trivial to implement, huge consistency win); activity-feed generation ledger; batch refine over selected regions.

### Krea (krea.ai)
- **Interaction model:** three distinct surfaces (reviews conflate them): **Realtime** = split-panel (draw left, live output right, debounced whole-frame regen, no render button); **Nodes** = the actual infinite canvas (typed, color-coded node graph; a Node Agent compiles natural language into a wired pipeline); **Krea Edit** (2026) = object-level region editing (move/relight/recolor/expand).
- **Per-region regen:** not in Realtime (whole-frame); lives in Edit nodes (inpaint/outpaint) and Krea Edit.
- **Style:** up to 3 style-reference images each with a **weight slider**, moodboards, creativity knob, LoRA training.
- **Open weights (notable):** Krea Realtime 14B video model and Krea 2 image weights are published (HF `krea/krea-realtime-video`, custom license).
- **Copy:** keep "steer whole image by drawing" and "surgically edit a region" as **separate modes**; typed node graph as the conveyor metaphor (generate → slice → matte → export as nodes); reference-image + weight-slider style lock.

### Leonardo.ai (owned by Canva)
- **Interaction model:** Canvas Editor built around a **generation frame** (bounding box) with mode switch (Text2Img/Img2Img/Sketch2Img/Inpaint/Outpaint); Realtime Canvas (debounced sketch→image at 512/640 — ideation only); **Flow State** (endless scrolling stream of variants; "More Like This" anchors a new stream to a picked image).
- **Per-region regen:** mature. Key semantic: **Mask vs Erase are different ops** — masking retains underlying subject/style (refinement), erasing fully removes (replacement), with erase **targets** (all/images/sketches/mask). **Focus Mode** = lasso-select → reprompt just that area.
- **Real vs marketing:** all shipping; Realtime output resolution genuinely low. Historically the most game-dev-oriented SaaS (asset-suite guides, API for game pipelines).
- **Copy:** mask-vs-erase distinction (maps directly to keep-and-refine vs clear-to-alpha-and-replace); lasso-to-reprompt; Flow State's cheap-candidates-stream + "more like this" reroll anchor; guidance-vs-creativity two-knob steering.

### Ideogram Canvas
- **Interaction model:** infinite canvas with a rigorous pixel model (1 image px = 1 grid px; on-canvas resize is visual only, resolution preserved). **Two object types: image frames (produced assets) and generation windows (draggable bounding boxes defining where generation happens)**. Full toolbar: Generate, Magic Fill, Extend, Remix, Select (with layer order), Text, Upload.
- **Per-region regen:** **Magic Fill** = mask region(s) + position a generation window covering mask **plus unmasked context** → prompt → 4 variants, cycled with arrow keys. Documented correctness rules: smaller window = higher pixel density on the region = finer detail; **upscale before inpainting**; simplify the prompt to what's inside the window. Mask toolkit: rect / freeform / brush+size / eraser / **invert** / **load previous mask**.
- **Licensing:** closed SaaS; metered API, no subscription needed ($0.03-0.09/image).
- **Copy:** the frame-vs-window object model (his slices ARE generation windows); window = mask bbox + context margin, auto-suggested; upscale-before-inpaint; the mask toolkit spec verbatim; 4-up + arrow-key cycling.

### Freepik (Pikaso / Magnific)
- **Interaction model:** Pikaso = realtime sketch-to-image on one canvas where **each partial regeneration stacks as a layer**; Spaces = node workflow canvas; plus standalone Retouch (inpaint) / Expand (outpaint). Fine-grained Pikaso mechanics are **medium-confidence** (official doc redirects; some pages 403'd).
- **Distinguishing pattern:** one canvas/API over **39+ swappable models** (Flux, Imagen, Mystic, Ideogram…), with LoRA/custom-model support via API.
- **Copy:** model-agnostic per-region regen (build the mask UI once, treat the generator as a swappable backend — exactly his gpt-image-via-Codex situation); layers as regen history (delete = remove layer, regen = new layer over same mask).

### Adobe Photoshop Generative Fill — the gold-standard masked-regen loop
- **The loop, precisely:** any selection tool (marquee/lasso/Select Subject) defines the mask (dilate 10-20% for blend context) → **Contextual Task Bar** appears near the selection with prompt field + Generate → **exactly 3 variations**, written to a **non-destructive Generative Layer** (mask baked in, source never touched) → thumbnails in a **Variations rail** (Properties panel) with inline prev/next arrows → **Generate** again appends 3 more; **Generate Similar** (2025+) seeds 3 new candidates off a chosen one. Variations persist with the file — re-pick later. **Generative Expand** = drag crop boundary outward + same loop.
- **Confirmed contract:** Adobe's own Firefly Services API Generative Fill endpoint is literally `image + mask + prompt → results`.
- **Copy:** everything above — this is the interaction spec for his "regenerate region" button. Selection IS the mask; prompt UI appears only when a region is selected; always N candidates on a non-destructive slot; "more" (new seed) vs "similar" (reseed off pick).

### Adobe Firefly Boards
- **Interaction model:** infinite-canvas ideation; every generation/variation is a **tile**, so the canvas itself is the history. Collect (one-click gather) + grid arrange. **Eyedropper visual prompting** — sample a tile's style or composition and apply it to the next generation without typing. Describe Image = reverse-prompt an upload into editable text. Multi-model per generation. Per-region "change parts" tool is real but under-documented; "make it a board" name unconfirmed.
- **Copy:** variants-as-tiles spatial history; eyedropper style sampling from existing approved assets; reverse-prompting imported references.

### Canva AI (Magic Studio) / Kittl
- **Canva Magic Grab (standout):** click a photo → AI lifts the subject into a **separate movable element "like a sticker" and auto-inpaints the revealed hole**. Grab Text does the same for text. Magic Edit = brush-mask + prompt → options; Magic Expand = outpaint. Mostly Pro-gated; candidate counts unconfirmed.
- **Kittl:** vector-first canvas (layers, artboards, pen tool, Boolean shape builder); **text-to-vector with genuinely editable anchor points**; **no masked region-inpaint found** — consistency via reference images instead.
- **Copy:** Magic Grab = the region-promotion primitive (segment → promote to layer + heal background) — nearly identical to his slice step; Kittl's lesson: keep exported regions editable (layered/segmented), not baked flat.

### InvokeAI Unified Canvas (open source — the strongest OSS reference)
- **Interaction model:** canvas with **raster layers + specialized layer types** (Inpaint Mask, Regional Guidance with per-region prompts, Control Layers, Reference Image layers; recent releases added mask invert). A **bounding box** defines where generation happens and **auto-selects the mode from its contents**: empty pixels → txt2img; existing content → img2img; content+mask → inpaint; content+transparent margin → outpaint. Mask tools: brush/eraser/rectangle. Results land in a **staging area**: prev/next arrows, accept to commit, save-to-gallery without committing, or discard. Hotkeys: Q layer swap, B brush, E eraser, V move.
- **Status/licensing:** **Apache 2.0**, self-hosted, community-maintained (the hosted service shut down Oct 31 2025 as founders joined Adobe; the OSS project continues, v6.x releases through 2026). Headless REST + WebSocket API with OpenAPI schema; node-graph backend.
- **Caveat:** the whole app assumes a **local diffusion backend** (SD/SDXL/Flux, local GPU) and its model-management machinery. The React frontend is large and coupled to that backend.
- **Copy:** bounding-box-contents → auto mode selection (brilliant, removes a mode picker); the staging accept/discard flow; regional-guidance-as-layers; Apache 2.0 means code can be mined freely.

### ComfyUI + Krita AI Diffusion (open source)
- **ComfyUI:** node-graph GUI/API/backend, **GPL-3.0** (custom nodes in-tree are derivative works and must be GPL; external tools that only call its API are not bound). Active (v0.22, May 2026). Powerful but the graph UI is the wrong surface for an artist conveyor — it's the backend layer.
- **Krita AI Diffusion (Acly):** GPL-3.0 Python plugin, v1.52.1 (June 2026), turns Krita into the artist surface over a managed-or-remote ComfyUI: **selection-based generative fill** (Krita selections are the masks), outpaint/expand, live painting, ControlNet suite, **"Regions" = per-layer text prompts**, job queue + generation history. Needs a 6GB+ GPU (CPU fallback exists, slow). Desktop app — **not embeddable in a browser tool**.
- **Copy:** the pattern "existing editor's native selections drive inpaint" and "regions = layers with attached prompts"; job queue + history as first-class objects.

### Fooocus / A1111 inpaint UX (open source, context)
- A1111: canonical but dated masked-inpaint form UI (mask blur, "only masked" vs "whole image", fill modes). Fooocus (GPL-3): simplified it to **method presets** ("Improve Detail" / "Modify Content") plus a mask-grow/shrink (erode/dilate) slider — converting expert knobs into named intents. Both are Gradio form-tools, not canvases. **Copy:** preset-named inpaint intents instead of raw denoise sliders; mask dilate/erode as a single slider.

### openOutpaint (open source — closest architectural precedent)
- **What it is:** MIT-licensed, **pure vanilla JS + HTML canvas, zero dependencies**, thin client over the A1111 HTTP API. Effectively infinite pan/zoom canvas, **marquee/bounding-box "dream" generation**, masked inpaint brush with overmask (dilate) control, layers, undo/redo, **queueable/cancelable generations + generate-ahead batching**, grid snapping, localStorage persistence, floating tool panels.
- **Why it matters:** it is proof that the lead's existing architecture (thin browser canvas + local HTTP image API) supports the full conveyor — someone already built exactly that shape, solo, in vanilla JS. Code is MIT and liftable.

### Excalidraw / tldraw (canvas primitives)
- **Excalidraw:** MIT, embeddable React component (`@excalidraw/excalidraw`), programmatic scene control. But it's a hand-drawn-shape whiteboard; image-region masking, pixel grids, and mask brushes would all be custom work on top.
- **tldraw:** excellent infinite-canvas SDK, custom shapes/tools; "make real" starters are MIT. **But since SDK 4.0 the license requires a paid production key (~$6,000/yr business) or a watermarked non-commercial hobby license** — dev use is free, production isn't. For a personal local tool the hobby watermark might be tolerable, but it adds React + a licensing dependency to what is currently a dependency-free vanilla stack.
- **Verdict as primitives:** neither buys much for an image-slicing tool — their value is shape editing/collab, not raster region work.

### PixelLab (pixellab.ai)
- **Interaction model:** three thin clients (web form; in-browser **Pixelorama** editor with AI bolted on; **Aseprite extension**) over one cloud backend. The Aseprite plugin is a docked panel that connects via **websocket** (cloud generation, not local) and reads/writes the open sprite's canvas/selection/layers directly. Also an MCP server (Claude Code/Cursor can drive it) and **MIT-licensed JS/Python SDKs**.
- **Per-region regen:** selecting Inpaint **auto-creates a dedicated "Inpainting" layer**; painted pixels = editable, unpainted = frozen context; hard region caps (max 100-160px by tier). Plus rotate-to-8-directions, skeleton animation, tileset generation, remove background.
- **Real vs marketing:** docs + MIT SDK make the mechanics credible; quality claims and the one glowing review could not be independently verified (fetch failures, no substantive forum threads found).
- **Copy:** inpaint-layer-as-mask (paint where regeneration is allowed); hard small-region budget to keep regens cheap and disciplined; MCP/SDK-driven tool automation.

### Retro Diffusion (Astropulse)
- **Two products:** Aseprite extension — **one-time purchase ($65/$20), runs fully locally/offline**, proprietary pixel model; website — cloud credits, better model, animation presets (walking/idle, four-angle, VFX). Per-region masked inpaint on the web tool **unverified** (JS-rendered pages wouldn't fetch).
- **Alpha/palette (its real strength):** one-click background removal + **palette upload/enforcement + smart color reduction applied in-generation** — palette-as-constraint is how it ships game-ready output.
- **Licensing angle:** closed source; outputs commercially yours; model trained on **licensed art from 30+ consenting/paid pixel artists** (vendor's ethical-training framing, not audited) — one of the few generative sprite models with a provenance story, relevant to this repo's strict asset-licensing invariant.
- **Copy:** palette-lock as a pipeline constraint; the local-first one-time-purchase model validates the local-conveyor thesis.

### Per-region regen on sprite sheets + alpha handling (the exact conveyor)
- **Spritesheets.ai:** has a **"Frame-Level Editor" — regenerate individual frames with a new prompt**, reorder, de-dup; exports transparent PNGs + JSON/XML atlas, premultiplied alpha. Closest commercial match to "reroll one region of a sheet." Proprietary.
- **blendi-remade/sprite-sheet-creator (GitHub):** open-source (license unspecified — verify) Next.js + HTML canvas over **fal.ai with swappable models (incl. GPT Image 2)**; **per-layer regeneration** (retry one parallax layer, keep the rest), adjustable grid dividers for slicing, Bria bg-removal, live canvas test sandbox. Essentially a reference implementation of the conveyor's front half.
- **Tilewise.ai:** for tilesets, generates **one base texture, then derives all 47 autotile variants procedurally through template masks** — iteration = re-stamp, not re-generate. Lesson: derive regions procedurally where continuity matters.
- **PixelRefiner (GitHub, MIT):** browser-only TS/Vite tool, algorithms in Web Workers: **corner-color detection + flood-fill background keying with tolerance**, grid detection (auto/hint/force), AA removal (snap to hard pixels), Oklab+K-means color reduction, dithering, auto-trim, batch ZIP export, before/after compare. **Directly liftable code for the alpha/cleanup stage.**
- **Gamelabs chroma recipe:** generate on a solid matte color absent from the art → key with threshold (~70) + edge strength (~80) sliders → checkerboard preview → check fringing. Sprite-AI adds the important **Global vs Flood-fill removal toggle** (flood-fill from edges preserves same-colored pixels inside the sprite).
- No mature Godot/Unity plugin doing masked per-tile AI regen was found — that niche is open; the lead's conveyor would not be reinventing a solved commodity.

## UX patterns to copy

1. **The masked-regen primitive (Photoshop/Firefly contract):** `regenerate(image, mask, prompt?) → N candidates` into a **non-destructive slot bound to the region**; contextual prompt bar appears only when a region is selected; thumbnail rail + inline prev/next; **"Generate more" (new seed) and "Generate similar" (reseed off a chosen candidate)**; candidates persist with the region forever.
2. **Generation windows over image frames (Ideogram):** treat each slice as a draggable regen target = mask bbox + auto-suggested context margin; enforce "small window + surrounding frozen context"; **upscale-before-inpaint** for pixel density. PixelLab's variant: cap region size hard so regens stay cheap.
3. **Region promotion, Magic-Grab style (Canva):** click/lasso a region → it becomes an independent movable layer with the hole behind it healed (or keyed to alpha). This is exactly slice → keep/delete/move, and it makes delete trivial (remove layer).
4. **Mask vs Erase semantics + a real mask toolkit (Leonardo + Ideogram):** mask = keep-and-refine; erase = clear-to-alpha-and-replace (with erase targets). Toolkit: rect / lasso / brush+size / eraser / **invert** / **load-previous-mask**; single mask engine behind eraser/lasso/modify presets (Recraft); mask grow/shrink as one slider (Fooocus/openOutpaint "overmask").
5. **Auto mode from bounding-box contents (InvokeAI):** empty → generate; content → variate; content+mask → inpaint; content+empty margin → outpaint. Plus its **staging area** (cycle results, accept/save/discard) — merge this with pattern 1.
6. **Style locks without training:** per-project **prompt prefix/suffix framing** (Layer.ai — prefix=view, suffix=art style, user types only the subject); ≤5 reference images with weight sliders as a saved style token (Recraft/Krea); **eyedropper-sample an approved asset as the style reference** for the next generation (Firefly Boards); per-region model choice + influence slider (Scenario).
7. **Canvas/ledger as history:** every generation is a tile/entry — spatial variants on canvas (Firefly Boards), searchable activity feed (Layer.ai), queue + generate-ahead so the model works while the user reviews (openOutpaint), cheap-candidate stream with "More Like This" anchoring (Leonardo Flow State).
8. **The alpha stage as its own conveyor step (all game tools):** force a chroma matte at generation (color absent from the palette); corner-sample + edge flood-fill keying with tolerance + Global/Flood-fill toggle; threshold/edge-strength sliders over a checkerboard preview; AA-removal + palette-lock (Oklab K-means) after keying; premultiplied-alpha PNG + atlas JSON export; heavy ops in Web Workers (PixelRefiner proves this out, MIT).

## Build vs adopt recommendation

**Extend the existing thin stage-view canvas. Do not embed InvokeAI, ComfyUI+Krita, or tldraw.** Reasoning:

- **The backend mismatch is decisive.** InvokeAI and Krita-AI-Diffusion's canvases are tightly coupled to a *local diffusion* backend (SD/Flux, GPU, model management). The lead's generator is **remote gpt-image via Codex CLI** — an `(image, mask, prompt)`-shaped API call with seconds-scale latency. Adopting those canvases means adopting their whole GPU stack to get their UX; wiring gpt-image into them is fighting their architecture. Conversely, his conveyor's unique stages (slice, region labels, 9-slice, local alpha ownership, promote-to-game with license/provenance) exist in none of them and would have to be forked in.
- **openOutpaint is the existence proof** that his current architecture (vanilla JS canvas + local Python HTTP API) carries the full pattern — infinite canvas, bbox generation, mask brush, queue, history — in MIT code he can read and lift. His stage-view modules (`ai_studio/assets/viewer/asset_tools_stage_view.mjs`, `asset_tools_viewport.mjs`, `asset_tools_history.mjs`, `asset_tools_region_label.mjs`, `asset_tools_slice9.mjs`) already cover viewport, history, and regions; what's missing is roughly: a mask layer per region, a `regenerate(region)` action calling the Codex/gpt-image path, a candidates rail with accept/discard, and a style-lock (prefix/suffix + reference image) store. That's incremental work on owned code, not a framework adoption.
- **Canvas SDKs don't pay for themselves here:** tldraw needs a paid production key or watermark and drags in React; Excalidraw (MIT) is shape-whiteboard-shaped — image masks, pixel grids, and mattes would all be custom anyway. The lead's surface is image-regions-on-a-plane, which his viewport module already does.
- **What to adopt rather than build:** lift **PixelRefiner's MIT algorithms** (corner-detect flood-fill keying, grid detection, AA removal, Oklab color reduction, Web-Worker layout) into the existing Python/JS alpha stage; mine **openOutpaint's MIT code** for mask brush/overmask/queue mechanics; study **sprite-sheet-creator** for the grid-divider + per-layer-regen flow (verify its unspecified license before copying code). Optionally, keep **InvokeAI (Apache 2.0, headless REST/WS API)** on the radar as a *second, local* generation backend behind the same `regenerate(image, mask, prompt)` interface if he ever adds a local GPU model — adopt it as a backend, never as the UI.
- **One architecture rule to bake in from day one (Freepik/Scenario lesson):** make the generator a swappable backend behind one interface. gpt-image today; a local model or another API tomorrow; per-region backend choice later for free.

## Pitfalls

- **gpt-image mask fidelity:** OpenAI's image-edit endpoint accepts masks, but users widely report gpt-image regenerating the whole image with drift outside the mask. Mitigation (do this regardless of model): **composite locally — take only masked pixels from the result and paste back over the untouched original**, treating the model's output as a donor patch, never as the new canvas. (Reported behavior; verify against the current Codex/gpt-image path.)
- **"Realtime canvas" hype doesn't transfer:** Krea/Leonardo realtime modes ride sub-second local/turbo models. With seconds-per-image gpt-image round trips, design for **async queue + N candidates + generate-ahead**, not live paint.
- **Canvas framework weight:** tldraw = paid production license (~$6k/yr) or watermark, plus React; ComfyUI frontend = GPL-3.0 (in-tree extensions become GPL; API-only clients are fine); Krita is desktop, not embeddable. Embedding any of them replaces a working ~zero-dependency stack with a framework migration.
- **Style drift across per-region regens:** regenerating regions independently slowly diverges the sheet. Mitigations: frozen-context margins around every mask, prefix/suffix framing lock, reference-image style token from already-accepted regions, and (Tilewise lesson) derive repeating regions procedurally from one authoritative texture instead of regenerating each.
- **Licensing traps in the ecosystem:** Recraft/Scenario/Kittl free tiers exclude commercial use (Recraft free outputs are public); sprite-sheet-creator's repo license is unspecified; SaaS model rosters churn monthly (Photoshop's partner-model lineup could not be pinned down). For this repo's provenance invariant, note Retro Diffusion's consent-trained model as the rare positive example — but it's vendor framing, not audited.
- **Unverified claims (honesty ledger):** Layer.ai's drawn-mask inpaint UI and Live Canvas internals; Retro Diffusion's web-editor per-region regen; Pikaso's fine-grained canvas mechanics; Firefly Boards' region-edit brush details and "make it a board" naming; PixelLab's real-world output quality; Canva Magic Edit candidate counts; all vendor latency claims. None of these change the recommendation.
