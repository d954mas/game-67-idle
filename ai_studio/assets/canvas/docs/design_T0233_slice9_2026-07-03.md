Author: deep-reasoner · 2026-07-03 · status: design for orchestrator review → lead confirm

> **Synthesis pass (2026-07-03, second deep-reasoner):** reconciled against T0232 (rotation/flip),
> which had already LANDED when this was first drafted. Two corrections folded in: (1) slice-9 must

Lead ask (verbatim): «Сделать что-то слайс9 картинкой (чтобы проверить а работает ли
слайс9)». He wants to give a canvas image 9-slice insets and TEST that scaling behaves —

## 0. Recommendation summary (read this first)

- **Data model**: additive top-level `element.slice9 = { left, top, right, bottom }`, insets
  in **source pixels** (integers). Absent = exactly today's single-`drawImage` behavior
  (zero migration). Image elements only; loud error on a text element.
- **Op**: one dedicated journaled op `setSlice9({ projectId, elementId, insets|null })` —
  same shape as `setRegions`/`setExportSettings`. `null` clears. Loud validation at set
  time: each inset a non-negative integer, `left+right < source_w`, `top+bottom < source_h`
  (the engine's `sl+sr < source_w` invariant). Undo is free (slice9 rides in the elements
  snapshot). **Dedicated op, not a `patchElement` field** — slice-9 is a structured, top-level,
  separately-validated render property with its own inspector panel, which is exactly the
  `regions`/`export` shape (each got a dedicated `setRegions`/`export-set` op + its own route),
  a stronger structural match than the scalar `rotation`/`flipH` fields. It localizes the loud
  source-dim validation and gives a clean null-to-clear. *(The `patchElement`-field alternative —
  the flip precedent — was independently weighed; it is viable but a weaker fit. This is the one
  genuine architectural fork — see Open Decision 0.)*
- **Shared math (parity keystone)**: a new pure `slice9.mjs` (fonts.mjs/tree.mjs pattern —
  imported by `ops.mjs` in node AND served to the site) plus an exact **Python twin**
  `tools/slice9.py` imported by `render_group.py`. Both expose `slice9Patches(insets, srcW,
  srcH, dstW, dstH) → 9 patches` with an **identical proportional clamp**. One source of
  truth per language, mirrored line-for-line.
- **Renderers**: page `paintElement` swaps its one `drawImage` for a loop of ≤9 `drawImage`
  patches; `render_group.py` `paint_element` builds the sliced layer from the same patch
  list. PIL stays the source of rendered truth; the page is a same-pixels approximation
  (~1px edge drift, corners crisp by construction) — the exact parity stance text already
  uses.
- **Transforms (T0232 — load-bearing)**: slice-9 **replaces the single "resize to box" step**
  in each renderer, producing the same box-sized surface, so it **composes with rotation/flip
  for free** — the canvas 9-draw loop runs INSIDE the existing `ctx.rotate/scale(flip)` wrapper,
  and PIL assigns `image = sliced` BEFORE the existing flip→rotate→paste chain. A rotated,
  flipped 9-slice panel Just Works, no new refusal (see §4.0). Region ops (detect/slice/alpha)
  are orthogonal — slice-9 never touches source pixels, so no new guard.
- **UI**: v1 = an inspector **Slice-9** section (4 numeric insets + Clear). v2 = draggable
  inset guides on the canvas.
- **Reuse verdict**: the "old viewer slice9 module" the lead remembers **does not exist in
  the repo**. The only real slice9 reference is the **engine** (`nt_sprite_comp.c` +
  `nt_sprite_renderer.c`), which is native C emitting GPU triangles for 3D sprites — lift
  the **model convention and the loud invariant**, port **zero code**; write the 2D
  drawImage/PIL math fresh in the shared module.
- **Increments**: 2 fast-worker packets. Packet 1 = parity-correct core (shared JS+Python
  math, op, CLI, API route, BOTH renderers, tests) — lead can test immediately. Packet 2 =
  inspector Slice-9 section + acceptance run + parity golden test + deferred-knob decisions.

## 1. Reuse verdict on the "old viewer slice9 module"

There is **no `ai_studio/assets/viewer/` in the working tree**. The only match is a stale
staging copy under `tmp/asset-stage-…/ai_studio/assets/viewer/` — and that is the **asset
gallery viewer**, which contains **zero** slice9 code (grep for `slice9|nine|panel` →
0 hits). A repo-wide grep for `slice9|nine-slice` finds slice9 only in:

- `external/neotolis-engine/engine/sprite_comp/nt_sprite_comp.c` — the SoA data model.
- `external/neotolis-engine/engine/renderers/nt_sprite_renderer.c` — the emit (GPU verts).
- `external/neotolis-engine/examples/slice9_demo/` — an engine demo.
- template/knowledge docs (prose only).

**Verdict — lift the concept, port no code.** The engine is native C that emits 16 vertices
/ 54 indices per sprite in world-matrix space for a 3D GPU pipeline (`emit_one_slice9_branch`
in `nt_sprite_renderer.c:673`). None of that transfers to 2D canvas `drawImage` or PIL
crop/paste. What IS worth mirroring (and is cheap to re-derive):

- **Model**: insets in **source pixels**; corner destination size fixed, middle bands
  stretch — engine stores `slice9_lrtb` = `[left, right, top, bottom]` uint16
  (`nt_sprite_comp.c:329`). (I reorder to `{left, top, right, bottom}` for a self-documenting
  object; note the engine's positional order differs.)
- **Loud invariant**: engine asserts `sl + sr < source_w && st + sb < source_h`
  (`nt_sprite_renderer.c:697`). We adopt this as the set-time loud validation.
- **Grid construction**: local splits `x = [0, L, W-R, W]`, `y = [0, T, H-B, H]`
  (`nt_sprite_renderer.c:746-747`). Our 2D patch grid is the same idea, minus the 3D
  transform.

The engine also carries a `slice9_scale` corner multiplier (`nt_sprite_comp.c:358`). We
**defer** that to v2 (see Open Decisions) — v1 keeps corner-dst = source-inset 1:1, which is
the most predictable thing to "test".

## 2. Data model

Add one **additive, optional** field to an image element:

```json
{ "id": "el_…", "type": "image", "x": …, "y": …, "w": …, "h": …,
  "src": "files/<hash>.png", "source_w": 100, "source_h": 100,
  "slice9": { "left": 24, "top": 24, "right": 24, "bottom": 24 } }
```

- **Insets are SOURCE pixels**, integers, `>= 0`. Absent field ⇒ render exactly as today
  (one `drawImage`, whole image scaled to `w×h`). Zero migration; legacy projects untouched.
- **Top-level, not `meta`.** slice9 is a first-class render property like `regions` and
  `export` (both top-level). `meta.*` is provenance (parent, alpha run metadata); slice9 is
  live layout, so it belongs beside `regions`/`export`.
- **Orthogonal to geometry.** `w`/`h` resize (`patchElementBox`) is unchanged: slice9 only
  changes HOW the source maps into the box, never the box itself. No coupling, no special
  resize handling.
- **Image only.** `type:"text"` + slice9 is a loud error (mirrors the `content/style` on a
  non-text element guard in `sanitizeTextPatch`, `ops.mjs:198`).
- **Composes with T0232 rotation/flip — no refusal.** `element.rotation`/`flipH`/`flipV`
  (landed) and `element.slice9` are independent: slice-9 controls how the source fills the box;
  rotation/flip transform that filled box about its center. Both renderers apply slice-9 as the
  box-assembly step *before* flip→rotate (§4.0), so a rotated 9-slice panel is correct with zero
  extra code. Nothing to guard.
- **Orthogonal to region ops — no new guard.** `detectRegions`/`sliceRegions`/`alphaCutout`
  read the element's **untransformed source pixels**; slice-9 is a display-time re-tiling that
  never moves source pixels or source coordinates, so stored regions stay valid and those ops
  keep working on a slice-9 element. slice-9 is **not** added to `isNodeTransformed`/
  `refuseIfTransformed` (a slice-9 element that is *also* rotated is already refused by the
  existing rotation guard).
- **Set-time validation (loud, at the op)** — the "insets exceeding image dimensions" law:
  - each of left/top/right/bottom finite, `Math.round`ed, `>= 0`;
  - `left + right < source_w` else throw `slice9 left+right (N) must be < source width W`;
  - `top + bottom < source_h` else the height twin.
  - `source_w`/`source_h` are already persisted by `addImage`; fall back to `w`/`h` if a
    legacy element lacks them (same fallback `setRegions` uses, `ops.mjs:787`).

Undo/redo: **free**. `commitMutation` snapshots the whole `elements` array
(`snapshotOf`, `ops.mjs:212`), so slice9 is restored like any element field with no journal
change.

## 3. Shared-math parity plan (the hard part)

### 3.1 One pure module per language

Follow the `fonts.mjs` / `tree.mjs` precedent exactly:

- **`ai_studio/assets/canvas/slice9.mjs`** — pure, no I/O. Imported by `ops.mjs` (node) AND
  served statically to the site over `/ai_studio/` (import in `workspace.js`). Exports:
  - `validateSlice9(insets, sourceW, sourceH)` → normalized `{left,top,right,bottom}` or
    throws (the loud set-time rules above). Used by the op.
  - `slice9Patches(insets, srcW, srcH, dstW, dstH)` → array of ≤9 patches
    `{ sx, sy, sw, sh, dx, dy, dw, dh }`. Source coords are integers; **dst coords are
    ELEMENT-LOCAL world units** in `[0..dstW] × [0..dstH]` (each consumer maps to its own
    device space). Zero-area patches are dropped.
- **`ai_studio/assets/canvas/tools/slice9.py`** — the exact Python twin of `slice9Patches`
  (+ a thin `validate` mirror for defense-in-depth). Imported by `render_group.py`. Same
  clamp formula, same float arithmetic, same patch order.

Both files carry a one-line header: "Mirror of the sibling slice9.{py,mjs}; keep the clamp
formula identical — parity is the contract." The parity golden test (Packet 2) pins them.

### 3.2 The algorithm (identical in both languages)

```
slice9Patches({left, top, right, bottom}, srcW, srcH, dstW, dstH):
    # source bands (fixed; middle > 0 guaranteed by set-time validation)
    sxs = [0, left, srcW - right, srcW]           # 3 source columns
    sys = [0, top,  srcH - bottom, srcH]          # 3 source rows

    # destination insets, proportionally clamped so corners never overlap the box.
    # (Standard CSS border-image behavior; deterministic; float.)
    dL, dR = left, right
    if dL + dR > dstW and dL + dR > 0:
        f = dstW / (dL + dR); dL *= f; dR *= f
    dT, dB = top, bottom
    if dT + dB > dstH and dT + dB > 0:
        f = dstH / (dT + dB); dT *= f; dB *= f
    dxs = [0, dL, dstW - dR, dstW]
    dys = [0, dT, dstH - dB, dstH]

    patches = []
    for c in 0..2:                                 # columns L, M, R
        for r in 0..2:                             # rows T, M, B
            sw = sxs[c+1]-sxs[c]; sh = sys[r+1]-sys[r]
            dw = dxs[c+1]-dxs[c]; dh = dys[r+1]-dys[r]
            if sw <= 0 or sh <= 0 or dw <= 0 or dh <= 0: continue   # skip empty band
            patches.push({ sx: sxs[c], sy: sys[r], sw, sh,
                           dx: dxs[c], dy: dys[r], dw, dh })
    return patches
```

Properties this guarantees (the lead's test criteria):
- **Corners** (c,r ∈ {0,2}): `dw = left|right`, `dh = top|bottom` — independent of `dstW/H`,
  so they stay a fixed size while the box resizes. Enlarge → corners crisp; the ask.
- **Edges**: top/bottom bands stretch on X only, left/right bands on Y only.
- **Center**: stretches both.
- **Extreme downscale** (`dstW < left+right` or `dstH < top+bottom`): the `f` factor squishes
  corners proportionally and the center/edge band on that axis collapses to `dw/dh = 0` and is
  dropped. No crash, no error — a resize drag passes through small sizes transiently. **Same
  clamp in both renderers because it lives in the shared function.**

### 3.3 Rounding / sampling parity stance

Adopt the text contract verbatim: **PIL is the single source of rendered truth; the page is a
faithful same-pixels approximation.**

- **Source boxes are exactly integer in both** (insets + `srcW/H` are ints) → corners sample
  identical source pixels.
- **Dst boxes**: the page draws the shared float boxes sub-pixel (`ctx.drawImage` takes
  floats); PIL rounds each patch's dst box with `round()` at crop/paste. Drift on stretched
  edges ≤ ~1px — the accepted text-parity tolerance.
- **Corners are crisp by construction**: at export scale 1, corner `dw/dh` are integers equal
  to the source crop size → PIL pastes with **no resize** (byte-exact). At zoom ≥ 2 on the
  page, existing `imageSmoothingEnabled = vp.scale < 2` (`workspace.js:119`) already makes
  corners nearest.
- **No NEW sampling asymmetry.** Today's non-slice9 path already differs (page uses the
  zoom-based smoothing flag; PIL always LANCZOS in `paint_element`, `render_group.py:68`).
  slice9 uses the same per-patch resample (LANCZOS default, matching current behavior) so it
  introduces no parity regression. A `nearest` per-element resample for pixel-art panels is a
  clean future add tied to a resample field (Open Decisions).

## 4. Rendering

### 4.0 Composition with T0232 rotation/flip (load-bearing — corrects the pre-T0232 sketch below)

**The current `paintElement` and `paint_element` already carry a rotation/flip transform stack
(T0232 3a).** slice-9 must slot INTO it, not beside it, or a rotated/flipped 9-slice panel breaks.
The clean framing: **slice-9 is a drop-in replacement for the single "resize/draw the whole image
to the box" step**, producing the identical box-sized surface; every transform step downstream is
unchanged. This is why it composes for free.

- **Canvas** (`workspace.js` `paintElement`, current lines 230-266): today, when
  `rotation || flipH || flipV`, the code does `translate(center) → rotate(+θ) → scale(flip) →
  translate(-center)` and then ONE `ctx.drawImage(img, origin.x, origin.y, w, h)`; else the same
  single draw. slice-9 replaces **that single draw** (in both the transformed and untransformed
  branch) with the 9-patch loop, drawn at the **unrotated** screen box (`origin + p.dx*vp.scale`).
  Because the loop sits inside the same `ctx` transform, the assembled panel rotates/flips as one
  unit — the exact trick the single `drawImage` already relies on.
- **PIL** (`render_group.py` `paint_element`, current lines 80-93): today, resize→(line 81)
  flipH/flipV transpose→(82-85) `rotate(-rotation, expand)`→(86-88) paste centered→(89-93).
  slice-9 replaces **the resize (80-81)** with `image = slice9_assemble(...) → box_w×box_h`, then
  the existing flip→rotate→paste run on `image` unchanged. Composition is automatic *as long as
  the branch sits at the resize step* (before flip/rotate) — call this out in the packet.

### 4.1 Page — `workspace.js` `paintElement` (current: line 230; transform branch 246-266)

Replace the single `ctx.drawImage(img, origin.x, origin.y, w, h)` (which appears in BOTH the
`rotation||flipH||flipV` branch and the plain `else`) with a shared local `drawBody()` that draws
either the 9 patches or the whole image — so slice-9 composes with the existing transform wrapper:

```js
const rotation = Number(element.rotation) || 0, flipH = element.flipH === true, flipV = element.flipV === true;
const patches = element.slice9 && img.complete && img.naturalWidth
  ? slice9Patches(element.slice9, img.naturalWidth, img.naturalHeight, element.w, element.h)  // dst element-local
  : null;
const drawBody = () => {
  if (patches) {
    for (const p of patches)
      ctx.drawImage(img, p.sx, p.sy, p.sw, p.sh,
        origin.x + p.dx * vp.scale, origin.y + p.dy * vp.scale, p.dw * vp.scale, p.dh * vp.scale);
  } else {
    ctx.drawImage(img, origin.x, origin.y, w, h);                 // unchanged fallback
  }
};
if (rotation || flipH || flipV) {                                 // EXISTING T0232 wrapper — reused verbatim
  const c = imageToScreenPoint({ x: element.x + element.w / 2, y: element.y + element.h / 2 }, vp);
  ctx.save(); ctx.translate(c.x, c.y);
  if (rotation) ctx.rotate((rotation * Math.PI) / 180);
  if (flipH || flipV) ctx.scale(flipH ? -1 : 1, flipV ? -1 : 1);
  ctx.translate(-c.x, -c.y);
  drawBody(); ctx.restore();
} else {
  drawBody();
}
```

`img.naturalWidth/Height` are the source pixels; `imageFor` caches the Image. Selection stroke /
region overlay / rotated-quad outline / clip-ghost paths are untouched (they use `element.w/h`,
which slice9 does not change). The clip-ghost redraw can stay a single-`drawImage` low-alpha
approximation — it is a "where did my sprite go" hint, not truth (note for the implementer;
optional to slice it too).

### 4.2 Export — `render_group.py` `paint_element` (current: line 55; resize at 80-81, flip/rotate 82-88)

Today (line 80-81): resize the whole image to `box_w × box_h` LANCZOS. **Replace that resize
step** (so the result flows into the EXISTING flip→rotate→paste chain at lines 82-93, giving
transform composition for free — §4.0):

```python
if node.get("slice9"):                                            # REPLACES the resize at 80-81
    patches = slice9_patches(node["slice9"], image.width, image.height,
                             node.get("w") or image.width, node.get("h") or image.height)
    sliced = Image.new("RGBA", (box_w, box_h), (0,0,0,0))         # box_w/h already scaled
    for p in patches:
        crop = image.crop((p["sx"], p["sy"], p["sx"]+p["sw"], p["sy"]+p["sh"]))
        dw, dh = max(1, round(p["dw"]*scale)), max(1, round(p["dh"]*scale))
        if (crop.width, crop.height) != (dw, dh):
            crop = crop.resize((dw, dh), Image.Resampling.LANCZOS)
        sliced.paste(crop, (round(p["dx"]*scale), round(p["dy"]*scale)), crop)
    image = sliced
elif (image.width, image.height) != (box_w, box_h):
    image = image.resize((box_w, box_h), Image.Resampling.LANCZOS)  # unchanged non-slice9 path
# then the UNCHANGED T0232 chain: flipH/flipV transpose -> rotate(-rotation, expand) -> paste centered
```

Because `image = sliced` is assigned *before* the flip/rotate steps, a rotated/flipped slice-9
element composites correctly with no other change. Spec change: `ops.mjs` element-node builder
(`compositeGroup` / `buildRenderNodes`, beside the `flipH`/`flipV` it already forwards) adds
`slice9: element.slice9 || undefined` to the emitted element node (one line). `paint_element`
reads `image.width/height` for the source grid — no extra `source_w` needed in the spec.

### 4.3 Standalone element export (`exportElements`)

`exportElements` scales the **source file** by the row's scale token and never consults
`element.w/h` — so it has no display box to slice into. **v1 default: `exportElements`
ignores slice9 and exports the source art as-is** (the file IS the panel; slice9 only has
meaning at a display box inside a screen). This mirrors the text stance ("standalone text
export is a v1.1 skip — put it in a group and export the screen"). Slice9 bakes through
`renderGroup` / `exportProject`, which honor `element.w/h`. Flagged in Open Decisions if the
lead wants standalone baking.

### 5.3 Acceptance script (this IS the lead's deliverable)

Real assets exist in-repo: `templates/template/assets/ui/panel.png` (100×100 RGBA) and
`button.png` (384×128). Steps (CLI + page; `<pid>` = project id, `<eid>` = element id):

1. `node ai_studio/assets/canvas/cli.mjs create --title "slice9 test"` → `<pid>`.
2. `… add-image <pid> --file templates/template/assets/ui/panel.png` → note `<eid>`
   (source 100×100).
3. `… slice9-set <pid> --element <eid> --left 24 --top 24 --right 24 --bottom 24`.
4. Open the page (`/canvas?project=<pid>`), select the panel, drag it **WIDE**
   (e.g. w=600, h=120): the 24px corners stay crisp and square, the top/bottom borders
   stretch only horizontally, left/right borders only vertically, the center fills. Drag
   **TALL** (w=120, h=600): same, corners fixed. ← the core proof.
5. **Contrast**: `… slice9-set <pid> --element <eid> --clear` → the whole panel now smears
   its corners when stretched (old behavior). Re-set to confirm the difference is slice9.
6. **Parity**: put the panel in a group and `… render-screen <pid> --group <gid> --scale 1`
   then `--scale 2`; open both PNGs — corners crisp, layout matches the page (render-truth
   law).
7. **Clamp (no crash)**: resize to w=30 (< left+right=48): corners squish proportionally,
   center vanishes, no error toast, no exception.
8. **Loud error**: `… slice9-set <pid> --element <eid> --left 60 --right 60` (60+60 ≥ 100)
   → CLI/API error `slice9 left+right (120) must be < source width 100`; nothing written.

Pass = steps 4/6 show fixed corners on page and in the baked PNG; step 8 errors loudly.

## 6. CLI / API surface

Mirror the `setRegions` wiring (one op, both clients — strict parity):

- **Op** (`ops.mjs`): `export function setSlice9(root, { projectId, elementId, insets })`.
  `insets` an object → `validateSlice9` then store `element.slice9`; `insets === null` →
  delete the field. Journaled via `commitMutation` (op name `"setSlice9"`,
  `args_summary: { elementId, insets }`). Returns `{ project, element }`.
- **API** (`api.mjs`): `PUT /api/canvas/projects/:id/elements/:eid/slice9` with body
  `{ insets }` (or `{ insets: null }`). Sits next to the existing
  `PUT …/elements/:eid/regions` and `…/export` routes. **Concurrency note**: `api.mjs` is
  being edited by another agent — Packet 1 must add this route as an isolated block and
  rebase if it conflicts; the route is self-contained (one handler calling `setSlice9`).
- **CLI** (`cli.mjs`): `slice9-set <id> --element <eid> [--left N --top N --right N --bottom N] | --clear`.
  Missing insets default to the element's current value merged with the flags (partial edit),
  OR require all four for v1 simplicity — recommend: read current `element.slice9` (or zeros),
  apply the provided flags, send the merged object; `--clear` sends `null`. One line in the
  CLI help block (after `regions-set`).
- **Action** (`site/actions.js`): `export async function setSlice9(elementId, insets)` →
  `applyMutation(await api("PUT", \`/projects/${pid()}/elements/${elementId}/slice9\`,
  { insets }))`. Consumed by the inspector section.

## 8. Open decisions (with my defaults)

0. **Op vs `patchElement` field (the one architectural fork).** Default **dedicated `setSlice9`
   op** — matches the `regions`/`export` structural precedent (each a top-level structured render
   property with its own op + route + panel), localizes the source-dim loud validation, clean
   null-clear. The alternative (`patchElement({slice9})`, the flip precedent, no new op/verb/route,
   minimal surface) is viable if the lead/orchestrator prefers minimum op-surface; it needs a
   `sanitizeSlice9Patch` doing the same element+source-dim lookup `sanitizeTextPatch` already does.
   Both are correct; recommend the op. *(Ephemeral "another agent is editing ops.mjs" concerns are
   a sequencing matter under the one-writer law, not a design driver.)*
1. **`slice9_scale` corner multiplier (engine has it).** Default **defer to v2** — v1 keeps
   corner-dst = source-inset 1:1, the most testable behavior. Additive later (`element.slice9.scale`,
   default 1), multiplies `dL/dR/dT/dB` before the clamp; zero migration.
2. **Standalone `exportElements` baking.** Default **v1 exports the source as-is** (no display
   box to slice into; mirrors the text standalone-export skip). Revisit if the lead wants a
   single panel exported pre-stretched — would need an explicit target size, which is really a
   "put it in a group" workflow.
3. **`nearest` sampling for pixel-art panels.** Default **LANCZOS everywhere in v1** (== current
   `paint_element`). Add a per-element `resample` later, shared by both renderers, when a
   pixel-art panel needs it.
4. **CLI partial vs full insets.** Default **merge flags over the element's current slice9**
   (partial edits are ergonomic: bump just `--left`); `--clear` sends `null`. Full-four-required
   is the simpler alternative if the fast-worker prefers.
5. **Inset object vs positional array.** Default **object `{left,top,right,bottom}`** (self-
   documenting, matches regions/export style). The engine's `[l,r,t,b]` array order is noted so
   nobody copies it by muscle memory.
6. **Clip-ghost redraw slicing.** Default **leave it a single low-alpha `drawImage`** (it is a
   hint, not truth). Slicing it is optional polish.
