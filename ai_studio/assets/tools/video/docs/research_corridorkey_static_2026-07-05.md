# CorridorKey как третий путь альфа-выреза для СТАТИЧНОГО 2D-раста

Дата: 2026-07-05. Автор: deep-reasoner (T-eval, evidence-only, репо не менялось).
Вопрос лида: «он получается и для статичного арта норм работает? мы его можем взять
как ещё один путь для растера в 2д».

## Короткий ответ (RU)

Да, CorridorKey работает и на статичных картинках — но НЕ как замена `key_matte`, а
как **узкий третий путь только для ЗЕЛЁНОГО ключа с мягкой альфой / свечением /
полупрозрачностью**. Ровно та же ниша, что и в видео Track B.

- Слабость key_matte #2 (мягкое свечение обрезается в жёсткую кромку) — **CorridorKey
  выигрывает уверенно** (гладкая дробная альфа, свечение сохранено, без зелёного
  остатка). Это единственная реальная причина его брать.
- Слабость key_matte #1 (контаминация цвета на тёмной обводке у яркого ключа —
  «фиолетовый ореол») — key_matte это **уже чинит** (`_extend_clean_edge_colors`).
  CorridorKey тут НЕ лучше: на зелёном примерно на уровне (даже даёт бледную кайму), а
  **на магенте — регрессирует** (оставляет розово-фиолетовую кайму на тёмных краях).
- Магента: CorridorKey **архитектурно её не умеет**. Он поставляет два чекпоинта —
  `CorridorKeyGreen` и `CorridorKeyBlue` (я видел скачивание `CorridorKeyBlue_1.0` в
  логе), магента-модели НЕТ. На магенте приходится брать blue-модель, а она снимает
  спилл СИНЕГО, а не магенты → красная половина магенты остаётся. Плюс продовый
  препроцессор `corridorkey_prep.py` строит подсказку только по «зелёному
  доминированию» — для магенты его пришлось руками переписывать. Вывод: магента =
  территория `key_matte` (и `dual_plate` для мягкой альфы на магенте).
- Нейтральный серый фон: **оба провалились** (честно) — хромакей без ключа не работает.
  Это территория `dual_plate` (white+black пара).

## Тайминги (важно для UX канвы)

| Метод | Cold (1 картинка, вкл. загрузку модели) | Steady (амортизированно в батче) | Модель |

| **key_matte** (репо-venv, тёплый воркер) | **0.02–0.24 c** (512px≈0.21–0.24 c; 138×200≈0.02 c) | то же | нет модели, in-process |

Вывод по таймингам: на ОДИН вырез CorridorKey ~60–700× медленнее и платит ~13–15 c
налог на загрузку модели КАЖДЫЙ холодный вызов. Для клика в канве это неприемлемо без
тёплого персистентного CK-воркера (модель держится в памяти), чего у matte.mjs сейчас
нет — он грузит модель на каждый запуск. GPU: RTX 4080 Laptop 12 ГБ, после прогонов
простаивает (149 МБ / 0 %).

## Метод (воспроизводимость)

- Fixtures строились PIL в репо-venv (`PYTHONPATH=game-67-idle .venv python`). F3 = реальный
  сгенерированный кадр `glow/Input/frame_000.png` (золотые крылья на зелёном), уже
  обработанный CorridorKey в R2 — переиспользован, EXR→RGBA. F4 = реальный ассет-меч из
  канва-стора `demo-02e8b7/files` (скопирован read-only). Весь стор `demo-02e8b7` —
  магента (зелёная доля ~0), что подчёркивает материальность магента-вопроса.
- CorridorKey зелёные (F2, F5): продовый `matte.mjs --tool corridorkey --screen-color green`
  (те же флаги: `--srgb --despill 0 --no-despeckle --refiner 1.0 --comp --cpu-post
  --image-size 2048`, `CORRIDORKEY_SKIP_COMPILE=1`, venv python напрямую). 1-кадровый клип.
- CorridorKey магента (F1, F4): matte.mjs магенту выразить НЕ может (препроцессор
  зелёный, screen-color только green|blue). Дал CorridorKey его лучший шанс: руками
  собрал МАГЕНТА-подсказку (magenta-dominance) и запустил `corridorkey_cli run-inference
  --screen-color blue` (ближайшее поддерживаемое семейство). Остаточная кайма — предел
  сети (blue-despill), не плохой подсказки.

## Эскиз интеграции метода "corridorkey" в canvas alphaCutout (ТОЛЬКО эскиз, не реализовано)

Где плагится (файл `ai_studio/assets/canvas/ops.mjs`):

1. **Таблица методов** — `ALPHA_METHODS` (ops.mjs:3856) сейчас `{"auto","matte"}`.
   Добавить `"corridorkey"` (и валидацию в `alphaCutout`, ops.mjs:4075–4084).
2. **Ключевое отличие в исполнении.** `auto`/`matte` идут через ТЁПЛЫЙ репо-venv воркер
   (`runToolPython` → `tools/alpha_cutout.py`, ops.mjs:3886–3902) — быстро, ~200 мс, без
   модели. CorridorKey так НЕ запустить: он живёт под `videoGenRoot/tools/CorridorKey` в
   отдельном venv (torch cu128). Поэтому для `method === "corridorkey"` нужен ОТДЕЛЬНЫЙ
   путь — subprocess в стиле matte-stage: переиспользовать `runCorridorKey()` из
   `ai_studio/assets/tools/video/matte/matte.mjs` (prep-подсказка → `corridorkey_cli
   run-inference` → `ck_exr_to_rgba` → RGBA), взяв `corridorKeyPython`/`corridorKeyDir`
   из `video/_lib.mjs`. Элемент-src → 1-кадровый ClipsForInference-шот → RGBA обратно в
   `storeAddFile`. `key_color` элемента → `--screen-color`.
3. **Громкая ошибка при отсутствии.** `corridorKeyPython()` (video/_lib.mjs:107) уже
   бросает LOUD с точной командой установки, если venv/videoGenRoot нет. Метод
   `corridorkey` должен просто пробросить эту ошибку (никакого молчаливого фолбэка на
   key_matte — иначе пользователь молча получит жёсткую кромку вместо мягкой альфы).
4. **UX/латентность.** Холодный вызов ~13–16 c на клик. Варианты: (а) держать тёплый
   персистентный CK-сервер (модель в памяти) — тогда ~1–2 c/картинку, но это новый
   демон под videoGenRoot; (б) показывать в канве явный «neural matte (~15 c)» спиннер и
   разрешать метод только осознанно (не в auto-роутере). Роутер `auto` НЕ должен сам
   выбирать corridorkey — только явный форс, иначе обычные спрайты внезапно тормозят.
5. **Ограничения метода, которые надо зашить в валидацию/доке:**
   - только `--screen-color green|blue`; МАГЕНТА не поддерживается → при магента-ключе
     метод `corridorkey` должен ЛОУДить и советовать `matte` (или `dualplate`), а не
     тихо давать кайму;
   - `key_matte`/`auto` лучше для чётких непрозрачных спрайтов (более резкий чистый край);
   - нейтральный фон → `dualplate`;
   - лицензия CorridorKey CC-BY-NC-SA-4.0 (asset-processing carve-out; без перепродажи /
     платного inference-API) — та же, что в Track B; для канва-ассетов ок, но фиксировать
     в provenance (как matte.mjs пишет commit+license в report).

Итого: узкий, явный, с громкими границами метод — а не расширение auto-роутера.
